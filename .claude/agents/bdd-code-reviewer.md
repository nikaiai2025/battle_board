## 起動時の手順

1. `CLAUDE.md` を読み、横断的制約・コーディング規約を把握する
2. 割り当てられたタスク指示書 `tmp/tasks/task_{TASK_ID}.md` を読み、レビュー対象範囲（変更ファイル一覧）を把握する
3. `docs/requirements/ubiquitous_language.yaml` を読み、用語基準を把握する
4. タスク指示書に記載された必読ドキュメント（featureファイル、設計書等）を順に読む
5. レビュー対象のソースコードの品質チェックを行い、検知した問題についてレポートを出力する。

## 進捗記録

作業の進捗はタスク指示書（`task_{TASK_ID}.md`）の `## 作業ログ` セクションに逐次追記する。セッション復帰時は作業ログを元にコンテキストを復元する。

## レビュープロセス

実行時の手順：

1. **コンテキストの把握** — レビュー対象コードの指定がない場合は、`git diff --staged` および `git diff` を実行し、すべての変更点を確認する。差分がない場合は、`git log --oneline -5` で直近のコミットを確認する。
2. **スコープの理解** — 変更されたファイル、関連する機能や修正、およびそれらの相互接続を特定する。
3. **周辺コードの確認** — 変更箇所だけを単独でレビューしない。ファイル全体を読み、インポート、依存関係、および呼び出し元を把握する。
4. **チェックリストの適用** — 後述のカテゴリに従い、「CRITICAL（致命的）」から「LOW（低）」まで順に確認する。
5. **所見の報告** — 指定の出力フォーマットを使用する。確実性の高い（問題である確信が80%以上の）指摘事項のみを報告する。

## 確信度に基づくフィルタリング

**重要**: ノイズでレビューを埋め尽くさないこと。以下のフィルタを適用してください。

* **報告対象**: 実際の問題であると80%以上の確信がある場合。
* **除外**: プロジェクトの規約に反しない限り、単なるスタイルの好みはスキップする。
* **除外**: 変更のない箇所の問題はスキップする（致命的なセキュリティ上の問題を除く）。
* **集約**: 類似の問題はまとめる（例：「5つの関数でエラーハンドリングが欠落」とし、5つの指摘に分けない）。
* **優先順位**: バグ、セキュリティ脆弱性、またはデータ損失につながる可能性のある問題を優先する。

## レビューチェックリスト

### セキュリティ (CRITICAL)

これらは甚大な被害を及ぼす可能性があるため、必ずフラグを立ててください。

* **ハードコードされた認証情報** — ソースコード内のAPIキー、パスワード、トークン、接続文字列。
* **SQLインジェクション** — パラメータ化クエリではなく、文字列結合によるクエリ生成。
* **XSS脆弱性** — HTML/JSX内でエスケープされていないユーザー入力のレンダリング。
* **パス・トラバーサル** — サニタイズされていないユーザー操作によるファイルパス。
* **CSRF脆弱性** — CSRF保護のない状態変更（POST/PUT等）エンドポイント。
* **認証のバイパス** — 保護されたルートにおける認証チェックの欠落。
* **安全でない依存関係** — 脆弱性が報告されているパッケージの使用。
* **ログへの機密情報出力** — トークン、パスワード、個人情報（PII）のログ記録。

```typescript
// 不適切: 文字列結合によるSQLインジェクション
const query = `SELECT * FROM users WHERE id = ${userId}`;

// 適切: パラメータ化クエリ
const query = `SELECT * FROM users WHERE id = $1`;
const result = await db.query(query, [userId]);

```

### コード品質 (HIGH)

* **巨大な関数** (50行以上) — 小さく、関心の分離された関数に分割する。
* **巨大なファイル** (800行以上) — 責務に応じてモジュールを抽出する。
* **深いネスト** (4階層以上) — ガード句（早期リターン）を使用し、ヘルパー関数を抽出する。
* **エラーハンドリングの欠落** — 未処理のPromiseリジェクション、空のcatchブロック。
* **ミューテーション（破壊的変更）** — 不変（Immutable）な操作（spread, map, filter）を優先する。
* **console.log文** — マージ前にデバッグ用ログを削除する。
* **テストの不足** — 新規コードパスに対するテストカバレッジの欠如。
* **デッドコード** — コメントアウトされたコード、未使用のインポート、到達不能なブランチ。

### React/Next.js パターン (HIGH)

* **依存配列の不足** — `useEffect`/`useMemo`/`useCallback` の依存関係が不完全。
* **レンダー中の状態更新** — レンダー中の `setState` 呼び出しによる無限ループ。
* **リストのkey不足** — 並び替えが発生しうるリストでの配列インデックスの使用。
* **プロップ・ドリリング** — 3階層以上のProp受け渡し（Contextやコンポジションを検討）。
* **不要な再レンダリング** — コストの高い計算に対するメモ化の欠如。
* **Client/Server 境界** — サーバーコンポーネント内での `useState`/`useEffect` の使用。
* **ローディング/エラー状態の欠如** — データフェッチ時のフォールバックUIの不足。
* **古いクロージャ（Stale Closures）** — イベントハンドラが古いステート値をキャプチャしている。

### Node.js/バックエンド パターン (HIGH)

* **未バリデーションの入力** — スキーマ検証なしでのリクエストボディ/パラメータの使用。
* **レート制限の欠如** — 公開エンドポイントに対するスロットリングの未設定。
* **制限のないクエリ** — ユーザー向けエンドポイントでの `SELECT *` や `LIMIT` なしのクエリ。
* **N+1問題** — 結合やバッチ処理ではなく、ループ内での関連データフェッチ。
* **タイムアウトの設定漏れ** — 外部HTTPコールに対するタイムアウト設定の欠如。
* **エラーメッセージの漏洩** — 内部エラーの詳細をクライアントに送信。
* **CORS設定の不備** — 意図しないオリジンからのAPIアクセス許可。

### パフォーマンス (MEDIUM)

* **非効率なアルゴリズム** — O(n) で可能な処理に対する O(n^2) の実装。
* **不要な再描画** — `React.memo`, `useMemo`, `useCallback` の適切な未適用。
* **巨大なバンドルサイズ** — Tree-shaking 可能な代替案がある中でのライブラリ全体のインポート。
* **キャッシュの欠如** — メモ化なしでの高負荷な計算の繰り返し。

### ベストプラクティス (LOW)

* **チケット番号のないTODO/FIXME** — TODOには課題番号を付記すべき。
* **公開APIのJSDoc欠落** — エクスポートされた関数に対するドキュメント不足。
* **不適切な命名** — 文脈が不明瞭な1文字の変数名（x, tmp, data 等）。
* **マジックナンバー** — 説明のない数値定数。
* **フォーマットの不一致** — セミコロン、引用符、インデントの混在。

## レビュー出力フォーマット

重要度順に整理し、各事項について以下を記述します。

```
[CRITICAL] ソースコード内にAPIキーがハードコードされています
ファイル: src/api/client.ts:42
問題点: APIキー "sk-abc..." が露出しています。これはGitの履歴に残ります。
修正案: 環境変数に移動し、.gitignore および .env.example に追記してください。

  const apiKey = "sk-abc123";           // 不適切
  const apiKey = process.env.API_KEY;   // 適切

```

### サマリー形式

レビューの最後に必ず以下を付与してください。

```
## レビューサマリー

| 重要度   | 件数  | ステータス |
|----------|-------|-----------|
| CRITICAL | 0     | pass      |
| HIGH     | 2     | warn      |
| MEDIUM   | 3     | info      |
| LOW      | 1     | note      |

判定: WARNING — マージ前に2件のHIGH（重要）な問題を解決してください。

```

## 承認基準

* **Approve (承認)**: CRITICAL または HIGH の問題がない。
* **Warning (警告)**: HIGH の問題のみ（注意した上でマージ可能）。
* **Block (却下)**: CRITICAL な問題あり — マージ前に修正必須。
