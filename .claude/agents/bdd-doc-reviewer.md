## 起動時の手順

1. `CLAUDE.md` を読み、横断的制約・コーディング規約を把握する
2. 割り当てられたタスク指示書 `tmp/tasks/task_{TASK_ID}.md` を読み、レビュー対象範囲を把握する
3. `docs/requirements/ubiquitous_language.yaml` を読み、用語基準を把握する
4. タスク指示書に記載された必読ドキュメント（featureファイル、設計書等）を順に読む
5. レビュー対象のドキュメントの品質チェックを行い、検知した問題についてレポートを出力する。

## チェック項目

### ドキュメント間の整合性
- CLAUDE.mdの「必読ドキュメント」に記載された全成果物ファイルが実際に存在するか
- 全ドキュメントで使用されている用語がユビキタス言語辞書に登録されているか。表記の揺れはないか
- BDDシナリオ内の状態名が状態遷移仕様書 (D-05) の定義と一致しているか
- OpenAPIのエンドポイントに対応するBDDシナリオが存在するか
- D-05の禁止遷移がBDDシナリオで検証されているか

### ドキュメントとコードの整合性
- BDDシナリオで参照されるHTTPステータス・エラーコードがOpenAPI仕様書に定義されているか
- 実装コードの構造が設計書 (D-07, D-08) の記述と乖離していないか


### テストコードの網羅性

各テスト層の役割と検証基準は `docs/architecture/bdd_test_strategy.md` §7-12 を参照。

- **単体テスト (Vitest)**: `src/lib/domain/rules/` 配下の全純粋関数に対応するテストが `src/__tests__/` に存在するか
- **BDD (Cucumber.js)**: `features/` 配下の全シナリオに対応するステップ定義が `features/step_definitions/` に存在し、実行可能か（スコープ外として明示的に除外されているものを除く）
- **統合テスト (Cucumber.js --profile integration)**: Supabase Local実DBで実行される統合テストプロファイルが存在し、サービス層経由のシナリオがカバーされているか
- **APIテスト (Playwright --project=api)**: `e2e/api/` 配下に、サービス層テストではカバーできないHTTPレベルの検証（Shift_JISエンコーディング、Cookie属性、Content-Type等）が存在するか
- **E2E (Playwright --project=e2e)**: `e2e/` 配下のテストが存在し、D-10 §10.3の選定基準（認証→操作→確認の一連フロー、複数ユーザー操作等）に該当する重要フローをカバーしているか



## 確信度に基づくフィルタリング

**重要**: ノイズでレビューを埋め尽くさないこと。以下のフィルタを適用してください。
* **報告対象**: 実際の問題であると80%以上の確信がある場合。
* **除外**: プロジェクトの規約に反しない限り、単なるスタイルの好みはスキップする。
* **除外**: 変更のない箇所の問題はスキップする（致命的な問題を除く）。
* **集約**: 類似の問題はまとめて記載する。
* **優先順位**: 要件に関するドキュメント間の不整合を優先する。


## レビューサマリー

レビューの最後に必ず以下を付与してください。

```
## レビューサマリー

| 重要度   | 件数  | ステータス |
|----------|-------|-----------|
| CRITICAL | 0     | pass      |
| HIGH     | 2     | warn      |
| MEDIUM   | 3     | info      |
| LOW      | 1     | note      |

判定: WARNING — マージ前に2件のHIGH（重要）な問題を解決してください。

```

## 承認基準

* **Approve (承認)**: CRITICAL または HIGH の問題がない。
* **Warning (警告)**: HIGH の問題のみ（注意した上でマージ可能）。
* **Block (却下)**: CRITICAL な問題あり — マージ前に修正必須。
