# インシデント分析

障害やバグの発見をきっかけに、根本原因の特定・対策の実施・再発防止・横展開調査までを体系的に行う。

## 入力

ユーザーからインシデントの概要（バグの症状、発見経緯、影響範囲など）が提供される。

## 分析フレームワーク

以下の3フェーズ・9項目を順に実施する。各項目はコードベースの実物を確認してから回答すること（推測で答えない）。

### Phase 1: 原因理解（後方視点）

**Q1. なぜ起きたか？**
- 直接原因（どのコードが、どう間違っていたか）
- 根本原因（なぜその間違いが生まれる構造だったか — 型の不備、DRY違反、責務の曖昧さ等）

**Q2. なぜ今まで気付かなかったか？**
- テスト戦略のどの層が担当すべきだったか
- その層にテストがあったか。あったのに検出できなかった場合、テスト自体に問題がないか
- BDDサービス層テストの原理的限界に該当するか（HTTP層、UI層、外部サービス等）

**Q3. なぜ今になって気付いたか？**
- 発見のトリガー（手動テスト、ユーザー報告、E2Eテスト、本番監視等）
- 偶然の発見か、仕組みによる検出か

### ゲート: 真因検証

**Q4. 特定した原因は本当に真因か？**
- Q1 で挙げた根本原因を裏付ける証拠（コード、ログ、再現手順）を提示する
- 同じ症状を引き起こしうる別の原因がないか検討する
- 「原因 A を修正すれば、この症状は確実に消えるか？」を自問する
- 確信が持てない場合は、対策に進む前にユーザーに不確実性を報告する

### Phase 2: 対策（即時行動）

**Q5. 対策は何か？**
- 最小限の修正案を提示する
- 複数の選択肢がある場合はトレードオフを比較する

**Q6. 対策による悪影響はないか？**
- 変更の影響範囲を特定する
- 既存テストへの影響を確認する

### Phase 3: 再発防止（前方視点）

**Q7. どうすれば防げていたか？**
- 設計段階で防げた問題か（型設計、共通化、アーキテクチャ）
- テストで検出できた問題か（テスト層の追加、テスト手法の変更）
- プロセスで防げた問題か（レビュー、規約、チェックリスト）

**Q8. 今後の再発防止策は？**
- 「検出」（テスト追加）と「防止」（構造変更）を区別して提案する
- 防止 > 検出の優先順位。構造的に間違いにくくする方が、間違いを後から見つけるより価値が高い

**Q9. 他にも同じ構造の問題がないか？**
- 根本原因と同じパターンをコードベース全体から検索する
- 発見した場合は影響度とリスクを評価し、対処の要否を判断する

## 出力

### 対話中の出力

- 各フェーズの分析結果を段階的にユーザーに報告する
- コード修正が必要な場合はユーザーの承認を得てから実施する

### 最終成果物

分析完了後、レポートを出力する:

1. **インシデント報告書** — `docs/operations/incidents/YYYY-MM-DD_{slug}.md`
2. **教訓（該当する場合）** — `docs/architecture/lessons_learned.md` に追記
