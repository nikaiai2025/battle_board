# 敵対的コードレビュー

BDDシナリオ単位で実装コードを精査する。攻撃者（Red）と防御者（Blue）の2エージェントによるディスカッションを通じ、即座にバグとなりうる本質的な問題のみを抽出する。

## 入力

ユーザーが対象を指定する:
- `features/{file}.feature` — featureファイル単位（全シナリオが対象）
- `features/{file}.feature "シナリオ名"` — 特定シナリオのみ
- `all` — 全featureファイル

## ステップ1: スコーピングとグルーピング

対象featureファイルを読み、シナリオを**レビュー単位**にグルーピングする。

### グルーピング基準
- 原則: 1シナリオ = 1レビュー単位
- 例外: 以下に該当する場合は2〜5シナリオをまとめる
  - 同一の正常系/異常系ペア（例: 「登録する」「登録できない」）
  - 同一のステップ定義を共有している
  - 同一のAPIエンドポイント/サービス関数を検証している

グルーピング結果を `tmp/discussion/{YYYYMMDD_HHmm}/scope.md` に記録する:

```markdown
# 敵対的レビュー: {feature名}

## レビュー単位

| # | シナリオ | 関連ファイル（推定） |
|---|---------|-------------------|
| R-001 | シナリオ名A, シナリオ名B | service.ts, route.ts |
| R-002 | シナリオ名C | handler.ts |
```

## ステップ2: 実装ファイルの特定

各レビュー単位について、シナリオ → ステップ定義 → 実装コードのトレースを行う:

1. `features/step_definitions/{対応するステップ定義}.ts` を読む
2. ステップ定義の `import` 先を追跡し、関連する実装ファイルを列挙する
3. `tmp/discussion/{session}/R-{NNN}/context.md` に記録する:

```markdown
# R-{NNN}: {シナリオ名}

## 対象シナリオ
（featureファイルからシナリオ全文を転記）

## 実装ファイル
- `src/app/api/auth/register/route.ts` — APIルート
- `src/lib/services/registration-service.ts` — サービス層
- `src/__tests__/lib/services/registration-service.test.ts` — 単体テスト

## ステップ定義
- `features/step_definitions/user_registration.steps.ts` L42-L80
```

## ステップ3: 攻撃ラウンド（Red Agent）

`general-purpose` エージェントを **sonnet モデル**で起動する。プロンプトに以下を含める:

```
あなたは敵対的コードレビュアー（Red Team）です。

## 指示
1. `tmp/discussion/{session}/R-{NNN}/context.md` を読み、対象シナリオと実装ファイルを把握する
2. 実装ファイルを全て読む（テストコードも含む）
3. 最も重大な実装上の問題点を **3つだけ** 指摘し、`tmp/discussion/{session}/R-{NNN}/attack.md` に出力する

## 観点（優先度順）
1. バグ: シナリオの期待動作と実装の乖離、エッジケースでの破綻
2. セキュリティ: 認証バイパス、入力検証漏れ、情報漏洩
3. データ整合性: 競合状態、部分的失敗時のデータ不整合
4. テストの欺瞞: テストがグリーンだが実際の振る舞いを検証できていない（モックの過剰使用等）

## 出力形式（attack.md）
各指摘について以下を記述する:
- **問題ID**: ATK-{NNN}-{1,2,3}
- **重大度**: CRITICAL / HIGH
- **問題の要約**: 1文
- **詳細**: 何が問題で、どう壊れるか。具体的なコード箇所（ファイル名:行番号）を示す
- **再現条件**: この問題が顕在化する条件（「〜のとき、〜が起きる」）

## 禁止
- MEDIUM/LOW の問題は指摘しない（3枠を本質的な問題に使え）
- コーディングスタイル・命名・フォーマットへの指摘禁止
- 「〜した方が良い」レベルの改善提案禁止
- 「将来的に問題になりうる」という予測的指摘禁止
- 実装ファイル以外（featureファイル・設計書）への指摘禁止
```

## ステップ4: 防御ラウンド（Blue Agent）

別の `general-purpose` エージェントを **sonnet モデル**で起動する。プロンプトに以下を含める:

```
あなたは防御側コードレビュアー（Blue Team）です。

## 指示
1. `tmp/discussion/{session}/R-{NNN}/context.md` を読み、対象シナリオと実装ファイルを把握する
2. `tmp/discussion/{session}/R-{NNN}/attack.md` を読み、指摘内容を把握する
3. 実装ファイルを全て読み、各指摘に対して抗弁または同意を行う
4. 結果を `tmp/discussion/{session}/R-{NNN}/defense.md` に出力する

## 判定基準
各指摘に対し、以下のいずれかの判定を下す:

### REJECT（却下）: 以下に該当する場合、指摘を却下する
- 本番環境では到達しないコードパス（テスト専用コード等）
- フレームワーク/ライブラリ側の責務であり実装側で対処不要
- BDDシナリオのスコープ外の振る舞い
- 理論上の可能性であり、現実的な再現条件がない
- 既にテスト（単体/BDD/E2E）でカバーされている

### ACCEPT（同意）: 以下に該当する場合、指摘を受け入れる
- 再現条件が現実的で、本番で発生しうる
- 発生した場合にデータ損失・セキュリティ侵害・サービス停止を引き起こす
- テストで検出できていない

## 出力形式（defense.md）
各指摘について:
- **問題ID**: （attack.md と同じID）
- **判定**: REJECT / ACCEPT
- **根拠**: なぜ却下/同意するか。具体的なコード箇所を示して論証する
  - REJECT時: 問題が顕在化しない理由（防御コード、フレームワーク保護、テスト等）
  - ACCEPT時: 攻撃者の指摘が正しい理由と、影響の評価
```

## ステップ5: 判定（オーケストレーター）

オーケストレーターが `attack.md` と `defense.md` を読み、以下を判定する:

- **REJECT 判定が妥当か**: 防御側の根拠がコードに基づいており論理的か
- **ACCEPT 判定が妥当か**: 攻撃側の指摘が実在し再現条件が現実的か

判定結果を `tmp/discussion/{session}/R-{NNN}/verdict.md` に記録:

```markdown
# R-{NNN} 判定

| 問題ID | Red | Blue | 最終判定 | 理由 |
|--------|-----|------|---------|------|
| ATK-001-1 | CRITICAL | REJECT | 却下 | フレームワーク保護で到達不能 |
| ATK-001-2 | HIGH | ACCEPT | **採用** | 競合状態の再現条件が現実的 |
| ATK-001-3 | HIGH | REJECT | 却下 | BDDテストでカバー済み |
```

## ステップ6: 次のレビュー単位へ

ステップ2〜5を全レビュー単位について繰り返す。
エージェントは毎回新規に起動する（コンテキスト分離）。

## ステップ7: 最終レポート

全ラウンドで「採用」となった問題を集約し、`tmp/discussion/{session}/final_report.md` に出力する:

```markdown
# 敵対的コードレビュー 最終レポート

- 実施日: {YYYY-MM-DD}
- 対象: {feature名}
- レビュー単位数: {N}
- 指摘総数: {M} / 採用: {K} / 却下: {M-K}

## 採用された問題

### 1. {問題の要約}
- 問題ID: ATK-{NNN}-{X}
- 重大度: {CRITICAL/HIGH}
- 対象シナリオ: {シナリオ名}
- ファイル: {パス:行番号}
- 詳細: {攻撃側の指摘}
- 防御側見解: {同意理由}

### 2. ...

## 問題なし（採用ゼロの場合）

全 {M} 件の指摘が防御側により却下された。現時点で重大な実装上の問題は検出されなかった。
```

## ステップ8: アーキテクトレビュー（CRITICAL問題）

最終レポートで **CRITICAL** として採用された問題が1件以上ある場合、各問題について独立した `bdd-architect` エージェントを起動し、対応要否を判定する。**1件 = 1エージェント**（コンテキスト量の制約による）。

### エージェントへの指示内容
- 問題の詳細（攻撃側の指摘 + 防御側の見解）
- 対象ファイルを実際に読んで問題の実在を確認すること
- 以下の3段階で判定を出すこと:
  - **対応必須**: 本番で現実的に発生し、機能破綻・セキュリティ侵害・データ損失を引き起こす
  - **対応推奨**: 問題は実在するが影響が限定的、または発生頻度が低い
  - **対応不要**: 既に修正済み、または設計意図として許容されている
- 対応必須・推奨の場合は修正方針を簡潔に記述すること

### 出力先
各エージェントの評価を `tmp/workers/bdd-architect_ATK-{NNN}-{X}/assessment.md` に出力させる。

### 結果の集約
全アーキテクトの評価完了後、最終レポートに **アーキテクト評価サマリー** を追記する:

```markdown
## アーキテクト評価

| 問題ID | 概要 | アーキテクト判定 | 修正方針 |
|--------|------|----------------|---------|
| ATK-001-2 | ... | 対応必須 | ... |
| ATK-003-1 | ... | 対応不要 | 既に修正済み |
```

「対応必須」の問題がある場合、オーケストレーターは修正タスクの発行要否を人間に報告する。
「対応推奨」以下の問題は技術負債として記録し、人間への報告に含める。
