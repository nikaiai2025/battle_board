# スプリントクリーンアップ

Phase完了後やスプリント蓄積時に、tmpディレクトリ内の完了済みファイルをアーカイブし、作業領域をクリーンな状態に戻す。

## 実行手順

### 1. 現状確認

以下のディレクトリの中身を一覧表示し、アーカイブ対象を特定する:

- `tmp/orchestrator/` — スプリント計画書
- `tmp/tasks/` — タスク指示書
- `tmp/escalations/` — エスカレーション
- `tmp/workers/` — ワーカー作業空間
- `tmp/reports/` — レビュー報告書

### 2. エスカレーションのステータス修正

`tmp/escalations/` 内で `status: open` のファイルがある場合:

- `sprint_current.md` の内容・テスト結果と照合し、実質解決済みかどうかを判定する
- 解決済みであれば `status: resolved` に更新し、`resolved_at` と `resolution` を追記する
- 判断できない場合はユーザーに確認する

### 3. アーカイブ移動

各ディレクトリに `archive/` サブディレクトリを作成（なければ）し、完了済みファイルを移動する:

#### tmp/escalations/
- 全ての `resolved` / `closed` のエスカレーションを `archive/` に移動

#### tmp/tasks/
- 全ての完了済みタスク指示書（`status: completed` / `escalated` で解決済み）を `archive/` に移動

#### tmp/workers/
- 対応タスクが完了済みのワーカー作業空間ディレクトリを `archive/` に移動

#### tmp/orchestrator/
- 完了済みスプリント計画書を `archive/` に移動
- `sprint_current.md` から、今後は使わない情報（履歴として残していた古い情報）を `archive/sprint_past.md` に移植する（コピーではなくカット＆ペースト）
- 移動対象の計画書は統合アーカイブファイル `archive/sprint_{NNN}_{NNN}.md` に連結する（10スプリント単位）
- 統合ファイルのフォーマット:

```markdown
# Phase X スプリントアーカイブ (Sprint N-M)

> 各スプリントの計画書原本を統合したアーカイブ。

---

========================================
# Sprint-N
========================================

（原本の内容）

---
```

- BDDガイド等の補助ファイルも `archive/` に移動
- `sprint_current.md` と直近のスプリント計画書（1〜2件）はアクティブに残す

#### tmp/reports/
- 次フェーズでも参照する可能性がある報告書はアクティブに残す
- 明確に不要な報告書のみ `archive/` に移動（判断できない場合は残す）

#### tmp/
- `tmp/` 直下に置かれた雑多なファイルを `archive/` に移動（今後のスプリントに必要なものは残す）

### 4. sprint_current.md の更新

- スプリント履歴テーブル: アーカイブしたスプリントを1行にまとめる（例: `Sprint-1〜9 | completed | アーカイブ参照`）
- アーカイブインデックス: 移動先ファイルパスと内容を記載する

### 5. Gitコミット

- クリーンアップ内容をコミット・プッシュする。
- `tmp` 内の文書移動のみであるためデプロイ後の検証は不要。
- `tmp` 内の文書以外のファイルはコミットしない。どのファイルが未トラッキング状態であるかをユーザーへ報告する。

### 6. 最終確認

クリーンアップ後の各ディレクトリの状態を一覧表示し、結果をユーザーに報告する。
報告には以下を含める:

| 対象 | 実施内容 |
|---|---|
| エスカレーション | ステータス更新件数、アーカイブ移動件数 |
| タスク指示書 | アーカイブ移動件数 |
| スプリント計画書 | アーカイブ移動件数、統合ファイル名 |
| ワーカー作業空間 | アーカイブ移動件数 |
| reports | 残置/移動の判断理由 |
| 未コミット状態のファイル | 概要 |
