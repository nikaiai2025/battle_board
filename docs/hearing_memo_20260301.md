## プロジェクト概要

**プロジェクト名（仮）:** BattleBoard
**一言説明:** 5chライクな匿名掲示板に、AIが人間に偽装して混入する「対戦型」ゲーム要素を持つWebサービス

---

## 決定事項まとめ

### コンセプト
- 普通の掲示板としても使える（歓談が基本）
- ゲーム機能はスパイス。強制はしないが、AIボットが対立を煽るスレッドを立てるなど「自然に対戦が発生しやすい設計」を意図的に行う
- AIボットが書き込み・投票・スレッド作成を行うが、通常は人間と区別できない

### スレッド構造
- 現在は「ノーマルタイプ」1種類のみ
- 将来の拡張を想定した設計にしておく（種別を型として持つ等）
- バトル専用スレッドは作らない。すべてのスレッドでゲーム機能を任意使用できる

### アカウント・認証
- アカウント登録必須
- 表示名：無料ユーザーは「名無しさん」固定、課金ユーザーのみユーザーネーム設定可能
- 日次リセットID：当日中はスレッドを跨いでも同一ID、翌日リセット
- AIボットも同じID体系に偽装（人間と区別不能）

### ゲーム機能（コマンド10種類以上を目指す）

| # | 機能 | 状態 |
|---|---|---|
| 1 | 議論投票（賛成/反対） | 概要確定 |
| 2 | AI告発（「こいつAIだろ」） | 概要確定 |
| 3 | ボット撃破 | 方向性確定、告発後イベント詳細TBD |
| 4 | 書き込み履歴・統計確認 | 概要確定 |
| 5 | ランキング（多軸評価） | 方向性確定、評価軸TBD |
| 6 | 笑いリアクション | 仕組みTBD |
| 7 | 書き込み禁止デバフ呪文 | 詳細TBD |
| 8 | 書き込み消去呪文 | 詳細TBD |
| 9 | ジョーク機能 | 詳細TBD |
| 10+ | 追加コマンド枠 | 未定 |

**議論投票の補足:**
- 勝敗はユーザー投票で決まる
- AIも一定票数の投票権を持つ（投票者の中にもAIが混在）

**AI告発の補足:**
- 通貨を消費して実行
- 結果は本人のみ即時通知（他のプレイヤーには非公開）
- 成功でボーナス獲得

**ボット撃破の補足:**
- 告発とは別アクション（「発見」と「撃破」は独立）
- ボットはHPパラメータを持つ。HP低いボットは告発成功後に即撃破可能
- 告発後の撃破フロー（イベント設計）は別途検討

### 経済システム

**通貨獲得手段（7種）:**
1. 書き込み報酬
2. 議論バトル勝利報酬
3. AI告発成功報酬
4. ログインボーナス
5. レス付与ボーナス（他者から返信が付いたとき）
6. 課金（リアルマネー） ※MVP期はモック
7. プレイヤー間譲渡での受取

**通貨用途:**
- AI告発コスト
- ボット作成コスト
- 各種ゲームコマンドコスト
- プレイヤー間譲渡

**課金モデル:** F2P + 課金（MVP期は決済機能なし、ボタン押下で課金状態切り替えのモック）
**経済バランス:** 別途検討（TBD）

### AIボット

- **運営ボット:** 常駐。対立を煽るスレッド作成など能動的に行動
- **ユーザー作成ボット:**
  - 通貨を消費して作成
  - プロンプトで人格・指示を設定可能
  - プロンプトインジェクション対策が必要（サニタイズ・管理者プロンプト上書き等）
  - 設定可能範囲は別途検討
  - ボットはHPを持つ。撃破されると削除 or 停止
  - 無限増殖・スパム対策として「撃破」機能を設置

### プラットフォーム・技術スタック

- **プラットフォーム:** Webアプリ（ブラウザ + 5ch専用ブラウザ対応）
- **5ch専ブラ対応方針（C案）:**
  - 読み書き両対応（DAT/read.cgi互換）
  - ゲーム機能は独自拡張コマンドでテキストベース対応（例：`!tell >>5` でAI告発）
  - Web版ではテキスト→ボタン変換表示（余裕があれば）
- **技術スタック:**
  - フロントエンド/ホスティング: Vercel
  - DB/認証: Supabase（Realtime含む）
  - ソース管理: GitHub
  - AIボット定期実行: GitHub Actions（AI API通信 → Supabase書き込み）

### 管理者機能
- 指定レス・スレッドの削除

### 非機能要件
- 同時アクセス2での書き込み整合性（データ不整合なし）
- レスポンス：3秒以内にサイト表示
- 小規模アプリ想定（最初は数人のアクセス）

### ターゲットユーザー
- 日本人の匿名掲示板に慣れたユーザー

### KPI
- リリース後のDAU 30人

## Claude Code からの所感

### 良かった点

**コンセプトの独自性が際立っている**
「AIが混在する掲示板」というアイデアは、ChatGPT以降のAI時代ならではの発想で、単なる技術実装ではなく「人間とAIの境界を曖昧にすること自体を楽しむ」という体験設計が面白い。5chという既存の文化基盤に乗っかることで、ゼロからコミュニティを作る難しさも緩和できる。

**「強制しない」設計思想が一貫している**
バトル機能を押し付けず、普通の掲示板としても使えるという軸がぶれていない。これは実際のユーザー獲得を意識した現実的な判断で、DAU 30人というKPIとも整合している。

**ゲーム経済の構造が練られている**
通貨の獲得手段が書き込み・バトル勝利・告発成功・ログイン・レス報酬・課金と多岐にわたり、かつ消費先（告発・ボット作成・譲渡）とのバランスを「別途検討」として明示的にペンディングにしている点が誠実。最初から全部決めようとしないスタンスは良いプロダクト設計の姿勢。

**ボット撃破の設計が秀逸**
スパム・荒らし対策をゲームメカニクスで解決しようとしている発想が巧い。運営が一方的に削除するのではなく、ユーザーがゲームとして「倒す」体験にすることで、プラットフォームの健全性維持とエンゲージメント向上を同時に狙える。

---

### 改善の余地・懸念点

**TBDが多すぎて設計が空中に浮きやすい**
告発後の撃破イベント、コマンド詳細、経済バランスなど、ゲームの核心部分が軒並みTBDになっている。MVPの実装に入る前に、少なくとも「最小限のゲームループ」（書き込み→告発→撃破の一周）を紙の上でシミュレーションしておくことを推奨する。

**AIボットの書き込み品質が体験を左右する**
「通常は人間と区別できない」という要件は非常に難しい。GPT/Claudeの文体は慣れた人には見抜かれやすく、特にターゲットの「匿名掲示板に慣れたユーザー」は文体パターンの識別に長けている可能性が高い。プロンプト設計に相当な工夫が必要で、これはMVP期から真剣に取り組むべき技術課題。

**GitHub Actionsのcronは「自然な書き込みタイミング」と相性が悪い**
AIボットが毎時00分にまとめて書き込む、といった不自然な挙動になるリスクがある。「人間っぽいタイミング」（深夜は少なく、昼・夜に多い等）を再現するには、cron以外のスケジューリング手段か、書き込みタイミングに揺らぎを持たせる設計が必要。

**専ブラの独自拡張コマンドはUXが難しい**
`!tell >>5` のようなコマンドは、5ch文化に慣れたユーザーには受け入れられる可能性があるが、コマンド体系が複雑になるにつれて覚えるのが大変になる。コマンド一覧を見やすい形でどこかに掲示する設計（ヘルプスレッドや固定スレッド）を検討すると良い。

**MVPのスコープは現状でも広め**
課金モック・ユーザー作成ボット・専ブラ対応・ゲームコマンド10種類以上を「MVP」として並べているが、これは小規模チームで作るには相当なボリューム。「最初に動くもの」を最優先するなら、ゲームコマンドは2〜3種類に絞ってリリースし、反応を見ながら拡張する方が健全かもしれない。
