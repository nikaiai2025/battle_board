# インシデント報告: 荒らし役BOT seedデータ配置漏れ（9体分欠損）

## 基本情報

| 項目 | 内容 |
|---|---|
| 発生日 | 2026-03-20 |
| 検出方法 | ヘルスチェック（`/bot-health-check`）+ 会話中の気付き |
| 影響 | 荒らし役BOTが仕様の1/10しか稼働していなかった（1体 / 期待値10体） |
| 対応状況 | 修正済み（マイグレーション `00019_seed_arashi_bot_remaining9.sql` を追加） |

## 症状

- ヘルスチェックエンドポイント（`/bot-health-check`）の結果から投稿数に不整合が確認された
- 調査の過程で、荒らし役BOTの稼働数が1体のみであることが判明した
- BDDシナリオ（`features/`）では荒らし役BOTは10体と定義されているが、本番DBのseedデータには1体分のINSERTしか存在しなかった

## 原因分析

### 直接原因

マイグレーション `00016_seed_arashi_bot.sql` に含まれるINSERT文が1体分のみであった。残り9体のseedデータが未作成のまま本番環境に適用されていた。

### 根本原因

BDDシナリオとインフラ層（seed SQL）の間にトレーサビリティの仕組みが存在しない。

BDDテストはインメモリリポジトリで自己充足的に検証する設計のため、テスト実行時に必要な初期データはGivenステップで動的に生成される。この設計はテストの独立性・再現性の観点から正しいが、副作用として本番DBのseedデータとの乖離を原理的に検出できない。

seedマイグレーションの作成時にBDDシナリオで定義された数量との突き合わせを行うプロセスが存在しなかったため、1体分しか作成されなかったことが見過ごされた。

## なぜ検出できなかったか

BDDステップ定義（`features/step_definitions/bot_system.steps.ts` 691〜715行目付近）では、Givenステップで10体の荒らし役BOTをインメモリリポジトリに作成し、Thenステップで10体存在することを検証している。テスト自体は正しく書かれており、CIでも全てGREENであった。

しかしBDDサービス層テストの設計原則上、インメモリリポジトリでの検証は本番DBのseedデータとは独立している。これは設計の問題ではなく、BDDサービス層テストが持つ原理的な限界に該当する。

結果として、seedマイグレーションの体数不足はいかなる自動テストにも検出されず、ヘルスチェックでの手動調査によって偶然発見された。

## 対策

マイグレーション `00019_seed_arashi_bot_remaining9.sql` を追加し、不足していた9体分の荒らし役BOTを本番DBに追加した。

## 再発防止策

| 種別 | 施策 | 優先度 |
|---|---|---|
| 検出 | ヘルスチェック（`/bot-health-check`）にBOT数チェック（期待値10体）を追加する | 高 |
| 検出 | スモークテストにBOT数の検証項目を追加する | 中 |
| 防止 | seedマイグレーション作成時のレビューチェックリストに「featureシナリオで定義された数量との突き合わせ」項目を追加する | 低 |

## 横展開リスク

今回の問題は「BDDシナリオで定義された数量とインフラ層の初期データの不一致」という構造的パターンである。

現時点では荒らし役BOTのみが対象だったが、今後ネタ師・論客など新種のBOTプロファイルを追加する際に同様の乖離が発生しうる。BDDテストはインメモリで自己充足的に検証するため、seedマイグレーションの体数不足を自動検出できない点は変わらない。

新しいBOTプロファイルを追加する際は、featureシナリオで定義した体数とseedマイグレーションのINSERT数を必ず突き合わせること。

## 教訓

- BDDサービス層テストはサービスロジックの正しさを保証するが、インフラ層（seedデータ・マイグレーション）との整合性は保証しない。この限界を開発者全員が認識する必要がある
- ヘルスチェックエンドポイントは「期待値との比較」を含む形で設計すると、今回のような数量の不一致を早期に検出できる。現状のヘルスチェックが投稿数の絶対値のみを返す設計であったため、正常値との比較が手動調査に依存していた
- seedマイグレーションはBDDシナリオを唯一の正本として作成し、レビュー時にシナリオとINSERT数を突き合わせるチェックを設けることで、同種の漏れを防止できる
