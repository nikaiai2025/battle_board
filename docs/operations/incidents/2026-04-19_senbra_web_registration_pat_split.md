# インシデント報告書: 専ブラ認証から Web 本登録への導線断絶と PAT 復帰不全

- 発生日: 2026-04-19
- 分類: 認証 / 専ブラ互換 / 本登録導線
- 対象: 専ブラユーザーが通常ブラウザで本登録・復帰しようとする経路

## 概要

専ブラから未認証書き込み時に案内される `/auth/verify?token=...` を通常ブラウザで開いて Turnstile 認証すると、専ブラの仮ユーザー自体はそのままブラウザに引き継がれる。
しかし、その後の Web UI 導線が自然につながっておらず、`senbra` チャネルの `edge-token` では `/api/mypage` が 403 になるため、通常の「認証後にマイページへ進んで本登録する」操作が成立しない。

さらに、Cookie が分岐した後の復帰手段として期待される PAT についても、現行実装は「Cookie があれば PAT を認証に使わない」仕様で固定されているため、別人の有効 Cookie が残っているケースでは PAT での復帰ができない。

## Phase 1: 原因理解

### Q1. なぜ起きたか？

#### 直接原因

1. 専ブラ認証 URL でブラウザに持ち込まれる `edge-token` は `senbra` チャネルのままであり、`/api/mypage` は `web` チャネル以外を 403 で拒否する。
   - 専ブラ認証案内は `/auth/verify?token={edgeToken}` を返す。
   - `POST /api/auth/verify` は `body.edgeToken` を優先し、そのトークンをそのまま Cookie に設定する。
   - `GET /api/mypage` は `authResult.channel !== "web"` の場合 403 を返す。

2. `bbs.cgi` の PAT 認証分岐は Cookie より後に評価され、Cookie が存在するだけで PAT 分岐に到達しない。
   - ルートハンドラは `parsed.edgeToken` の存在だけで先に Cookie 経路へ進む。
   - この分岐では PAT は strip のみで、`loginWithPat()` は呼ばれない。
   - そのため「別人の有効 Cookie が残っている」「失効 Cookie が残っている」どちらでも PAT による復帰が阻害される。

3. Header / Layout の認証表示はチャネルを考慮せず、`senbra` トークンでも「認証済み」と見なす。
   - `getLayoutAuthStatus()` は `channel` を見ずに `isAuthenticated` と `isRegistered` を返す。
   - その結果、Web UI は「マイページへ行けそう」に見えるが、実際の `/api/mypage` は 403 となる。

#### 根本原因

1. チャネル分離を API レベルで導入した一方で、UI 導線設計が追従していない。
   - `web` と `senbra` の権限制御は追加されたが、「`senbra` トークンをブラウザに持ち込んだ後にどう `web` へ昇格させるか」が設計されていない。

2. PAT の役割が「Cookie 喪失時の復帰」に限定されており、「Cookie が誤って別人に紐づいている」ケースが仕様から漏れている。
   - 仕様・設計・テストが一貫して「Cookie 優先、PAT は補助」と固定している。
   - 実利用では、専ブラ Cookie と通常ブラウザ Cookie の分岐後に「PAT を正として上書きしたい」という要求が自然に発生する。

3. BDD がユーザーの自然導線ではなく、局所仕様の成立だけを見ている。
   - `認証完了後に write_token を使って書ける`
   - `mypage は web token のみ`
   - `PAT 認証後は Cookie を優先`
   これらは個別には成立しているが、組み合わせたときの end-to-end 導線が破綻している。

### Q2. なぜ今まで気付かなかったか？

#### 本来担当すべきテスト層

- 主担当は E2E / インテグレーション層。
- 理由:
  - 専ブラ → 認証 URL → 通常ブラウザ → Web UI → 本登録 → 専ブラ復帰、という跨層導線のため。
  - サービス層や個別 Route Handler の単体テストだけでは検出しにくい。

#### 実際のテスト状況

1. `mypage` のチャネルガードはテストされている。
   - `senbra channel token returns 403 FORBIDDEN`

2. PAT の Cookie 優先挙動もテストされている。
   - `Cookie有効の場合、PATで認証せず loginWithPat は呼ばれない`

3. つまり今回の問題は「テストが無かった」だけではなく、「現行の不整合な挙動がテストで固定化されていた」。

4. 欠けているのは次の E2E シナリオ。
   - 専ブラ認証 URL を通常ブラウザで開き、そのまま本登録導線に自然接続できる
   - `senbra` トークンでブラウザ認証したあと、誤誘導なく `web` 本登録へ進める
   - 別人 Cookie が残っている状態でも PAT で本人へ復帰できる

#### BDD サービス層テストの限界か？

- 一部は原理的に限界がある。
  - Header 表示と API 403 の不整合
  - 専ブラ URL ハンドオフ後のブラウザ導線
- ただし PAT 優先順位については限界ではない。
  - 現行テストが意図的に Cookie 優先を固定しているため、単なるテスト不足ではなく仕様・設計判断の問題。

### Q3. なぜ今になって気付いたか？

- 発見トリガーは手動での利用状況整理・ユーザー観察。
- 仕組みによる検出ではない。
- 既存テストはすべて通っており、監視や自動テストがこの問題を上げる構造にはなっていなかった。

## ゲート: 真因検証

### Q4. 特定した原因は本当に真因か？

#### 根拠

1. 専ブラ認証 URL は `token` クエリ付きで専ブラの `edge-token` をブラウザへ渡している。
2. `POST /api/auth/verify` はその `edge-token` を優先利用し、同じトークンを Cookie に保存する。
3. `GET /api/mypage` は `senbra` チャネルを拒否する。
4. `register` 系 API は `channel` を見ていないため、直接 URL を知っていれば本登録自体は可能。
5. PAT は Cookie 存在時に使われないことが feature / 設計 / 実装 / テストの全層で一致している。

#### 別原因の検討

- 「通常ブラウザで認証した時点で新規ユーザーが作られている」ことは、専ブラ案内 URL から入ったケースでは主因ではない。
  - `token` パラメータがあるため、同じ `edge-token` が使われる。
- ただし `token` を持たない汎用 `/auth/verify?redirect=/mypage` に流れた場合は、新規ユーザーが発行される。
  - これは別の分岐であり、今回の主ケースを悪化させる副次要因。

#### 真因判定

- 真因は単一ではなく、以下の複合障害である。
  1. `senbra` トークンをブラウザに持ち込んだ後の Web 導線未設計
  2. Header / Layout のチャネル非考慮
  3. PAT の「Cookie 優先」固定仕様

### Q4b. 他に隠れている要因はないか？

ある。

1. `bbs.cgi` の Cookie 分岐は「有効 Cookie」ではなく「Cookie が存在するか」で先に進む。
   - コメントでは `verifyEdgeToken` を行うように見えるが、実際には分岐前で検証していない。
   - そのため stale cookie が残っているだけでも PAT 自動復帰に入れない。

2. `senbra` トークンで本登録 API は通るが、UI 上にそこへ自然に導くリンクがない。
   - `/auth/verify` 成功後は write_token 表示に留まる。
   - Header には `mypage` と `login` が見えるが、本登録用の直接導線がない。

3. OAuth / メール確認完了後は `web` トークンが再発行されるため、そこまでたどり着けば解決する。
   - つまり本質は「本登録実処理」より、その前段の導線断絶にある。

## Phase 2: 対策

### Q5. 対策は何か？

#### 方針A: Web 本登録導線へ自然接続できるようにする

最小修正案:

1. `token` 付き `/auth/verify` からの認証成功後、`senbra` トークン保持者向けの専用導線を明示する。
   - 例: 「このまま本登録へ進む」ボタンを表示
   - `mypage` ではなく、本登録ページまたはトークン昇格 API へ直接誘導する

2. Header / Layout をチャネル考慮にする。
   - `senbra` トークンでは `mypage` を見せない
   - 代わりに「本登録」「専ブラ復帰」など適切な導線を見せる

3. 必要なら `senbra` → `web` の明示的な昇格ステップを追加する。
   - Turnstile 認証済みの同一ユーザーに対して `web` チャネルの edge-token を発行する
   - これが最も UI 一貫性が高い

#### 方針B: PAT を正として Cookie を上書きできるようにする

最小修正案:

1. `bbs.cgi` で PAT を Cookie より優先する
   - 少なくとも `mail` に PAT がある場合は先に PAT を検証し、その user に紐づく新しい `edge-token` を発行する
   - これにより別人 Cookie / stale cookie から復帰できる

2. その際、旧専ブラ Cookie のアカウントとの紐付けは失われる
   - これは要求どおりであり、仕様として明文化が必要

#### トレードオフ

- 方針Aのみ:
  - 初回導線は改善する
  - 既に Cookie が分裂したユーザーの復帰は不十分

- 方針Bのみ:
  - 復帰は改善する
  - 初回導線の不自然さ、UI 403 問題は残る

- A+B併用:
  - もっとも実利用に合う
  - ただし feature 更新が必要

### Q6. 対策による悪影響はないか？

#### 影響範囲

1. `src/app/(senbra)/test/bbs.cgi/route.ts`
2. `src/lib/services/registration-service.ts`
3. `src/app/(web)/auth/verify/page.tsx`
4. `src/app/(web)/layout.tsx`
5. `src/lib/services/auth-service.ts` (`getLayoutAuthStatus`)
6. 可能なら専用昇格 API / 本登録導線 UI
7. BDD / 単体テスト / E2E

#### 既存テスト影響

- 次の既存テストは仕様変更により落ちる見込み。
  - `PAT認証後は Cookie で認証され PAT は認証処理に使われない`
  - `Cookie有効の場合、PATで認証せず loginWithPat は呼ばれない`

- これは回帰ではなく、仕様変更に伴う期待値更新。

## Phase 3: 再発防止

### Q7. どうすれば防げていたか？

#### 設計

- チャネル分離を導入した時点で、「`senbra` トークンを Web に持ち込んだ後の正規遷移」を設計すべきだった。
- Header 表示と API 権限制御は同一の認可モデルを参照すべきだった。

#### テスト

- 次の E2E があれば検出できた。
  - 専ブラ認証 URL → ブラウザ認証 → 本登録 → 専ブラ投稿継続
  - Cookie 分裂後に PAT で本人へ復帰

#### プロセス

- チャネルガード追加時に「UI 導線の整合性チェック」をレビュー観点に入れるべきだった。
- 専ブラ互換の仕様レビューで、`Cookie喪失` だけでなく `Cookie分岐` も異常系として扱うべきだった。

### Q8. 今後の再発防止策は？

#### 防止

1. 認可モデルを UI と API で共通化する
   - Header 表示判定に `channel` を含める
   - 「表示できるリンク」と「実際に使える API」を一致させる

2. `senbra` → `web` 昇格フローを明示的に設計する
   - 認証済み専ブラユーザーが通常ブラウザで本登録・設定変更へ入る正規経路を用意する

3. PAT の責務を再定義する
   - 「Cookie 喪失時のみ」なのか
   - 「Cookie 分岐時の正本」なのか
   - ここを曖昧にしない

#### 検出

1. E2E を追加
   - 専ブラ認証 URL からの本登録
   - PAT による別人 Cookie 上書き復帰

2. Route Handler テストを追加
   - stale cookie + PAT
   - valid but wrong-user cookie + PAT

3. UI テストを追加
   - `senbra` token のとき Header が `mypage` を表示しないこと

### Q9. 他にも同じ構造の問題がないか？

#### 発見した同型パターン

1. `mypage` 系 API 全般は `web` チャネル必須だが、Layout / Header はチャネル非考慮
   - `history`, `username`, `upgrade`, `theme`, `vocabularies`, `copipe`, `auth/pat`
   - したがって「見えているのに使えない」導線不整合は `mypage` 以外にも波及しうる

2. `register` 系 API は `channel` ガードがない
   - これは現状では救済策でもあるが、設計意図が明文化されていない
   - 将来の保守で `mypage` と同じノリで `web` ガードを追加すると、専ブラからの本登録自体が完全に詰む

#### 対処要否

- 高優先度で対処すべき。
- 理由:
  - 現在の不整合は単発ではなく、チャネル分離と Web UI 導線の設計不一致という構造問題だから。

## まとめ

今回の事象は、ユーザーの誤操作を主因とみなすのは不適切である。
現行システムは、

1. 専ブラの `edge-token` をブラウザへ引き継ぐ導線を提供しつつ
2. そのトークンで Web UI の主要画面を拒否し
3. Cookie 分裂後の復帰手段として期待される PAT も Cookie 優先仕様で封じている

ため、設計・導線・仕様の複合不整合として扱うべきである。

実装方針としては、

1. Web 本登録導線へ自然接続できるようにする
2. Cookie 分裂後の復帰手段として PAT を正として上書き可能にする

の 2 本立てが妥当である。
