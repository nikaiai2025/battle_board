# 5ch/2ch互換アーキテクチャ・専用ブラウザ対応仕様 統合リサーチレポート

本ドキュメントは、ChMateやSikiといった専用ブラウザ（専ブラ）からアクセス可能な「5ch互換掲示板」を自作するための仕様・設計要件に関する3つのリサーチレポートを統合したものです。重複を排除しつつ、各リサーチからの重要な技術的詳細・実装方針・インフラ上の注意点を網羅しています。

---

## 1. 全体概要とディレクトリ構造

専用ブラウザは、登録された「板のURL」を基準に各ファイルのパスを自動推論し、ハードコードされた相対パスに対してGET/POSTリクエストを発行します。現代のRESTful API等ではなく、「ファイルパスの偽装」と「プレーンテキストの提供」によるエミュレーションが必要です。

**要求されるディレクトリ構造とエンドポイント**
```text
example.com/
 ├ bbsmenu.html            (GET) 板一覧（ディスカバリハブ）※ルート等任意の場所に配置
 ├ testboard/              ← 板のルートURL（専ブラに登録するURL）
 │  ├ subject.txt        (GET) スレッド一覧インデックス
 │  ├ SETTING.TXT        (GET) 掲示板コンフィギュレーション
 │  └ dat/
 │     ├ 1234567890.dat  (GET) 個別スレッドデータ
 │     └ 1234567891.dat
 └ test/
    └ bbs.cgi            (POST) 新規スレ作成・レス書き込み用トランザクション
      read.cgi/testboard/1234567890/ (GET) Webブラウザ用等の読み込みURL形式を要求される場合あり
```

## 2. 各ファイルのフォーマットとデータ構造

### 2.1 最大の前提：Shift_JIS（CP932）エンコーディングの絶対順守
専用ブラウザは、HTTPヘッダの文字エンコーディング指定に関わらず、すべての入出力を **Shift_JIS** として解釈します。サーバー内部（DBなど）はUTF-8で構築し、APIのレスポンス出力直前およびPOST入力直後の境界で動的なトランスコード（例: Node.jsなら `iconv-lite` の使用）を行う設計が必須です。これを怠ると、デリミタ（`<>`）が文字化けし、ブラウザのパース処理が破綻します。

### 2.2 bbsmenu.html（メニュー登録ハブ）
複数の板を専ブラのメニューに登録させるためのシンプルなHTML。
*   **構造**: `<b>`タグでカテゴリ名、`<a>`タグで板へのリンクを羅列する、余計な装飾のないプレーンなHTMLを出力します。
```html
<B>カテゴリ名</B><br>
<A HREF="https://example.com/testboard/">テスト掲示板</A><br>
```

### 2.3 SETTING.TXT（掲示板設定ファイル）
板の名前や挙動を専ブラに伝えるためのテキストファイル。改行区切りの `キー=値` 形式。
*   **主要項目**: `BBS_TITLE`(板名), `BBS_NONAME_NAME`(デフォルト名無し), `BBS_FORCE_ID`(ID強制), `BBS_MESSAGE_COUNT`(最大レス数。通常1000) など

### 2.4 subject.txt（スレッド一覧）
アクティブな全スレッドのリスト。1行1スレッドで、改行(LF)区切り。
*   **フォーマット**: `スレッドキー.dat<>スレッドタイトル (レス数)`
*   **スレッドキー**: 通常はスレッド作成時の10桁のUNIXタイムスタンプ。
*   **仕様上の注意**:
    *   文字列内に意図せず `<>` が含まれる場合はエスケープ必須。
    *   `(レス数)` の数値が実際のレス数と異なる場合、特に「実際の数より多い」値を記述すると、専ブラが無限ロードやクラッシュを起こす危険性があります。
    *   古いスレや1000レス到達スレは動的にこの一覧から除外し（過去ログ化・DAT落ち扱い）、メモリ肥大化を防ぎます。

### 2.5 .datファイル（スレッド本文データ）
個別スレッドの全レス生データ。1行が1レスに対応。区切り文字は `<>`。
*   **フォーマット**: `名前<>メール<>日付とID<>本文<>スレッドタイトル`
*   **フィールド詳細**:
    *   **名前**: `<b>`タグ等を含むことが可能。トリップ（パスワードハッシュ）を使用する場合は `</b>◆ozOtJW9BFA <b>` のように不可視タグでカプセル化します。
    *   **メール**: `sage` などのコマンドメタデータを格納。
    *   **日付とID**: 例 `2026/03/01(日) 12:34:56.78 ID:abcd1234`。厳密にフォーマットを守らないとID抽出機能等が崩壊します。外部アカウント連携（`BE:xxxx`等）が末尾に付与される場合もあります。
    *   **本文**: 複数行は1行にシリアライズし、改行は `<br>` に変換。`<` `>` `"` は `&lt;` `&gt;` `&quot;` などのHTMLエンティティへエスケープ必須。
    *   **スレッドタイトル**: **1行目（1レス目）の末尾にのみ**格納し、2行目以降は空（`<>`で終端）にする必要があります。

## 3. 書き込みAPI (bbs.cgi) トランザクション仕様

専ブラからの書き込みは `application/x-www-form-urlencoded` 形式のPOSTリクエストです。

### 3.1 POSTパラメータ
*   `bbs` (板ID), `key` (スレッドID, 新規作成時は空), `time` (専ブラ取得のUNIXタイム, スパム検証用)
*   `FROM` (名前), `mail` (メール欄), `MESSAGE` (本文)
*   `submit` (送信ボタンキャプション。無いと弾く仕様があるため必須)
*   `subject` (スレタイ。新規スレッド作成時のみ)

### 3.2 成功・失敗のレスポンス判定（ステートマシン）
専ブラはHTTPステータスコードに加え、返却されるHTML文字列のパターンマッチングでトランザクション状態を判定します。
*   **成功**: `<title>`に `書きこみました` を含み、2ch用コメントアウトタグを適切に配置する。
*   **エラー**: `ＥＲＲＯＲ` をタイトルにし、本文の `<b>` タグ内や `ERROR:` という文字列でエラー理由を出力すると専用のポップアップで通知されます。
*   **クッキー確認（2フェーズコミット）**: CSRFや連投防止のための「書き込み確認画面」。Cookie（PON/SPID）を発行し、`<title>書き込み確認</title>`を返すと、専ブラはそのCookieを付与してPOSTを自動再送します（実装緩和も可能）。
*   **クールダウン**: `お茶でも飲みましょう。`

## 4. ネットワークインフラと最適化要件

### 4.1 差分同期（チャンク転送）の実装 [最重要]
専ブラはレスポンスサイズ削減のため、前回取得したデータ量を基に `If-Modified-Since` および `Range` ヘッダ（例: `Range: bytes=15024-`）を付与してリクエストします。
*   更新がない場合: `304 Not Modified` を返却する。
*   更新がある場合: `206 Partial Content` とともに、リクエストされたバイトオフセット以降の**差分文字列のみ**を返却する。全体を `200 OK` で返すと、クライアント側で既存データに丸ごとマージされ重複描画バグが発生します。

### 4.2 インフラストラクチャにおける罠
*   **HTTPSリダイレクトとPOST消失**: `bbs.cgi` への通信に対して安易に 301/302 リダイレクトをかけると、POSTペイロードが消失し、書き込みが失敗します。307/308を導入するか、事前のHTTPSスキーム登録を徹底させる、もしくは特定のパスのみリダイレクトから除外する設定が必要です。
*   **WAFの誤検知**: 専ブラ特有のUser-Agent（`Monazilla/1.00`, `Mozilla/5.0` の特殊な模倣など）が各種WAFやCloudflare等にボットとして弾かれる可能性が高いです。ホワイトリスト化などのチューニングが必須となります。

### 4.3 専ブラごとのアーキテクチャ特性
*   **ChMate (Android)**: モバイル環境特有の最適化が強力。バックグラウンドで `subject.txt` を頻繁にポーリングするため、DBを都度クエリせずキャッシュファーストな提供処理が求められます。
*   **Siki (PC)**: `ワークスペース`機能などにより、同時多発的な並列リクエスト（バースト的なGET）が発生します。Keep-Aliveの最適化やデータベースのコネクションプーリングが重要になります。

## 5. 推奨される実装スタック (Next.js + Supabase) のアプローチ

Next.js と Supabase を利用した現代的システムへ適応させるベストプラクティス：

1.  **データ保存 (Supabase)**:
    `.dat` ファイルを Supabase Storage 等に物理保存して追記していくのは管理が煩雑です。`posts`テーブル（RDB）にデータを保持し、Next.js API Routes (Route Handlers) が必要なフォーマットで動的にDATテキストを構築する（SSR / 動的API）モデルを推奨します。RLSポリシーを用いて適切な権限管理を行います。
2.  **エンコーディング (Node.js)**:
    APIでのレスポンス生成時に `iconv-lite` で `Shift_JIS` バッファを生成し、`Content-Type: text/plain; charset=Shift_JIS` とともに送出します。
3.  **パフォーマンス対策 (Vercel)**:
    サーバーレス環境のコールドスタートや、数千レスのテキスト組み立てによる負荷を抑えるため、ISR機能の併用、エッジキャッシュ機構 (CDN層での応答) や `Partial Content` 対応ロジックの作り込みが重要です。
4.  **ルーティングによる偽装**:
    実際のファイルシステム構造を構築する必要はなく、Next.jsの動的ルーティング機能（例: `app/api/[board]/dat/[key].dat/route.ts` 等）でURL要件を満たします。フロントエンド（Web画面用）と専ブラ（API用）のルーティングを明確に切り分けることで効率的に開発可能です。
