# BDD中心のAIエージェンティック開発プロセス

---

# 第1章　概要

---

## 1-1. 開発プロセスの全体像

本プロセスは「ビジネス側（人間）」と「AIエージェント群」が明確な役割分担のもとで協働する開発フレームワークである。要件定義から検証・リリースまでを5フェーズに分割し、各フェーズの境界に「人間承認ゲート（G）」を設置することで、AIの自律的な動作と人間による品質管理を両立させる。

```
┌─────────────────────────────────────────────────────────────────────┐
│                         開発プロセス全体像                            │
│                                                                      │
│  フェーズ1          フェーズ2          フェーズ3                       │
│  要件定義    ─G1─  外部仕様定義  ─G2─  内部仕様定義  ─G3─            │
│  [人間主体]         [アーキテクトAI]    [アーキテクトAI]                │
│                                                                      │
│              フェーズ4              フェーズ5                         │
│         ──  実装      ─G4(ESC)─  検証・受け入れ  ─G5─  リリース      │
│             [コーディングAI]         [BDDゲート/コードレビュー/ドキュメントレビューAI]  │
└─────────────────────────────────────────────────────────────────────┘

G1: 要件承認       G2: 外部仕様承認    G3: 技術設計承認
G4: エスカレーション（動的発生）        G5: 受け入れ承認
```

## 1-2. 登場人物と役割

### ビジネス側（人間）

| 役割 | 主な責務 |
|---|---|
| ビジネス責任者 | 要件の最終意思決定、BDDシナリオの承認（G2）、G1・G5の承認 |
| 技術リード（人間） | 技術設計の承認（G3）、ユーザーストーリー作成、BDDシナリオの承認（G2）、エスカレーション対応（G4） |

### AIエージェント群（多層構造）

AIエージェント群は**オーケストレーター層**と**ワーカー層**の2層構造をとる。オーケストレーターAIがタスクを分解してワーカーAIに委譲し、ワーカーAIの成果物・進捗を統合・監視する。各エージェントはセッション間の記憶を持たず、`tmp/`ディレクトリのファイルを通じて状態を共有する。


| エージェント | 層 | 主な責務 | 主な担当フェーズ |
|---|---|---|---|
| オーケストレーターAI | オーケストレーター | スプリント管理・タスク分解・エージェント間調整 | フェーズ4〜5全体 |
| アーキテクトAI | ワーカー | BDDシナリオ生成・各種仕様書・設計ドキュメント作成（フェーズ3）、必要に応じて修正（フェーズ4） | フェーズ1〜4 |
| コーディングAI | ワーカー | BDDシナリオ駆動での実装、単体テスト作成・実行（対象領域のみに限定） | フェーズ4 |
| BDDゲートAI | ワーカー | BDDシナリオ全件の自動実行とレポート作成 | フェーズ5 |
| コードレビューAI | ワーカー | コード品質・横断的制約違反・スコープ逸脱チェック | フェーズ5 |
| ドキュメントレビューAI | ワーカー | 成果物ドキュメントの整合性等のチェック | フェーズ5 |

#### モデル選択指針

エージェントの役割に応じて、使用するLLMのクラスを使い分ける。

| モデルクラス | 選択基準 | 対象エージェント |
|---|---|---|
| **推論力重視**（高精度・低速） | 設計判断・品質評価など、誤判断の影響が大きい役割 | オーケストレーターAI、アーキテクトAI、コードレビューAI、ドキュメントレビューAI |
| **速度・コスト重視**（高速・低コスト） | 明確な仕様に基づく実行が主な役割 | コーディングAI、BDDゲートAI |

- エスカレーション解決後に再アサインするタスクは、問題の複雑さに応じて推論力重視モデルへの昇格を検討する
- モデルの具体的な選定はプロジェクト開始時にTDRに記録する

> ワーカーAIへの指示はすべて`tmp/tasks/task_{TASK_ID}.md`を介して行う。オーケストレーターAIが直接ワーカーAIの出力を書き換えることはしない（tmpファイル経由の非同期指示が原則）。ワーカーの作業進捗・テスト結果はタスク指示書内の`## 作業ログ`セクションに追記する。ファイル出力が必要な場合（仕様書ドラフト・テストエビデンス等）のみ`tmp/workers/`配下のタスク別ディレクトリを使用する。

## 1-3. フェーズ別の進め方

### フェーズ1：要件定義（人間主体）

ビジネス側が要件を定義する。重要な変化は「AIが読む前提で構造化する」点である。自然言語の箇条書きではなく、スコープ外・優先度・否定ケースを明示した構造化フォーマットで記述する。人間とアーキテクトAIが協働して作成。

**主要成果物：** 要件定義書（ビジネス要件・ユーザーストーリー・非機能要件を統合）、ユビキタス言語辞書

**G1（要件承認）：** ビジネス責任者が承認。承認後のドキュメントはAIが変更不可となる。

---

### フェーズ2：外部仕様定義（アーキテクトAI主体・人間レビュー）

#### PoC（技術検証）の実施

フェーズ2の冒頭で、キーとなる機能について**PoC（Proof of Concept）を実施する**。PoCは本番コードとは完全に分離した**使い捨ての環境**で行い、必要最小限の機能に絞って技術的な実現可能性を検証する。

**PoCの目的**
- 技術的リスクの早期発見（選定技術の制約、外部サービス連携の実現性、性能要件の達成可能性など）
- フェーズ2以降の仕様策定に必要な技術的知見の獲得

**PoC対象の決定**
- PoC対象は**AI（アーキテクトAI）と人間（技術リード）が協議して決定する**
- リスクが高い技術要素、未知の外部サービス連携、非機能要件の達成に不確実性がある領域を優先的に選定する

**PoCの進め方**
- 使い捨て環境（サンドボックス・検証用プロジェクト等）で実施し、本番のプロジェクト成果物には含めない
- 検証項目ごとに「成功基準」を事前に定義し、結果を記録する
- PoCの成果（得られた知見・制約事項）はフェーズ2以降の仕様策定およびフェーズ3のTDRに反映する
- PoC完了後、結果をPoC報告書（D-09）として記録する

**致命的な問題が発覚した場合**
- PoCで致命的な技術的問題（選定技術では要件を満たせない、外部サービスの制約により実現不可能など）が発覚した場合は、**フェーズ1（要件定義）に差し戻す**。要件の見直し・技術的前提の再検討を行い、G1の再承認を経てフェーズ2を再開する

---

G1承認済み要件を入力として外部仕様を生成する。**BDDが最も力を発揮するフェーズ。** D-01のユーザーストーリーを整理・構造化し、Gherkin記法のBDDシナリオとして定義する。ユーザーストーリーの内容（actor, goal, reason等）はBDDシナリオのFeature句の下に記載する。BDDシナリオの初期作成後は、BDDシナリオがリビングドキュメント（正）となり、以降ユーザーストーリー（D-01セクション2）の個別メンテナンスは行わない。人間とアーキテクトAIが協働して作成。

**主要成果物：** PoC報告書、BDDシナリオ集（UIの振る舞い含む）、OpenAPI仕様書、状態遷移仕様書、画面要素定義書

**G2（外部仕様承認）：** ビジネス責任者＋技術リードが承認。承認後のBDDシナリオは受け入れ基準として確定する。

---

### フェーズ3：内部仕様定義・システム設計（アーキテクトAI主体・人間レビュー）

外部仕様を満たすための技術的実現方針を決定する。フレームワーク選定などの技術的意思決定はアーキテクチャ設計書のTDRセクションに記録する。横断的制約はCLAUDE.mdに集約し、AIエージェントへの大域ルールもCLAUDE.mdに追加する。

**主要成果物：** アーキテクチャ設計書（技術的意思決定記録含む）、コンポーネント境界定義書、CLAUDE.md 更新（横断的制約含む）

**G3（技術設計承認）：** 技術リードが承認。

---

### フェーズ4：実装（オーケストレーターAI→コーディングAI、アーキテクトAI）
オーケストレーターAIが自律的にタスクを分解し、ワーカーにタスク指示を出す。人間への確認が必要な場合はG4エスカレーションを発生させる。

**主要成果物：** ソースコード、単体テストコード、単体テスト結果、エスカレーションログ
**G4（エスカレーション）：** イベント駆動で発生。


#### 実装手法：Double-Loop TDD

コーディングAIはBDDシナリオを外側ループ、単体テストを内側ループとする二重ループのTDDで実装を進める。

```
外側ループ（BDD）
  対象BDDシナリオを実行 → FAIL確認
    ├─ 内側ループ（単体テスト）
    │   1. 失敗する単体テストを書く（Red）
    │   2. テストが通る最小限のコードを実装（Green）
    │   3. コード品質を改善（Refactor）
    │   4. 次の単体テストへ（またはループ終了）
    │
  対象BDDシナリオを実行 → PASS確認
次のBDDシナリオへ
```

単体テストでは、正常系に加えて以下のカテゴリのエッジケースを網羅的にカバーする。プロジェクト特性に応じて該当するカテゴリを選択すること。

- null / 未定義値
- 空値（空文字列、空配列、空オブジェクト）
- 不正な型・フォーマット
- 境界値（上限・下限・ちょうど境界）
- 例外・エラー発生時の挙動
- 競合状態（並行処理がある場合）
- 大量データ
- 特殊文字・マルチバイト文字

#### 具体的な処理の流れ
- オーケストレーターAIとコーディングAIとの間の会話（指示および報告）は、簡潔に状況を示すシグナルのみとする。（具体的な作業進捗や作業指示はtmpディレクトリに配置したファイルを介して行う）。
- コーディングAIはCLAUDE.mdを最初に読み、コンテキストパッケージの指示に従い実装する。
- コーディングAIは上述のDouble-Loop TDDに従い、単体テストも含めて実装する。
- コーディングAIは実装後に単体テスト・該当部分のBDDシナリオテストを実施し、問題があればその場で修正と再テストを繰り返す（３回までの試行。それで解消しない場合はエスカレーション実施）。
- コーディングAIの作業進捗はタスク指示書（`task_{TASK_ID}.md`）の`## 作業ログ`セクションに逐次追記する。
- コーディングAIが自己判断で進めることができない状況ではエスカレーションを起票し、タスク指示書の作業ログ内チェックポイントに中断状態を記録した上で、オーケストレーターAIに報告して作業を終了する。
- オーケストレーターAIは、コーディングAIのエスカレーションを確認し、対応方針を決定する。
  - システム上の問題（コーディングAIが解消できないバグなど）に関してはアーキテクトAIを呼び出し、システム全体を俯瞰した上での設計上の問題点・改善点がないかを確認させる。
  - 必要な場合はアーキテクトAIに仕様書の変更指示を出し、完了後に人間の承認を得る（G4エスカレーション発生）
  - オーケストレーターAIが自己判断できない場合もG4エスカレーションを発生させる
  - エスカレーション解決後、オーケストレーターAIは既存タスク指示書に解決内容を追記してステータスを`assigned`に戻し、同一TASK_IDでコーディングAIを再起動する。コーディングAIはタスク指示書の作業ログ内チェックポイントから中断地点を復帰する


#### オーケストレーターAIへの権限移譲
オーケストレーターAIには、人間が判断すべき権限の一部を委譲する場合がある

権限移譲の例（チャットにて指示）
```
以下を**すべて満たす場合のみ**、オーケストレーターAIは人間承認なしで判断して継続すること。
仕様書の変更についてはアーキテクトAIに仕様書の修正指示を出すこと（この場合は仕様書の人間承認は省略）
その後、必要な場合は再度コーディングAIへ作業指示を出すこと。
- BDDシナリオ（`features/`）の変更を伴わない
- 外部API契約（`x-api-visibility: public` のエンドポイント）の変更を伴わない
- 内部API（`x-api-visibility: internal`）の変更は、以下をすべて満たす場合に限り許容する：
  - 呼び出し元のコードも同一スプリント内で同期更新される
  - 新規エンドポイント追加 or 既存の後方互換な変更のみ（破壊的変更は不可）
- 状態遷移仕様（D-05）の変更を伴わない
- CLAUDE.mdの横断的制約およびTDRに反しない

以下のいずれかに該当する場合、オーケストレーターAIは自己判断せず停止して人間に報告すること
- `features/`配下のBDDシナリオの追加・変更・削除
- 外部API契約（`x-api-visibility: public` のエンドポイント、リクエスト/レスポンス、HTTPステータス）の変更
- 内部APIの破壊的変更（エンドポイント削除、必須パラメータの変更等）
- セキュリティ・法規制・データ保管方針など横断的制約への影響
- TDRの追加・変更、または新規依存関係の導入
- ユーザーから見た振る舞いが変わる可能性がある場合

> **注：** `x-api-visibility`が未指定のエンドポイントは`public`（安全側）として扱う。
```

### フェーズ5：検証（BDDゲートAI・コードレビューAI・ドキュメントレビューAI主体・人間最終確認）

外部BDDシナリオの全件自動実行がメインの検証活動となる。「G2で合意したシナリオが全件PASS」であることがG5承認の主たる根拠となる。
このフェーズでは各AIはレポートの作成のみを行い、**その他の成果物（ソース及びドキュメント）への変更は禁止**。レビューで検知された問題は人間・オーケストレーターAIへエスカレーションする。
BDDゲートAIはBDDシナリオ全件実行してレポートを作成する。
コードレビューAIはコード品質をチェックする。
ドキュメントレビューAIはドキュメントの整合性と最新性をチェックする。

**主要成果物：** BDDシナリオ実行レポート、コードレビューレポート、ドキュメントレビューレポート

**G5（受け入れ承認）：** ビジネス責任者が承認。BDDシナリオ全件PASS・非機能要件達成（人間が確認）・エスカレーション全件解決が条件。

#### フェーズ5でFAILが発生した場合の差し戻しフロー

フェーズ5のレビューで問題が検出された場合、以下の手順でフェーズ4に差し戻す。

1. **レポート作成：** BDDゲートAI・コードレビューAI・ドキュメントレビューAIがそれぞれのレポート（L-04〜L-06）に問題を記録する
2. **問題分析：** オーケストレーターAIがレポートを読み込み、修正に必要なタスクを特定する
3. **差し戻しスプリント起動：** オーケストレーターAIが修正タスクを新スプリントとしてフェーズ4に差し戻す。差し戻しスプリントの計画には、元のFAILレポートへの参照を含める
4. **修正完了後の再検証：** 差し戻しスプリント完了後、再度フェーズ5のレビュー（BDDゲート・コードレビュー・ドキュメントレビュー）を全件実行する

> **注：** フェーズ5のAIはレポート作成のみを行い、成果物（ソース・ドキュメント）の修正は行わない。修正はフェーズ4への差し戻しを通じてのみ実施する。

## 1-4. 実装根拠の管理方針

「なぜこのコードが存在するか」を明確にするため、コードには対応するfeatureファイルとシナリオタグをコメントで参照する。番号の採番・管理コストをなくし、シナリオファイルを読めば振る舞いの意図が直接わかる形にする。

```python
# See: features/order_purchase.feature @ORD-1
def create_order(...):
    ...

```

リファクタリングは既存のBDDシナリオが全件PASSし続けることで正当性を担保する。バグ修正はシナリオを先に追加・修正してからコードを直すことで根拠を明確にする。複数シナリオに関わる共通ロジックは複数のfeatureファイルを列挙する。

## 1-5. エージェント間通信：tmpディレクトリ

AIエージェントはセッションをまたいで状態を保持しない。複数エージェントの協調動作・作業再開（アベンド時の復帰）を可能にするため、**プロジェクトルートの`tmp/`ディレクトリをエージェント間の共有状態管理領域として使用する。**

### ディレクトリ構造

`tmp/`はロール別にサブディレクトリを分割し、ファイルの混在と並行作業時の衝突を防止する。

```
project-root/
  tmp/
    ├── orchestrator/                       ← オーケストレーター専用（L-07）
    │   ├── sprint_current.md               ← L-07 直近の状況＋アーカイブインデックス
    │   ├── sprint_{N}_plan.md              ← L-07 進行中スプリント計画・結果
    │   └── archive/                        ← L-07 過去スプリントの原本（サマリー付き完全保存）
    │       ├── sprint_001_010.md
    │       └── ...
    │
    ├── tasks/                              ← タスク指示書（オーケストレーター作成）
    │   └── task_{TASK_ID}.md               ← L-01 コンテキストパッケージ（指示＋作業ログ一体型）
    │
    ├── workers/                            ← ファイル出力が必要な場合のみ使用（TASK_IDで分離）
    │   ├── coding_{TASK_ID}/               ← コーディングAI：テストエビデンス等
    │   │   └── (テストの詳細ログ等、タスク指示書に収まらないファイル出力)
    │   └── architect_{TASK_ID}/            ← アーキテクトAI：仕様書ドラフト等
    │       └── draft/                      ← 仕様書ドラフト（承認前一時領域）
    │
    ├── escalations/                        ← エスカレーション専用
    │   ├── escalation_{ESC_ID}.md          ← L-03
    │   └── resolved/                       ← 解決済み（アーカイブ）
    │
    └── reports/                            ← フェーズ5レビューレポート
        ├── bdd_report.md                   ← L-04
        ├── doc_review.md                   ← L-05
        └── code_review.md                  ← L-06
```

### ロール別アクセス権限

各エージェントは自分の書き込み権限があるディレクトリのみを変更する。他エージェントの作業空間を書き換えることは禁止する。

| ディレクトリ | 書き込み権限 | 読み取り権限 |
|---|---|---|
| `orchestrator/` | オーケストレーターAIのみ | 全エージェント |
| `tasks/` | オーケストレーターAI（指示セクション）、ワーカーAI（ステータス更新＋`## 作業ログ`セクションへの追記のみ） | 割り当てられたワーカー |
| `workers/{role}_{TASK_ID}/` | 該当タスクのワーカーAIのみ（ファイル出力が必要な場合のみ作成） | オーケストレーターAI |
| `escalations/` | 起票：ワーカーAI、応答追記：人間 | オーケストレーターAI、該当ワーカー |
| `reports/` | 各レビューAI（自分の担当レポートのみ） | 人間、オーケストレーターAI |

#### ワーカーAIの書き込み権限原則

ワーカーAIが書き込み可能な領域は以下に限定される。これ以外のディレクトリは読み取り専用とする。

- **タスク指示書の`## 作業ログ`セクション**（進捗記録・チェックポイント・テスト結果サマリーの追記）
- **タスク指示書のステータス更新**（フロントマターの `status` フィールドのみ）
- **成果物出力ディレクトリ**（タスク指示書の `artifacts_dir` で指定されたパス。ファイル出力が必要な場合のみ）
- **エスカレーション**（`tmp/escalations/` への新規起票のみ）
- **実装対象ファイル**（`locked_files` に記載されたファイル。コーディングAIのみ）
- **成果物ディレクトリ**（`docs/`, `features/` 等。アーキテクトAIのみ。タスク指示書で指定された出力先）

### ワーカー空間の分離（TASK_IDによる管理）

ワーカーAIインスタンスとタスクは1:1の関係にある（1ワーカーが同時に複数タスクを実行しない）。このため、TASK_IDがそのままワーカー空間の識別子となり、別途WORKER_IDを管理する必要はない。

- **作業進捗はタスク指示書に一体化する。** ワーカーはタスク指示書（`task_{TASK_ID}.md`）の`## 作業ログ`セクションに進捗・チェックポイント・テスト結果サマリーを追記する。指示と進捗が1ファイルに集約されるため、コンテキスト把握が容易になる
- **ファイル出力が必要な場合のみ**、オーケストレーターAIがアサイン時に`workers/{role}_{TASK_ID}/`ディレクトリを作成し、タスク指示書の`artifacts_dir`にパスを記載する。ファイル出力が不要なタスク（典型的なコーディングタスク等）では`artifacts_dir`を省略し、workersディレクトリも作成しない
- ファイル出力が必要なケースの例：アーキテクトAIの仕様書ドラフト（`draft/`）、テストの詳細ログ・エビデンス等
- **異常終了（aborted）時のリトライ**は新しいTASK_IDを振ることで旧タスク指示書・旧ディレクトリとの衝突を回避する
- **エスカレーション解決後の継続**は同一TASK_IDを再利用する。オーケストレーターAIが既存タスク指示書に解決内容を追記し、ステータスを`assigned`に戻した上で、同一TASK_IDでワーカーを再起動する。ワーカーはタスク指示書の作業ログ内チェックポイントを参照して中断地点から作業を継続する

### 並行衝突防止（タスク指示書内管理方式）

複数のコーディングAIが同時にスポーンされた場合の衝突防止は、タスク指示書内の`locked_files`リストで管理する。

- `tmp/tasks/task_{TASK_ID}.md` の `locked_files` リストが排他制御の**唯一のソース**
- オーケストレーターAIはアサイン前に全アクティブタスク（status が `assigned` または `in_progress`）の `locked_files` を照合し、重複するファイルがあれば直列化する
- 新規ファイル作成は `[NEW]` プレフィックスで宣言する
- ワーカー異常終了時：オーケストレーターがタスクのステータスを `aborted` に更新し、locked_files を解放扱いとする

### タスク指示書のステータス管理

タスク指示書の先頭にYAMLフロントマターを付与し、ステータスを管理する。

```yaml
---
task_id: TASK-001
sprint_id: Sprint-1
status: in_progress   # assigned | in_progress | completed | escalated | aborted
assigned_to: bdd-coding   # bdd-coding | bdd-architect | bdd-gate | bdd-code-reviewer | bdd-doc-reviewer
artifacts_dir: tmp/workers/coding_TASK-001  # ファイル出力が必要な場合のみ指定（省略可）
depends_on: []        # 先行TASK_IDのリスト（なければ空）
created_at: 2024-04-20T09:00:00
updated_at: 2024-04-20T09:30:00
locked_files:
  - src/services/order_service.py
  - "[NEW] src/services/refund_service.py"
---
```

| ステータス | 意味 |
|---|---|
| assigned | オーケストレーターが作成済み。ワーカー未着手（エスカレーション解決後の再アサインも含む） |
| in_progress | ワーカーが作業中 |
| completed | ワーカーが完了報告済み |
| escalated | エスカレーション発生中。ワーカーは作業を中断して終了済み。人間またはオーケストレーターの判断待ち |
| aborted | 異常終了。オーケストレーターがロック解放済み。リトライ時は新TASK_IDで再作成する |

### アーカイブのサマリー併記方式

スプリントの原本は完全保存する。ただし各スプリントの先頭に3行サマリー（目的・主要成果・未解決事項）を追加し、オーケストレーターは通常サマリーのみを読む。詳細が必要な場合のみ原本を参照する。

```markdown
# Sprint-005 サマリー
- 目的: 注文作成API＋在庫確認APIの実装
- 主要成果: TASK-010〜012完了、BDD @ORD-1〜3 PASS
- 未解決: ESC-007（決済タイムアウト閾値の確定待ち）

## Sprint-005 詳細（以下、原本）
...
```

`tmp/`ディレクトリの構造・用途・アクセス権限・ファイル命名規則はCLAUDE.mdに明記し、すべてのエージェントが参照する。

## 1-6. 人間承認ゲートの設計

| ゲート | 発生タイミング | 承認者 | 主な確認内容 |
|---|---|---|---|
| G1 | 要件定義完了後 | ビジネス責任者 | 要件の網羅性・スコープ・非機能数値 |
| G2 | 外部仕様定義完了後 | ビジネス責任者＋技術リード | BDDシナリオの網羅性・状態遷移・異常系 |
| G3 | 内部仕様定義完了後 | 技術リード | アーキテクチャ設計書（TDRセクション含む）・CLAUDE.md（横断的制約含む）の妥当性 |
| G4 | AIが判断不能時（随時） | 技術リード | エスカレーション内容の判断・仕様確定 |
| G5 | 検証完了後 | ビジネス責任者 | BDD全件PASS・非機能達成・ESC全解決 |


停止時の応答はチャットへの直接指示、またはエスカレーションファイルへの人間による追記のいずれかで行う。AIは次回起動時にエスカレーションファイル（`tmp/escalations/`）を確認し、応答があればタスク指示書（`task_{TASK_ID}.md`）の作業ログ内チェックポイントを参照して作業を再開する。応答がなければ再度停止して待機する。

### G2の重要性とAI自律運用の前提

全ゲートのうち**G2（外部仕様承認）が最も重要**である。G2で承認されたBDDシナリオは以降の全工程（設計・実装・検証）の「契約」として機能するため、G2の品質がプロジェクト全体の成否を決定する。人間はG2において全BDDシナリオを精読し、網羅性・正確性・否定ケースの妥当性を検証しなければならない。

G2承認後のフェーズ3〜5は**AI自律実行を基本**とし、問題発生時のエスカレーション（G4）で人間が介入するモデルで運用する。G3・G5は「AIが自律実行した結果を人間が差分レビューする」形で運用可能である。すなわち、人間の注力配分は以下のようになる。

| ゲート | 人間の関与度 | 運用の実態 |
|---|---|---|
| G1 | 高 | 人間が主体的に要件を定義 |
| **G2** | **最高** | **全BDDシナリオを人間が精読・承認。ここが防衛線** |
| G3 | 中〜低 | AI生成の設計書を差分レビュー。問題なければ承認 |
| G4 | イベント駆動 | AIが判断不能な場合のみ発生 |
| G5 | 低〜中 | AIが生成したレポート（L-04〜L-06）を確認。BDD全件PASSなら承認 |

> **根拠：** AIが最も間違えやすいのは「何を作るか」の判断であり、「どう作るか」の実行は比較的得意である。仕様の正しさ（G2）を人間が担保すれば、実現方法（G3〜G5）はAIに委ねても品質を維持できる。

変更したドキュメントより下流に位置するドキュメントは必ず連動更新する。
BDDは振る舞いの意図の正本、D-05は状態名・遷移条件の正本である。
衝突時の解決順序：まずBDDで振る舞いの意図を確定し、次にOpenAPI/D-05を同一変更で同期する。

## 1-8. スプリント管理

オーケストレーターAIが自律的にスプリントを進行し、特定の条件時のみ人間に確認する設計とする。

### スプリントサイクルの全体像

```
┌─────────────────────────────────────────────────────────────────┐
│  Sprint N                                                        │
│                                                                  │
│  1. 状況把握          2. スコープ確定       3. 並行アサイン      │
│  オーケストレーター    オーケストレーター     オーケストレーター   │
│  ・sprint_current.md  ・重要度基準に従い     ・locked_files照合  │
│  ・features/配下        課題をピックアップ   ・依存関係がない    │
│    を読み込む           ・sprint_N_plan.mdに   　課題は並行起動    │
│  　                       記録                                     │
│                                                                  │
│  4. ワーカー実装                5. 結果取りまとめ               │
│  ・各ワーカーが独立して実装      ・全ワーカーの報告を集約         │
│  ・G4エスカレーションは随時      ・sprint_N_plan.mdに結果記録     │
│  　オーケストレーターに報告      ・人間確認条件を判定             │
│                                  　→ 確認不要ならSprint N+1へ   │
└─────────────────────────────────────────────────────────────────┘
```

---

### ステップ1：状況把握

オーケストレーターは以下を読み込み、プロジェクトの現状を把握する。

```
tmp/orchestrator/sprint_current.md  ← 直近スプリントの結果・未解決事項
tmp/orchestrator/archive/           ← 過去スプリントの原本（必要に応じて参照）
features/                           ← 現在のBDDシナリオ（何がPASSしているか）
```

---

### ステップ2：スコープ確定

オーケストレーターは以下の重要度基準に従い課題を自律的にピックアップし、`tmp/orchestrator/sprint_N_plan.md`に記録する。**ここにヒューマンゲートは置かない。** 重要度判定の権限はオーケストレーターに委譲されている。

**重要度判定基準（優先度の高い順）**

| 優先度 | 基準 | 具体例 |
|---|---|---|
| 最高 | 他の多くの課題がブロックされている | 認証基盤・DB接続・共通エラーハンドリング |
| 高 | BDDシナリオが存在するが未実装 | 承認済みシナリオのPASS率が低い機能 |
| 中 | 前スプリントからの継続課題 | G4で中断されたタスクの再開 |
| 低 | 独立した機能追加 | 他機能に影響しない新規エンドポイント |

```markdown
# tmp/orchestrator/sprint_001_plan.md

## Sprint-001 計画
作成日時: 2024-04-20 09:00  作成者: オーケストレーターAI

## スコープ（重要度基準による自律選定）
| タスク | 理由 | 並行可否 |
|---|---|---|
| TASK-001: 注文作成API実装 | BDDシナリオ未実装・基幹機能 | TASK-002と並行可 |
| TASK-002: 在庫確認API実装 | TASK-001の依存コンポーネント | TASK-001と並行可 |
| TASK-003: 認証ミドルウェア | TASK-001・002両方がブロック中 | 単独先行 |

## 実行順序
Phase-A（単独）: TASK-003
Phase-B（並行）: TASK-001, TASK-002

## 結果（スプリント完了後に記録）
```

---

### ステップ3：並行アサインと競合管理

オーケストレーターはアサイン前に各タスクの`locked_files`を照合し、重複するファイルを複数タスクが同時に変更しないよう直列化する。

```
TASK-001の locked_files: [order_service.py, orders.py]
TASK-002の locked_files: [inventory_service.py, inventory.py]
→ 重複なし → 並行アサイン可

TASK-004の locked_files: [order_service.py]  ← TASK-001と重複
→ TASK-001完了後にTASK-004をアサイン
```

---

### ステップ4：ワーカー実装とG4処理

各ワーカーは`tmp/tasks/task_{TASK_ID}.md`の指示に従い独立して実装する。エスカレーションが発生した場合、オーケストレーターが状況を判断し別AIへのタスク指示または人間へのG4エスカレーションを行う。

---

### ステップ5：結果取りまとめと人間確認判定

オーケストレーターは全ワーカーの報告を`tmp/orchestrator/sprint_N_plan.md`の結果欄に集約し、以下の条件で人間確認の要否を判定する。

**人間への確認が必要な条件（いずれかに該当する場合）**

```
条件1: ワーカーからのエスカレーションが1件以上発生し、かつオーケストレーターAIが判断できないものが1件以上ある場合
条件2: BDDシナリオ（features/配下）に変更が生じた
```

**条件に該当しない場合（全PASS・全DONE・変更なし）：** オーケストレーターは自律的にSprint N+1を開始する。

---

### ステップ6：スプリントログの管理（インデックス＋アーカイブ）

スプリントが蓄積したらアーカイブに移動する。要約はせず**原本を完全保存**し、インデックスのみを更新する。これにより過去の判断をオーケストレーターが直接参照できる。

```
tmp/orchestrator/
  sprint_current.md          ← 直近の状況サマリー＋アーカイブインデックス
  sprint_{N}_plan.md         ← 進行中スプリントの計画・結果
  archive/
    sprint_001_010.md        ← Sprint 1〜10の原本をそのままアーカイブ（各スプリント先頭にサマリー付き）
    sprint_011_020.md        ← Sprint 11〜20の原本をそのままアーカイブ（各スプリント先頭にサマリー付き）
```

`sprint_current.md`の先頭にはアーカイブへのインデックスのみを記載する。

```markdown
# tmp/orchestrator/sprint_current.md

## 過去スプリントの参照先
| 期間 | ファイル | 主な決定事項・ESC |
|---|---|---|
| Sprint 001〜010 | tmp/orchestrator/archive/sprint_001_010.md | 注文フロー基盤完了・ESC-001〜003 |
| Sprint 011〜020 | tmp/orchestrator/archive/sprint_011_020.md | 決済統合完了・TDR-003追加 |

## Sprint-021（最新）
...
```

## 1-9. 全エージェント大域ルール

全AIエージェントがタスク開始時に最初に読む「憲法」となるファイルをプロジェクトルートに作成する。（ClaudeCodeで開発を行う場合のファイル名は CLAUDE.md。他のプラットフォームの場合は適宜ファイル名を読み替える）。必読ドキュメント・禁止事項・エスカレーション条件・tmpディレクトリの使い方に加え、**破ってはならない横断的制約**を記載する。個別機能に関する制約は各仕様ドキュメントに記載する。必ず人間が編集し、AIは読み取り専用とする。

**記載内容の要点**

- プロジェクト概要と必読ドキュメントの場所
- 命名・コーディング規約
- **横断的制約（セキュリティ・法規制・プロジェクト全体に関わるルール）**
- 禁止事項（BDDシナリオの無断変更等）
- エスカレーション条件と手順
- tmpディレクトリの用途・ファイル命名規則・更新タイミング
- **技術的安全ガードの一覧（後述）**

### 技術的安全ガード

CLAUDE.mdに記載する禁止事項は「規約」であり、AIが無視または見落とすリスクがある。**重大な禁止事項はプラットフォームのHooks機構等を用いて技術的に強制すべきである。** 規約による制約と技術的強制による制約を明確に区別し、後者をCLAUDE.mdにも明記する。

| 制約の種類 | 手段 | 例 |
|---|---|---|
| **技術的強制**（違反を物理的に阻止） | Pre-Tool Hook（ツール実行前に検査・ブロック） | 破壊的操作（`rm -rf`等）のブロック、手動デプロイの禁止 |
| **技術的強制**（品質の自動適用） | Post-Tool Hook（ツール実行後に自動処理） | コードフォーマッタの自動適用、静的解析の自動実行 |
| **規約**（AIの自律的遵守に委ねる） | CLAUDE.md記載 | 命名規約、エスカレーション条件、スコープ制限 |

技術的強制の対象とすべき制約の選定基準：

- **違反時の影響が不可逆**（データ削除、本番環境への誤デプロイ等）
- **AIが「良かれと思って」違反しやすい**（リファクタリングによるスコープ外変更等）
- **検知が容易**（コマンドパターンやファイルパスで機械的に判定可能）

> **注：** 具体的なHooksの設定方法はプラットフォームに依存する。プロジェクト開始時に使用するAI開発プラットフォームのHooks機構を確認し、技術的に強制可能な制約を洗い出すこと。

## 1-10. バージョン管理とデプロイ

### 単一ブランチ運用

本フレームワークでは**単一ブランチ（main）での運用を推奨する**。並行作業の衝突防止はGitブランチではなく、§1-5で定義した`locked_files`による排他制御で実現する。

**根拠：** AIエージェントはマージコンフリクトの解決が苦手である（コンフリクト発生時にファイル全体の文脈を正しく把握できない）。ブランチを分けるとコンフリクトが不可避的に発生するが、単一ブランチ＋ファイル単位の排他制御であれば、**コンフリクトという問題自体を発生させない**設計となる。

### デプロイフロー

デプロイはコード変更のpushをトリガーとするプラットフォーム側の自動デプロイを基本とする。オーケストレーターAIの手順に以下を含める。

1. コーディングAIが単体テスト・BDDシナリオテストをローカルで実行しPASSを確認
2. オーケストレーターAIがcommit・pushを実行
3. プラットフォームが自動デプロイを実行
4. オーケストレーターAIがCLI等でデプロイ完了を確認
5. **本番環境での自動スモークテストを実行し、デプロイ後の疎通を検証する**

ステップ5は不可欠である。ローカルテストが全件PASSしても、デプロイ先プラットフォームの固有事情（リダイレクト挙動、エッジサーバーのキャッシュ、環境変数の差異等）により本番環境でのみ発生する障害がある。本番スモークテストはこの種の問題を検出する唯一の手段であり、D-10 BDDテスト戦略書のテスト層定義における「本番スモークテスト」層に相当する。

プロジェクト規模の拡大に伴い、push時にCI環境でテストを自動実行するパイプラインの追加を検討する。これはAIの自己申告とは独立した検証層として機能する。

# 第2章　背景

本フレームワークの設計はすべて、AIエージェントが人間エンジニアと根本的に異なる特性を持つという事実から導出されている。

### 特性1：暗黙知を持たない

人間エンジニアはプロジェクトの経緯・組織の文化・業界の常識を暗黙の前提として開発に活かす。AIエージェントにはこれが存在しない。**すべての前提は明示されなければならない。**

たとえば「適切なエラーハンドリングを行うこと」という仕様は人間エンジニアには伝わるが、AIエージェントには「適切」の定義がなければ独自解釈で実装される。

### 特性2：セッション間の記憶を持たない

AIエージェントは新しいセッションを開始するたびに、過去の会話・決定・文脈をすべて失う。「先日の議論でこう決めた」という共有記憶は存在しない。**すべての決定はドキュメントとして記録・注入されなければならない。**

### 特性3：仕様の穴を仮定で埋め、疑わずに進む

AIエージェントは曖昧な仕様に直面したとき、「質問して確認する」より「もっともらしい仮定で埋めて先に進む」傾向がある。この仮定は静かに、一貫して間違い続けることがある。人間エンジニアなら「何かおかしい」と気づくような矛盾も、AIはそのまま実装し続ける。

### 特性4：スコープを超えて実装することがある

「ついでにこちらも改善した」という形で、指示されていない機能・最適化・リファクタリングを実施することがある。これは一見好意的に見えるが、要件との対応関係が不明な実装が混入することを意味する。

### D-01　要件定義書

| 項目 | 内容 |
|---|---|
| 作成者 | 人間（ビジネス責任者・技術リード）、アーキテクトAIが構造化を補助 |
| フェーズ | 1 |
| 関連ゲート | G1（承認対象） |

**目的・役割**

プロジェクトの「なぜ作るか（WHY）」「誰が何をしたいか（WHAT）」「どの水準で作るか（HOW MUCH）」を1つのドキュメントに統合した最上位の要件定義文書。ドキュメントの単一ソース化の原則に基づき、要件に関する情報はすべてこのドキュメントに集約する。以下の3つのセクションから構成され、AIエージェントはこのドキュメントを読んでプロジェクトの目的・機能・品質基準を一貫して把握する。

**G1（要件承認）：** ビジネス責任者が承認。承認後のドキュメントはAIが変更不可となる。

---

#### セクション1：ビジネス要件（採番：REQ-XXX）

**記載内容の要点**

- ビジネスゴールと解決すべき課題
- 成功指標（定量値必須）
- **明示的なスコープ外**（AIがスコープ外を実装しないための安全装置）
- 前提条件・制約条件
- ステークホルダー一覧

**具体例（Markdown形式・要点のみ）**

```markdown
# REQ-001　商品購入フロー改善
## ビジネスゴール
カート放棄率を現状の68%から50%以下に改善する。
## 成功指標
- カート放棄率: 68% → 50%以下（リリース後3ヶ月で達成）
## スコープ
- 対象: 商品カート操作、購入フロー、在庫確認
- スコープ外（AIはこれらを実装しない）: 定期購入、ポイント・クーポン、外部モール連携
## 前提・制約
- 決済はStripeを使用（変更不可）
```

---

#### セクション2：構造化ユーザーストーリー

「誰が・何を・なぜ」の形式でシステムに求める機能を列挙する。従来のユーザーストーリーとの最大の違いは、`negative_cases`と`out_of_scope`が必須項目であること。AIは「書かれていないこと」を仮定で補完するため、明示的に除外事項を記述する。ユーザーストーリーは**自然文**で記述する（Gherkin記法は使用しない）。Gherkin記法によるBDDシナリオへの変換はフェーズ2で実施する。

**記載内容の要点**

- `actor`：誰が使うか
- `goal`：何を達成したいか
- `reason`：なぜ必要か
- `priority`：MoSCoW法（Must/Should/Could/Won't）
- `acceptance_criteria`：受け入れ条件（自然文で記述。フェーズ2でBDDシナリオに変換される）
- `negative_cases`：**必須。仕様に含む異常系・境界値**
- `out_of_scope`：**必須。このストーリーが明示的に扱わないこと**

**具体例（YAML形式・要点のみ）**

```yaml
user_stories:
  - id: US-001
    title: ユーザーが商品を購入する
    actor: 認証済みユーザー
    goal: カートの商品を注文確定する
    reason: 商品を購入して配送を受けるため
    priority: Must
    acceptance_criteria:   # ← 自然文で記述（Given/When/Thenは使用しない）
      - カートに在庫あり商品があるとき、購入操作を行うと注文が確定し注文IDが発行される
      - 決済情報が未登録の場合は購入できない
      - 購入完了後に確認メールが送信される
    negative_cases:       # ← 必須項目
      - 在庫切れ商品がカートに含まれている
      - 決済サービスがタイムアウトした
    out_of_scope:         # ← 必須項目
      - ゲストユーザーによる購入
      - 定期購入
```

> **BDDシナリオへの移行：** フェーズ2でアーキテクトAIがユーザーストーリーを入力としてBDDシナリオ（D-03）を作成する。BDDシナリオのFeature句の下にユーザーストーリーの要約（actor, goal, reason）を記載し、以降はBDDシナリオがリビングドキュメント（正）となる。ユーザーストーリーの個別メンテナンスは行わない。ユーザーストーリー一覧が必要な場合は、各featureファイルのFeature句に記載された情報を積み上げて作成する。

#### セクション3：非機能要件

性能・可用性・セキュリティ等の品質属性を定義する。AIエージェントが技術選定（TDR作成）と実装の判断基準として参照する。**すべての要件は定量値で記述する**。「高速であること」はAIに伝わらない。

**記載内容の要点**

- 性能要件（レスポンス時間、スループット、同時接続数）
- 可用性要件（稼働率、RTO、RPO）
- セキュリティ要件（認証方式、暗号化規格）
- スケーラビリティ要件
- 保守性要件（ログ、監視）

**具体例（YAML形式・要点のみ）**

```yaml
non_functional_requirements:
  performance:
    api_response: { p95_ms: 200 }   # 定量値必須。「高速」はNG
    concurrent_users: 10000
  security:
    authentication: OAuth2.0 + JWT
    pii_storage: EU域内サーバーのみ  # CLAUDE.md横断的制約に記載
```

### D-03　BDDシナリオ集（外部・UIの振る舞い含む）

| 項目 | 内容 |
|---|---|
| 作成者 | アーキテクトAI（人間レビュー） |
| フェーズ | 2 |
| 関連ゲート | G2（承認対象）→ 受け入れ基準として確定 |

**目的・役割**

本フレームワークの**最重要成果物**。G2で承認された後は「受け入れ基準」として確定し、G5の承認根拠となる。コーディングAIへの実装命令・BDDゲートAIへの検証基準・ビジネス側への合意証跡を兼ねる。

**単一ソース化の原則：** UIの振る舞い（ボタンの有効/無効、エラーメッセージの表示条件、画面遷移など）もBDDシナリオに集約する。画面要素定義書（D-06）には振る舞いを記述せず、本シナリオ集がUIの振る舞いの単一ソースとなる。

**リビングドキュメント方針：** BDDシナリオの初期作成後は、本シナリオ集がリビングドキュメント（正）となる。D-01のユーザーストーリー（セクション2）の個別メンテナンスは以降行わない。ユーザーストーリー一覧が必要な場合は、各featureファイルのFeature句に記載されたユーザーストーリー情報を積み上げて作成する。

**記載内容の要点**

- `Feature`：機能の概要とビジネスルール。**Feature句の直下に、D-01ユーザーストーリー由来の情報（As a〜 / I want to〜 / So that〜）を記載する。** これにより、ユーザーストーリーの内容がBDDシナリオに統合される
- `Background`：共通前提条件
- 正常系・異常系・境界値を網羅
- HTTPステータス・エラーコード等の具体値まで記述（AIが迷わないように）
- 否定的検証（「変化がないこと」「作成されないこと」）を明示
- **UIの振る舞い（表示条件・有効化条件・画面遷移）もシナリオとして記述する**
- **各シナリオに`@プレフィックス-連番`タグを付与する**（会話・ログ内での参照用。採番台帳は不要。）

**命名規則：** featureファイル名から2〜4文字のプレフィックスを決め、ファイル内で連番を振る。プレフィックスはfeatureファイルを見れば自明になるよう、ファイル名と対応させる。

| featureファイル | プレフィックス | 例 |
|---|---|---|
| order_purchase.feature | ORD | @ORD-1, @ORD-2 |
| cart.feature | CART | @CART-1, @CART-2 |
| auth.feature | AUTH | @AUTH-1 |

**フォーマット：** Gherkin（.featureファイル）

**具体例**

```gherkin
# features/order_purchase.feature

Feature: 商品購入
  As a 認証済みユーザー
  I want to カートの商品を注文確定する
  So that 商品を購入して配送を受けることができる

  ビジネスルール: 在庫がある商品のみ購入可能
  ビジネスルール: 決済完了後に注文が確定する

  Background:
    Given 認証済みユーザーとしてログインしている
    And ユビキタス言語辞書の定義に従い"注文"はorderエンティティを指す

  @ORD-1
  Scenario: 在庫あり商品の正常購入
    Given カートに在庫あり商品"商品A"が1件ある
    And 決済情報が登録済みである
    When 購入ボタンを押す
    Then HTTPステータス201が返る
    And レスポンスにorder_idが含まれる
    And 注文ステータスは"confirmed"である
    And inventoryの在庫数が1減少する
    And 確認メールが送信キューに積まれる

  @ORD-2
  Scenario: 在庫切れ商品を含む購入試行
    Given カートに在庫なし商品"商品B"が1件ある
    When 購入ボタンを押す
    Then HTTPステータス409が返る
    And エラーコードは"OUT_OF_STOCK"である
    And 注文エンティティは作成されない
    And inventoryの在庫数に変化はない

  @ORD-3
  Scenario: 決済処理中に他ユーザーが在庫を購入した場合
    Given カートに残在庫1件の商品"商品C"がある
    And 別ユーザーが同時に同商品を購入し在庫が0になる
    When 決済処理が完了する
    Then 注文は自動キャンセル状態で作成される
    And 注文ステータスは"cancelled"である
    And 決済は自動返金処理がキューに積まれる
    And ユーザーに在庫切れ通知メールが送信キューに積まれる

  # 以下、UIの振る舞い（ボタン有効化条件・エラー表示条件）もシナリオとして記述する
```

### D-04　OpenAPI仕様書

| 項目 | 内容 |
|---|---|
| 作成者 | アーキテクトAI（人間レビュー） |
| フェーズ | 2 |
| 関連ゲート | G2（確認対象） |

**目的・役割**

APIインターフェースの単一ソース。コーディングAIのスタブ生成・BDDゲートAIのリクエスト生成・バリデーションの根拠となる。`x-bdd-feature`拡張フィールドでBDDシナリオファイルとの対応を、`x-api-visibility`拡張フィールドでAPI可視性の分類を明示する。

**記載内容の要点**

- エンドポイント・HTTPメソッド・operationId
- リクエスト/レスポンスのスキーマ定義（型・必須・制約）
- 全エラーレスポンスの定義（HTTPステータス・エラーコード・メッセージ）
- `x-bdd-feature`：対応するfeatureファイルのパス
- `x-api-visibility`：API可視性の分類（全エンドポイントに必須）
  - `public`：外部クライアント/外部システムが消費するエンドポイント。変更時は人間確認必須
  - `internal`：同一プロジェクト内のサービス間通信。条件付きでAI自律変更可（詳細は1-3節の権限移譲ルール参照）
  - 未指定の場合は`public`（安全側）として扱う

**フォーマット：** YAML（OpenAPI 3.1.0）

**具体例（要点のみ）**

```yaml
# docs/api/openapi.yaml（抜粋）
paths:
  /orders:
    x-api-visibility: public       # ← 外部API：変更時は人間確認必須
    post:
      operationId: createOrder
      x-bdd-feature: features/order_purchase.feature  # ← BDDシナリオとの紐付け
      responses:
        '201': { description: 注文確定成功 }
        '409': { description: 在庫切れ, example: { error_code: OUT_OF_STOCK } }
  /internal/inventory/check:
    x-api-visibility: internal     # ← 内部API：後方互換な変更はAI自律判断可
    post:
      operationId: checkInventory
      x-bdd-feature: features/inventory.feature
```

### D-05　状態遷移仕様書

| 項目 | 内容 |
|---|---|
| 作成者 | アーキテクトAI（人間レビュー） |
| フェーズ | 2 |
| 関連ゲート | G2（承認対象） |

**目的・役割**

エンティティの状態とその遷移条件を機械可読形式で定義する。AIコーディングエージェントはこれを参照して状態管理ロジックを実装する。ユビキタス言語辞書のstates定義と整合が取れていること。

**単一ソースの原則：** 状態名・遷移条件に関する情報はこのドキュメントが権威を持つ。BDDシナリオ（D-03）内の状態記述はD-05に従う（詳細は1-7節「正本と衝突解決ルール」を参照）。

**記載内容の要点**

- エンティティ名（ユビキタス言語辞書に従う）
- 全状態の列挙
- 各遷移：from, to, trigger, condition, action
- 禁止遷移の明示

**フォーマット：** YAML

**具体例**

```yaml
# docs/specs/order_state_transitions.yaml
entity: order

  - [pending, confirmed, shipped, delivered, cancelled]
transitions:
  - { id: T-001, from: pending, to: confirmed, trigger: 決済成功, condition: 在庫確保済み }
  - { id: T-002, from: pending, to: cancelled, trigger: 決済失敗 OR タイムアウト }
forbidden_transitions:
  - { from: delivered, to: any, reason: 配送完了後の状態変更は不可 }
```

### D-06　画面要素定義書

| 項目 | 内容 |
|---|---|
| 作成者 | アーキテクトAI（人間レビュー） |
| フェーズ | 2 |
| 関連ゲート | G2（承認対象） |

**目的・役割**

UI要素の静的なカタログ。各UI要素のID・種類・ラベルを定義する。**振る舞い（表示条件・有効化条件・画面遷移）はBDDシナリオ（D-03）で定義するため、本ドキュメントには記述しない。** フロントエンドのコーディングAIが「どの要素が画面に存在するか」を把握するための参照資料。ワイヤーフレーム・デザインカンプへのリンクを含む。

> **注：** プロジェクトにフロントエンドがない場合（API onlyの場合）、本ドキュメントは省略可能。

**記載内容の要点**

- 画面名・ID
- 各UI要素のid・種類・ラベル
- デザインカンプ（Figma等）へのリンク
- 対応するBDDシナリオのfeatureファイル（振る舞いの参照先）

**フォーマット：** YAML（デザインカンプは別添リンク）

**具体例**

```yaml
# docs/screens/checkout.yaml
id: checkout
name: チェックアウト画面
design_link: "https://www.figma.com/file/xxx/checkout"
bdd_feature: features/order_purchase.feature  # UIの振る舞いはこちらを参照

elements:
  - id: btn_purchase
    type: button
    label: 購入する
    # 有効化条件・振る舞いはD-03 BDDシナリオ @ORD-1, @ORD-4 を参照

  - id: msg_out_of_stock
```

### D-10　BDDテスト戦略書

| 項目 | 内容 |
|---|---|
| 作成者 | アーキテクトAI（人間レビュー） |
| フェーズ | 3 |
| 関連ゲート | G3（承認対象） |

**目的・役割**

BDDシナリオをどのテスト層で実行し、外部依存をどう扱うかの戦略を定義する。コーディングAIがテストを書く際の判断基準、BDDゲートAIがテストを実行する際の構成指針となる。

**記載内容の要点**

**1. テスト層の定義**

プロジェクトで使用するテスト層を定義し、各層の目的・実行速度・実行頻度を明確にする。以下は標準的な4層構成の例であり、プロジェクト特性に応じて調整する。

| 層 | 目的 | 外部依存 | 実行頻度 |
|---|---|---|---|
| 単体テスト | 純粋なビジネスロジックの検証 | なし（モック不要） | 毎回 |
| BDDシナリオテスト | ユースケース単位の振る舞い検証 | モック or 実接続（後述） | 毎回 |
| E2Eテスト | ブラウザ操作を含むシステム全体の検証 | 実環境 | リリース前 |
| 本番スモークテスト | デプロイ後の疎通確認 | 本番環境 | デプロイ後 |

**2. モック戦略**

BDDシナリオテストにおける外部依存（DB、外部API、認証サービス等）の扱いを定義する。

- **モック実行（推奨デフォルト）：** インメモリ実装等で外部依存を置換し、高速かつ再現性のあるテスト実行を実現する。BDDゲートAIによる全件実行の基本モードとする
- **実接続実行：** 実際のDB・外部サービスに接続して実行する。モック実行との乖離を検出するために定期的に実施する
- モックと実接続の切り替えは、テスト基盤の設定により制御する（コード変更なしで切り替え可能な設計とする）

**3. Pendingシナリオの管理**

自動実行できないBDDシナリオ（Pending）の分類と管理方法を定義する。Pendingシナリオは放置すると仕様のブラックホールになるため、分類と解消計画を明確にする。

| 分類 | 例 | 対応方針 |
|---|---|---|
| インフラ制約 | ブラウザ自動操作が必要だがCI環境にブラウザがない | E2Eテスト層で対応。対応予定時期を記載 |
| 外部サービス依存 | OAuth認証フローのテスト | モック実装で代替、または手動テスト手順を別途記載 |
| 意図的延期 | 次フェーズで実装予定の機能 | 対応予定フェーズを明記 |

**4. カバレッジ基準**

プロジェクトごとにテストカバレッジの定量目標を設定する。目標値はTDRに記録し、BDDゲートAIのレポート（L-04）で達成状況を報告する。

**フォーマット：** Markdown

# 第4章　作業ログファイル（L）

## L-01　コンテキストパッケージ（tmp/tasks/task_{TASK_ID}.md）

| 項目 | 内容 |
|---|---|
| 作成者 | オーケストレーターAI（指示セクション）、ワーカーAI（作業ログセクション） |
| フェーズ | 4 |
| 格納場所 | `tmp/tasks/task_{TASK_ID}.md` |

**目的・役割**

ワーカーAI（コーディングAI・アーキテクトAI等）に渡すタスク定義書であると同時に、作業進捗・テスト結果・チェックポイントの記録先を兼ねる。タスクの「指示→進捗→結果」を1ファイルに集約することで、セッション復帰時のコンテキスト把握を容易にする。

**ファイル構造（2つのゾーン）**

タスク指示書は「指示ゾーン」と「作業ログゾーン」の2つに分かれる。

- **指示ゾーン（フロントマター＋本文セクション）：** オーケストレーターAIが作成・管理する。ワーカーはステータス更新のみ可
- **作業ログゾーン（`## 作業ログ`セクション以降）：** ワーカーAIが作業中に追記する。オーケストレーターはエスカレーション解決内容の追記のみ可

**記載内容の要点**

フロントマター（YAMLメタデータ）:
- **`task_id`：タスクID**
- **`sprint_id`：所属スプリント**（結果取りまとめ時の紐付けに使用）
- **`status`：ステータス**（assigned | in_progress | completed | escalated | aborted）
- **`assigned_to`：対象ワーカー種別**（bdd-coding | bdd-architect | bdd-gate 等）
- **`artifacts_dir`：成果物出力ディレクトリのパス**（ファイル出力が必要な場合のみ指定。省略可。オーケストレーターがアサイン時に作成・指定する。仕様書ドラフトやテストエビデンス等のファイル出力先）
- **`depends_on`：先行タスクIDのリスト**（依存関係がなければ空。オーケストレーターが直列化判断に使用）
- **`locked_files`：このタスク中に変更するファイル一覧**（オーケストレーターが並行アサインの競合チェックに使用。新規作成ファイルは `[NEW]` プレフィックス）
- `created_at` / `updated_at`：タイムスタンプ

本文セクション（指示ゾーン）:
- **タスク概要**：タスクの目的・背景を1〜3文で自己完結的に記述
- **対象BDDシナリオ**：実装対象のfeatureファイルパスと対応シナリオタグ
- **必読ドキュメント（優先度順）**：ワーカーが読むべきドキュメントのパスを `[必須]` / `[参考]` の区分付きで列挙
- **入力ファイル**：前工程の成果物（ワーカーが読み取る既存ファイル）
- **出力ファイル**：この工程で生成すべきファイル
- **完了条件**：BDDシナリオのPASS条件、テストコマンド、その他の達成基準
- **スコープ外**：このタスクでは変更しない領域の明示（スコープクリープ防止。locked_files外の変更が必要と判明した場合はエスカレーション）
- **補足・制約**：特記事項（省略可）

作業ログゾーン（`## 作業ログ`セクション）:
- **チェックポイント**：現在の作業状態（どこまで完了し、次に何をすべきか）、未解決の問題・懸念事項、中断理由（エスカレーション時）。エスカレーション解決後の再開時にワーカーが最初に参照する箇所
- **進捗ログ**：実装中のファイルと進捗状況を逐次追記
- **テスト結果サマリー**：単体テスト・BDDシナリオテストの実行結果（PASS/FAIL件数、失敗シナリオの概要）。詳細ログが大きい場合は`artifacts_dir`に出力し、ここにはサマリーとファイルパスのみ記載
- **エスカレーション解決内容**：オーケストレーターAIがエスカレーション解決後に追記する（ワーカーの再開時に参照）

**フォーマット：** Markdown（先頭にYAMLフロントマター）

---

## ~~L-02　進捗ファイル~~ → L-01に統合（廃止）

> **変更履歴：** 旧L-02（`{work_dir}/progress.md`）は廃止し、タスク指示書（L-01）の`## 作業ログ`セクションに統合した。指示と進捗を1ファイルに集約することで、セッション復帰時のコンテキスト把握を容易にし、ファイル管理を簡素化する。

| ファイル | 役割 |
|---|---|
| `sprint_current.md` | 直近の状況サマリーと過去スプリントへのアーカイブインデックス。オーケストレーターがスプリント開始時に最初に読むファイル |
| `sprint_{N}_plan.md` | 進行中スプリントの計画と結果。スプリント完了後に結果欄が追記される |
| `archive/sprint_XXX_YYY.md` | 過去スプリントの原本を完全保存（10スプリント単位で1ファイルにまとめる）。各スプリント先頭にサマリーを付与 |

**sprint_current.md の記載内容**

- **アーカイブインデックス**：過去スプリントのファイルパス・期間・主な決定事項をテーブル形式で記載
- **最新スプリントの状況**：現在進行中または直近完了のスプリントの概要

**sprint_{N}_plan.md の記載内容**

計画セクション（スプリント開始時に作成）：
- 作成日時・作成者
- **スコープ（重要度基準による自律選定）**：各タスクの内容・選定理由（重要度基準に基づく）・並行可否をテーブル形式で記載
- **実行順序**：依存関係に基づくPhase分け（単独先行→並行実行の順序）

結果セクション（スプリント完了時に追記）：
- **各タスクの完了状況**：TASK_IDごとの最終ステータス（completed / escalated / aborted）
- **エスカレーションの発生状況**：オーケストレーター判断で解決したエスカレーション/ 人間判断が必要なエスカレーションを区別し、対応結果を記録
- **BDDシナリオ（features/配下）の変更有無**

**archive/sprint_XXX_YYY.md の記載内容**
10スプリント単位で、蓄積された`sprint_{N}_plan.md`をarchiveに移動させる。`tmp/orchestrator`直下をクリーンに保つことが目的。

- **各スプリント先頭の3行サマリー**：`sprint_{N}_plan.md`を移動させた後、各ファイルの先頭に3行サマリーを追記する。その他の部分は原文を維持。（内容把握が必要な場合は通常サマリーのみを読み、詳細が必要な場合のみ全文を参照する）

**ライフサイクル**

```
1. スプリント開始
   └─ sprint_current.md を読み込み、状況把握
   └─ features/ 配下を読み込み、BDDシナリオの実装状況を確認

2. 計画記録
   └─ sprint_{N}_plan.md を新規作成
   └─ 重要度基準に従いスコープを自律選定、実行順序を決定
   └─ 具体的なタスクに分解し、TASK_IDを振る（sprint_{N}_plan.mdに追記）

3. ワーカー実行
   └─ タスク指示書（L-01）を作成しワーカーにアサイン
   └─ G4エスカレーション発生時は対応方針を決定

4. 結果記録
   └─ sprint_{N}_plan.md に結果セクションを追記
   └─ 人間確認要否を判定
   └─ sprint_current.md を更新

5. アーカイブ（スプリント蓄積時）
   └─ 過去のsprint_{N}_plan.md を archive/ に移動（10スプリント単位）
   └─ sprint_current.md のインデックスを更新
```

### 作業ログファイル（L）

tmp/配下に格納される一時ファイル。プロジェクト成果物ではなく、AI間通信・状態管理・意思決定記録を目的とする。ドキュメントレビューAIのレビュー対象外だが、各AIのインプットとして参照される。

| # | ファイル名 | 採番 | フェーズ | 作成者 | 備考 |
|---|---|---|---|---|---|
| L-01 | コンテキストパッケージ | TASK-XXX | 4 | オーケストレーターAI（指示）、ワーカーAI（作業ログ） | タスク指示書＋作業ログ一体型（tmp/tasks/task_{TASK_ID}.md） |
| ~~L-02~~ | ~~進捗ファイル~~ | - | - | - | L-01の`## 作業ログ`セクションに統合（廃止） |
| L-03 | エスカレーションログ | ESC-XXX | 4〜 | AI→人間 | 仕様変更フローの実行ログ（tmp/escalations/escalation_{ESC_ID}.md） |
| L-04 | BDD実行レポート | - | 5 | BDDゲートAI | G5承認根拠（tmp/reports/bdd_report.md） |
| L-05 | ドキュメントレビューレポート | - | 5 | ドキュメントレビューAI | G5承認根拠（tmp/reports/doc_review.md） |
| L-06 | コードレビューレポート | - | 5 | コードレビューAI | G5承認根拠（tmp/reports/code_review.md） |
| L-07 | スプリント管理ログ | Sprint-XXX | 4〜5 | オーケストレーターAI | スプリント状況・計画・アーカイブ（tmp/orchestrator/配下） |
| L-08 | スプリント固有実装ガイド | Sprint-XXX | 4 | オーケストレーターAI（アーキテクトAI成果物を元に作成） | スプリントスコープに特化した実装指針（tmp/orchestrator/配下） |

> **スプリント固有実装ガイドについて:** プロジェクト成果物（D-series）は普遍的な方針・判断を記録するが、スプリント遂行中にはスコープ固有の詳細な実装指針が必要になることがある（対象シナリオの絞り込み、feature別の注意点、タスク境界の定義等）。この場合、オーケストレーターAIはD-seriesの方針に基づきスプリント固有実装ガイド（L-08）を `tmp/orchestrator/` に作成し、タスク指示書の必読ドキュメントとして参照させる。L-08はスプリント完了後にアーカイブされる一時文書であり、D-seriesのように長期保持されるものではない。
