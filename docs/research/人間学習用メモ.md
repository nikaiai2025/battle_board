## ドッカー初心者メモ
Supabase Studio（DB管理UI）は http://127.0.0.1:54323 で確認可能

---
Dockerとは

「アプリをまるごと箱に詰めて動かす仕組み」 です。

PostgreSQLやSupabaseのような複雑なサーバーソフトを、PCに直接インストールせずに「コンテナ」という隔離された箱の中で動かせます。

あなたのPC
├── Docker Desktop（コンテナを管理するエンジン）
│   ├── コンテナA: PostgreSQL（Supabase用DB）
│   ├── コンテナB: Supabase Auth サーバー
│   ├── コンテナC: Supabase Studio（管理UI）
│   └── ... （supabase startで全部まとめて起動）
└── あなたのコード（npm run dev）

---
このプロジェクトで覚えること（3つだけ）

1. 起動・停止

- 開発前に起動（初回はイメージDL済みなので数秒）
npx supabase start

- 開発終わったら停止（任意。PCの動作が重ければ止める）
npx supabase stop

- 今の状態確認
npx supabase status

2. DBをリセットしたいとき

ローカルのDBのデータを全部消してマイグレーションからやり直す:

npx supabase db reset

テストで変なデータが入ったときや、マイグレーションを変更したときに使います。

3. Docker Desktopは「事前に起動しておく」

npx supabase start はDocker Desktopが起動していないと動きません。
PCを再起動したら、Docker Desktopを先に立ち上げてから開発を始めてください。
（タスクトレイにクジラのアイコンが出ていればOK）

---
よくある質問

Q: データはどこに保存される？
コンテナの中（Docker管理領域）。supabase stop しても消えません。supabase stop --no-backup または db reset で消えます。

Q: PCを再起動したらどうなる？
コンテナは停止します。Docker Desktop起動 → npx supabase start で再開できます。

Q: 本番DBに影響する？
まったくありません。ローカルとSupabase本番は完全に別です。



## 認証仕様をわかりやすく

認証の仕組み（2層構造）

┌─────────────────────────────────────────────────────────┐
│                   書き込みリクエスト                       │
│              (Web API or 専ブラ bbs.cgi)                  │
└──────────────────────┬──────────────────────────────────┘
                        ▼
            ┌─── edge-token Cookie あるか？ ───┐
            │                                  │
            ある                               ない
            │                                  │
            ▼                                  ▼
    edge-token有効か？                  ① edge-token新規発行
            │                          ② 認証コード(6桁)発行
            有効                          ③ 「認証してください」を返す
            │                               │
            ▼                                ▼
        書き込み成功                    ユーザーが認証コードを入力
                                        + Turnstile検証
                                            │
                                            ▼
                                        認証成功 → 書き込み可能

第1層: edge-token（セッション識別）

- 役割: 「このブラウザ/端末は認証済みか」を識別するCookie
- 発行タイミング: 初回書き込み時に自動発行（30日有効）
- 検証: IPハッシュとの照合（ソフトチェック＝不一致でもログのみ）

第2層: 認証コード + Turnstile（本人確認）

- 認証コード: サーバーが生成した6桁の数字。ユーザーがそれを入力して「知っている」ことを証明
- Turnstile: Cloudflareの「人間か？」チェック。認証コード入力時にセットで検証
- 検証タイミング: POST /api/auth/auth-code で認証コード + Turnstileトークンを同時に検証

## IPが同じになる条件は何か？

  日次リセットIDが同一になる条件
  生成アルゴリズム： dailyId = sha256( dateJst + boardId + authorIdSeed )[0:8]
  3つの入力が全て同じなら同一IDになる。
  ┌──────────────┬──────────────────────┬──────────────────────────────────────────┐
  │     入力     │    決定タイミング    │                 値の性質                 │
  ├──────────────┼──────────────────────┼──────────────────────────────────────────┤
  │ dateJst      │ 書き込み時           │ JST日付（YYYY-MM-DD）。翌日0時でリセット │
  ├──────────────┼──────────────────────┼──────────────────────────────────────────┤
  │ boardId      │ 固定                 │ 現状 "battleboard" 固定                  │
  ├──────────────┼──────────────────────┼──────────────────────────────────────────┤
  │ authorIdSeed │ ユーザー登録時に固定 │ sha512(reduceIp(登録時のIP))             │
  └──────────────┴──────────────────────┴──────────────────────────────────────────┘
  ポイントは authorIdSeed が登録時に固定される点。
  別のPCで同一回線の場合

  IPv4の場合
  reduceIp はIPv4をそのまま返す（192.168.1.1 → 192.168.1.1）。
  同一回線（同一グローバルIP）から別々に登録した2台のPC → authorIdSeed が同じ → 同一IDになる。5ch本家と同じ挙動。

  IPv6の場合
  reduceIp は /48 プレフィックスに縮約する（先頭48bit）。
  同一ISP契約（通常 /48 〜 /56 が割り当てられる）の2台 → /48プレフィックスが同一 → 同一IDになる。

  重要な補足: seedは登録時に固定
  // issueEdgeToken（ユーザー新規登録）
  const user = await UserRepository.create({
      authorIdSeed: ipHash,  // ← 登録時のIPから算出、以後不変
  });
  authorIdSeed は登録時の1回だけ計算され、以後変更されない。つまり：
  ┌──────────────────────────────────────┬────────────────────────────────────┐
  │               シナリオ               │                 ID                 │
  ├──────────────────────────────────────┼────────────────────────────────────┤
  │ 同じPCで回線が変わった（引っ越し等） │ 同じIDのまま（seedが登録時固定）   │
  ├──────────────────────────────────────┼────────────────────────────────────┤
  │ 同じ回線で別のPC（別ユーザー登録）   │ 同じIDになる（同じIPでseedが同値） │
  ├──────────────────────────────────────┼────────────────────────────────────┤
  │ 同じ回線でもVPN経由で登録した        │ 異なるID（登録時IPが違う）         │
  └──────────────────────────────────────┴────────────────────────────────────┘






## デザイン


❯ Webサービスのあるべき論としてはどうあるべき？このプロジェクトの現状はそれに沿っている？

● あるべき姿（3層構造）

  成熟したWebサービスのスタイル管理は、以下の3層に分離されています。
  ┌─────────────────────────────────────────┐
  │  デザイントークン                          │  色・余白・フォントサイズ等の「値」の定義
  │  （1箇所変えたら全部変わる）                 │  例: primary-color = #2563eb
  ├─────────────────────────────────────────┤                                                                                        │  共通UIコンポーネント                       │  ボタン・入力欄・カード等の「部品」
  │  （デザイントークンを参照して見た目を統一）    │  例: <Button variant="primary">
  ├─────────────────────────────────────────┤
  │  ページ固有レイアウト                       │  各画面で部品を並べる「配置」
  │  （共通コンポーネントを組み合わせるだけ）      │  例: page.tsx
  └─────────────────────────────────────────┘                                                                                                                                                                                                                                                               各ページが直接色やサイズを指定することはなく、トークンと部品を通じて間接的にスタイルが決まる。だから1箇所の変更が全体に波及する。
  このプロジェクトの現状
  ┌──────────────────────┬─────────────────────────────────┬────────────────────────────────────────────────────┬────────────────┐   │          層          │           あるべき姿            │                        現状                        │      評価      │
  ├──────────────────────┼─────────────────────────────────┼────────────────────────────────────────────────────┼────────────────┤
  │ デザイントークン     │ 色・余白・フォント等の変数定義  │ --background と --foreground の2変数のみ           │ ほぼ未整備     │
  ├──────────────────────┼─────────────────────────────────┼────────────────────────────────────────────────────┼────────────────┤
  │ 共通UIコンポーネント │ Button, Input, Card, Heading 等 │ ドメイン部品（PostItem等）はあるが汎用UI部品がない │ 未整備         │
  ├──────────────────────┼─────────────────────────────────┼────────────────────────────────────────────────────┼────────────────┤   │ ページ固有レイアウト │ 共通部品を並べるだけ            │ Tailwindクラスで色・サイズ・余白を直接指定         │ 全部ここに集中 │
  └──────────────────────┴─────────────────────────────────┴────────────────────────────────────────────────────┴────────────────┘
  要するに、3層あるべきところが1層に潰れている状態です。
  具体的に何が困るか
  // 現状: 「灰色の見出し」がページごとにベタ書き
  <h1 className="text-base font-bold text-gray-800">   // thread-view
  <h1 className="text-base font-bold text-gray-700">   // thread-list（微妙に色が違う）
  - 色を統一したい → 全ファイルを grep して直す必要がある
  - 見出しのスタイルを変えたい → 同上
  - 誰かが新ページを作る → 既存ページからコピペ → 微妙にズレる（上の例が既にそう）
