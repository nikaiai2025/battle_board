# 認証アーキテクチャ決定ログ（2026-03-04）

## 1. 目的
- 本ログは、実装着手前に認証方式と実装責務の境界を固定するための決定記録である。
- 対象は「掲示板投稿認証」「管理者ログイン」「運用バッチ」の3領域。

## 2. 最終方針（決定）

### 2.1 投稿者認証（一般ユーザー）
- 一般ユーザーは **メール必須にしない**。
- 投稿認証は以下で構成する。
  - `edge-token` Cookie
  - 6桁認証コード（`/auth-code`）
  - CAPTCHA検証
- 投稿IDは日次更新の匿名IDとして扱う（同日内同一性を許容）。
- Cookie削除後の再認証時も、IPベースseedにより同一性が部分的に継続しうる設計を採用する（IP依存度: 強め）。

### 2.2 管理者認証
- 管理者ログインは **メールログイン方式を採用**（一般ユーザー投稿認証とは完全分離）。
- 管理者ログイン方式は **メール+パスワード** を採用する。
- 管理者認証は投稿者認証トークンを共有しない（Cookie名・セッション・ミドルウェアを分離）。
- 2FA導入を推奨（可能なら必須）。

### 2.3 レート制限
- 初期実装では、一般投稿フローに対する専用レート制限は導入しない（明示的な方針）。
- 将来、必要時に追加可能な構造で実装する（拡張ポイントのみ確保）。

### 2.4 CAPTCHA
- 初期採用は **Cloudflare Turnstile** とする。
- CAPTCHAは `/auth-code` の認証コード有効化時に適用し、投稿API本体では毎回実施しない。

## 3. 実装責務の境界

### 3.1 Vercel + Supabase で実装する範囲
- 認証コード発行/検証API
- `edge-token` 発行・検証
- CAPTCHA検証（API内で実行）
- 投稿時認証判定
- 日次ID生成ロジック
- 管理者メールログイン（投稿者認証と分離）
- 永続化（Supabase Postgres）
  - トークン
  - 認証コード
  - 投稿・監査ログ（必要項目）
- 失効・有効期限判定ロジック

### 3.2 GitHub Actions で実装する範囲
- 定期バッチのみ
  - 期限切れデータ掃除
  - 軽量集計更新（必要時）
  - アーカイブ/保守処理（必要時）
- 実行頻度は cron で管理（例: 毎時/毎日）

### 3.3 GitHub Actions で実装しない範囲（禁止）
- 認証コード検証
- CAPTCHA判定
- 投稿可否判定
- Cookie/セッション発行
- その他、ユーザーリクエストに対する即時同期処理

## 4. 非機能上の前提
- サーバレス前提のため、メモリ内状態に依存しない（必ずDB等を正とする）。
- クライアントIPはVercel配下のヘッダを使って抽出する。
- Cloudflareは必須構成としない（利用時はヘッダ信頼境界を別途固定）。

## 5. 実装担当者向け技術メモ（実装に直結する事項のみ）

### 5.1 認証フロー（最小）
1. 投稿時に有効トークン未所持なら、サーバで `auth_code` と `auth_token` を発行して未認証応答を返す。
2. 同時に `edge-token` Cookieを設定する（有効期限を明示）。
3. ユーザーが `/auth-code` で6桁コード + Turnstile応答を送信。
4. コード・期限・IP整合を検証してトークン有効化。
5. 有効化後、投稿処理で `edge-token` を通す。

### 5.2 投稿ID（IP依存度: 強め）
- `author_id_seed` は reduced IP を主材料に生成する。
- 表示IDは「日付（JST）+ board_key + seed」で生成する。
- これにより同日・同回線で同一IDになりやすく、Cookie削除後も同一性が継続しうる。

### 5.3 ヘッダ信頼境界（Vercel）
- IP抽出は `x-forwarded-for` 系を使用し、先頭IPを採用する。
- 直接公開エンドポイントでヘッダ偽装を許さないよう、信頼プロキシ前提で実装する。
- 将来Cloudflare前段化する場合は `cf-connecting-ip` 優先に切替可能な構造で実装する。

### 5.4 Cookie/セッション分離
- 投稿者: `edge-token`（投稿認証専用）
- 管理者: `admin_session`（管理画面専用）
- 同一ミドルウェアで混在させない（認証ガードを分離）

### 5.5 GitHub Actionsで実施する具体バッチ
- 期限切れ `auth_code` / 一時トークンの削除
- 長期不要セッション/監査ログの保守削除（保持ポリシーに従う）
- 集計テーブル更新（導入時のみ）

## 6. スコープ外（初期）
- 一般ユーザーのメール必須ログイン
- 高度な不正対策（指紋採取エンジン、複雑なスコアリング）
- 常時監視・自動BAN運用の高度化
