# D-08 コンポーネント設計書: IncentiveService

> 作成日: 2026-03-07
> 対象: Phase 1 (MVP)

## 1. 概要

8種のボーナスイベントの発火条件を判定し、通貨を付与するコンポーネント。書き込み処理のトランザクション内で呼び出される即時判定と、後続書き込み時にチェックする遅延評価の2種類がある。

## 2. 責務

- 各ボーナスイベントの発火条件判定
- 重複付与の防止（incentive_logs による冪等性担保）
- 通貨付与の CurrencyService への委譲
- ボーナス付与記録の永続化（incentive_logs）

## 3. 依存関係

```
IncentiveService
  ├── CurrencyService         (通貨付与)
  ├── IncentiveLogRepository  (付与記録の読み書き)
  ├── PostRepository          (返信・ホットレス判定用)
  ├── ThreadRepository        (スレッド成長判定用)
  └── UserRepository          (ストリーク更新)
```

## 4. ボーナスイベント分類

### 同期判定（書き込みトランザクション内）

| イベント | 条件 | 金額 | 重複制御 |
|---|---|---|---|
| 書き込みログインボーナス | 当日初回書き込み | +10 | `(user_id, 'daily_login', date)` |
| スレッド作成ログインボーナス | 当日初回スレ立て | +10 | `(user_id, 'thread_creation', date)` |
| 新スレッド参加ボーナス | 未参加スレへの初書き込み | +3 | `(user_id, 'new_thread_join', thread_id)` + 1日3回上限 |
| キリ番ボーナス | レス番号が100の倍数 | +10/+100 | `(user_id, 'milestone_post', post_id)` |
| ストリークボーナス | N日連続書き込み達成 | +20/+100 | `(user_id, 'streak', date)` |
| 返信ボーナス（被返信者） | アンカー付き返信を受ける | +5 | `(target_user_id, 'reply', date, sender_daily_id)` |

### 遅延評価（後続書き込みトランザクション内でチェック）

条件が「未来の書き込み」に依存するため、対象レスの書き込み時点では判定不可能。後続の書き込みトランザクション内で過去レスをチェックして発火する（メッセージキュー等の非同期基盤は使用しない）。

| イベント | 条件 | 金額 | トリガー |
|---|---|---|---|
| ホットレスボーナス | 60分以内に3人以上から返信 | +15 | 返信書き込み時に過去レスをチェック |
| スレッド復興ボーナス | 低活性スレ書き込み後30分以内に他者レス | +10 | 後続書き込み時にチェック |

### 非対象

| 項目 | 理由 |
|---|---|
| 書き込み報酬 | incentive.feature v4で「報酬なし」に統一 |
| スレッド成長ボーナス | 同期判定可能だが、ユニークID数クエリのコストを考慮し実装方針は要検討 |

## 5. 処理フロー

### 5.1 メイン評価（書き込み時）

```
IncentiveService.evaluate(context, tx)
│
├── 1. 書き込みログインボーナス判定
│   ├── 今日の書き込みログインボーナス記録が存在するか？
│   │   → 存在: スキップ
│   │   → 不在: +10 付与、incentive_logs INSERT
│   └── ストリーク更新
│       ├── lastPostDate が昨日 → streakDays +1
│       ├── lastPostDate が今日 → 変更なし
│       └── lastPostDate がそれ以前 → streakDays = 1（リセット）
│       └── streakDays が 7 or 30 → ストリークボーナス付与
│
├── 2. スレッド作成ボーナス判定（スレ立て時のみ）
│   └── 今日のスレッド作成ボーナス記録が存在するか？
│       → 不在: +10 付与
│
├── 3. 新スレッド参加ボーナス判定
│   ├── このスレッドへの過去書き込み記録が存在するか？
│   │   → 存在: スキップ
│   └── 今日の新スレッド参加ボーナス回数 < 3 か？
│       → +3 付与
│
├── 4. キリ番ボーナス判定
│   ├── postNumber % 1000 == 0 → +100 付与
│   ├── postNumber % 100 == 0 → +10 付与
│   └── それ以外 → スキップ
│
├── 5. 返信ボーナス判定
│   ├── 本文中のアンカー（>>N）を抽出
│   ├── 各アンカー先のレスの著者を取得
│   ├── 著者が自分自身 → スキップ
│   ├── 今日すでに同一IDから返信ボーナスを受け取っている → スキップ
│   └── +5 付与（被返信者に）
│
├── 6. ホットレスボーナス判定（遅延評価）
│   ├── アンカー先のレスが60分以内のものか確認
│   ├── そのレスへの返信者数（異なるdailyId）をカウント
│   ├── 3人目の返信か？
│   │   → ホットレスボーナス未付与なら +15 付与（元レスの著者に）
│   └── 自分が3人目でない → スキップ
│
└── 7. スレッド復興ボーナス判定（遅延評価）
    ├── このスレッドの前のレスの書き込み者を取得
    ├── 前のレスが低活性スレッド復興書き込みか？
    │   （直前レスの前のレスから24時間以上空いている）
    ├── 前のレスから30分以内か？
    ├── 前のレスの書き込み者 ≠ 自分か？
    └── 条件を満たす → 前のレスの著者に +10 付与
```

## 6. 重複防止の実装

`incentive_logs` テーブルのユニーク制約で冪等性を担保:

```sql
-- 日次ボーナス（1日1回）
CREATE UNIQUE INDEX idx_incentive_daily
  ON incentive_logs (user_id, event_type, context_date)
  WHERE event_type IN ('daily_login', 'thread_creation', 'streak');

-- レス単位ボーナス（1レスにつき1回）
CREATE UNIQUE INDEX idx_incentive_post
  ON incentive_logs (user_id, event_type, context_id)
  WHERE event_type IN ('milestone_post', 'hot_post');

-- 返信ボーナス（同一IDから1日1回）
-- context_id にはレスIDではなく送信者のdailyIdハッシュを使う
```

INSERT 時に `ON CONFLICT DO NOTHING` を使い、重複は無視。

## 7. エラーハンドリング

**原則: ボーナス付与の失敗で書き込み自体を巻き戻さない。** （architecture.md §7.3 参照）

| 失敗種別 | 対応 |
|---|---|
| incentive_logs INSERT のユニーク制約違反 | 正常系（重複防止の意図的動作）。`ON CONFLICT DO NOTHING` で無視 |
| CurrencyService の付与失敗 | エラーログに記録して続行。後で手動補填可能 |
| 判定クエリの失敗（DB一時障害等） | エラーログに記録して続行。該当ボーナスをスキップ |

> この方針は architecture.md §7.3「インセンティブ判定エラー → 書き込みは成功、ボーナスはスキップ」と一致する。
