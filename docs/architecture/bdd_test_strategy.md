## 1. テストレベルの決定: サービス層テスト

**結論:** サービス層の公開関数を直接呼び出す。APIルート（Next.js Route Handler）は経由しない。

**理由:**
- APIルートは「リクエスト受付 → Service呼び出し → レスポンス整形」の薄いアダプターであり、ビジネスロジックを持たない
- HTTPサーバーの起動・リクエスト生成が不要になり、テスト実行速度とセットアップの単純さが向上する
- vitestの単体テストが同じサービス層を対象としており、テスト対象の粒度が一致する
- featureファイルのシナリオは全てビジネスロジックの振る舞いを記述しており、HTTPステータスコードやヘッダーの検証を含まない

**注意:** サービス間の内部呼び出し連鎖（例: PostService → IncentiveService）はモックせず、結合した状態でテストする。BDDシナリオが検証するのはユーザーから見た振る舞いの結果であり、サービス間の呼び出し構造ではない。

## 2. 外部依存のモック戦略

**方針:** リポジトリ層のモジュールをインメモリ実装に差し替える。実DB接続は行わない。

**理由:**
- Supabase（PostgreSQL）への実接続はCI環境での信頼性・速度に問題がある
- vitestでは `vi.mock()` を使っているが、Cucumber.jsの実行環境（ts-node/CommonJS）では利用不可
- Cucumber.js v12ではESMサポートが限定的であるため、CommonJS互換の手法が安全

**インメモリ実装の設計方針:**
- 各リポジトリのエクスポート関数と同一シグネチャの関数を持つオブジェクトを用意する
- データは `Map<string, T>` や配列で保持し、Worldのライフサイクル（シナリオ単位）でリセットする
- DB固有動作（`ON CONFLICT DO NOTHING`、楽観的ロック等）は、インメモリ実装内で同等のロジックを再現する
- モック機構の具体的な実装方式（requireキャッシュ書き換え、tsconfig-paths誘導等）はコーダーが選定する。ただしプロジェクト内で1方式に統一すること

**ライフサイクル:**
- `BeforeAll`: モジュール差し替えをインストール
- `Before`（各シナリオ前）: インメモリデータをクリア
- `After`（各シナリオ後）: 必要に応じたクリーンアップ（時刻スタブの復元等）

## 3. Cucumber World 設計

**Worldが保持すべき状態カテゴリ:**

| カテゴリ | 説明 | 用途例 |
|---|---|---|
| 現在のユーザー | userId, edgeToken, ipHash, isPremium, username | 「ユーザーがログイン済みである」の状態管理 |
| 名前付きユーザーマップ | `Map<string, UserContext>` | 「ユーザー "UserA"」「ユーザー "UserB"」の複数ユーザー管理 |
| 現在のスレッド | threadId, threadTitle | 「スレッド "今日の雑談" を閲覧している」の状態管理 |
| 最後の操作結果 | lastResult, lastError | Then ステップでのアサーション対象 |
| 時刻制御 | currentTime (Date) | 日付変更・経過時間のシミュレーション |

**設計原則:**
- Worldのプロパティは型付きで定義する（anyを避ける）
- 各シナリオの独立性を保証するため、Beforeフックで全状態をリセットする
- テスト用エンティティの生成ヘルパーをWorldのメソッドとして提供する

## 4. ディレクトリ構成とファイル分割方針

```
features/
  support/
    world.ts              # World クラス定義
    hooks.ts              # BeforeAll / Before / After フック
    mock-installer.ts     # リポジトリモジュール差し替えの機構
    in-memory/            # インメモリリポジトリ実装群（1リポジトリ = 1ファイル）
  step_definitions/
    common.steps.ts       # 複数featureで共有されるステップ
    {feature}.steps.ts    # feature固有のステップ（1 feature = 1ファイル）
```

**分割基準:**
- 1 feature = 1 stepsファイルが原則
- 3つ以上のfeatureで使われるステップは common.steps.ts に集約する
- 2つのfeatureでのみ共有されるステップは、先に定義されたfeatureのstepsファイルに置く（重複定義エラーを避ける）

## 5. 時刻制御の方針

多くのシナリオが日付・時間に依存する（日次リセットID、ストリーク、低活性スレッド判定、時間内ボーナス等）。時刻依存テストはFlakyテスト（実行タイミングにより結果が変わる不安定テスト）の最大の原因であり、以下の原則を厳守する。

### 5.1 時計凍結の原則

**方針:** テスト中は実時間に一切依存せず、凍結した時計のみを使用する。

- `Date.now` をグローバルにスタブ化し、World経由で任意の時刻に設定・進行する
- Beforeフックでオリジナルの `Date.now` を保存し、Afterフックで復元する
- sinon の `useFakeTimers` やカスタム実装のどちらでもよいが、1つの方式に統一すること

### 5.2 相対時刻の禁止

テストデータの時刻を `Date.now() - offset` で算出してはならない。テスト実行中に実時間が経過するため、ステップ間でミリ秒単位の不整合が生じFlakyテストの原因になる。

```typescript
// NG: 実行タイミングでミリ秒がずれる
const pastTime = new Date(Date.now() - 25 * 60 * 60 * 1000)

// OK: 時計を固定時刻に設定してからデータを作る
world.setCurrentTime(new Date('2026-03-12T10:00:00+09:00'))
// この時点で Date.now() は固定値を返すため、サービス内の new Date(Date.now()) も同じ値になる
```

**時刻依存シナリオの標準パターン:**

1. 時計を基準時刻 T に凍結する
2. 時計を過去（T - N時間）に設定し、事前データを作成する
3. 時計を T に戻し、テスト対象の操作を実行する
4. 必要に応じて時計を T + M 分に進め、後続操作を実行する
5. 結果をアサートする

### 5.3 サービス層の時刻取得統一

サービス層で現在時刻を取得する際は `new Date(Date.now())` を使用し、`new Date()` を直接使用しない。`Date.now` のスタブがサービス内部の時刻生成にも反映されることを保証するためである。

### 5.4 テストデータ作成の優先順位

時刻依存の事前データ（過去のレス、スレッドの最終書き込み時刻等）を作成する際は、以下の優先順位に従う:

1. **サービス経由での作成**（推奨）: 時計を過去に設定し、通常のサービスAPI（PostService.createPost等）で作成する。サービスの実際の動作と一致するため最も信頼性が高い
2. **インメモリリポジトリの直接操作**（許容）: サービス経由では作成が困難な状態（途中状態、異常状態等）に限りリポジトリの `_insert` 等を使用する。この場合もcreatedAt等の時刻フィールドは凍結時計の値を使うこと

## 6. スコープ外シナリオの扱い方針

featureファイルは変更禁止（CLAUDE.md禁止事項）であるため、タグの追加によるスキップはできない。

**方針:**
- 対象外のfeatureファイルは cucumber.js の `paths` で除外する
- 対象featureファイル内のスコープ外シナリオは `name` オプション（正規表現 negative match）で除外する
- 具体的な除外対象はスプリント固有の実装ガイドで定義する

## 7. テストピラミッドと多層テスト戦略

> Phase 2以降に適用。Phase 1はサービス層テスト（§1）+ 単体テスト（Vitest）で完結していた。

### 7.1 全体像

```
        ┌──────────────┐
        │    E2E       │  Playwright — 重要フローのみ（数本〜10本）
       ─┼──────────────┼─
       │  API テスト    │  HTTPレベル検証 — 専ブラ互換・Cookie等（10〜20本）
      ──┼───────────────┼──
      │  統合テスト      │  Supabase Local実DB — 既存BDDシナリオを流用
     ───┼────────────────┼───
     │  BDD サービス層    │  InMemoryリポジトリ — 高速・開発中の主力（既存）
    ────┼─────────────────┼────
    │   単体テスト (Vitest)  │  純粋関数・ドメインモデル（既存）
   ─────┴────────────────────┴─────
```

**原則:** 下層ほどテスト数を多く、上層ほど厳選する。各層の責務を明確に分離し、重複検証を避ける。

### 7.2 各層の責務と検証対象

| 層 | 検証対象 | 検証しないこと |
|---|---|---|
| 単体テスト | 純粋関数の入出力、ドメインルール | 外部依存、統合動作 |
| BDDサービス層 | ビジネスロジックの振る舞い、サービス間連携 | SQL、HTTP、UI |
| 統合テスト | SQL・RLS・マイグレーション・リポジトリ層の実動作 | HTTP、UI |
| APIテスト | ルーティング、Cookie処理、Content-Type、エンコーディング | ビジネスロジック（サービス層で検証済み） |
| E2E | 認証フロー、複数ユーザー操作、画面遷移の一連の流れ | 個別のビジネスルール |

## 8. 統合テスト方針

### 8.1 基本方針

既存BDDシナリオ（`.feature`）とステップ定義をそのまま流用し、リポジトリの注入先をInMemory実装からSupabase Local実装に切り替えて実行する。

**メリット:** 新規テストコードの作成がほぼ不要。既存BDDシナリオが自動的に統合テストになる。

### 8.2 切替の仕組み

- テストプロファイル or 環境変数（例: `TEST_DB=supabase`）でリポジトリ実装を分岐する
- `BeforeAll` フックで判定し、InMemory / Supabase のいずれかをインストールする
- ステップ定義側は変更不要（リポジトリのインターフェースが同一であるため）

### 8.3 前提条件

- Docker Desktop + Supabase Local が起動していること（`npx supabase start`）
- マイグレーションが適用済みであること（`npx supabase db push`）
- テスト用の `.env` に `TURNSTILE_SECRET_KEY` を設定しないこと（§10参照）

### 8.4 データライフサイクル

- `Before`（各シナリオ前）: テスト用テーブルのTRUNCATE
- シナリオ間の独立性はインメモリ版と同様に保証する

## 9. APIテスト方針

### 9.1 基本方針

Next.jsサーバー（`next build && next start` または `next dev`）を起動し、fetchでAPIエンドポイントを叩くHTTPレベルのテスト。**全APIに書くのではなく、サービス層テストでカバーできない検証が必要な箇所に限定する。**

### 9.2 対象領域

| 対象 | 検証内容 | 理由 |
|---|---|---|
| 専ブラ互換API | Shift_JISエンコーディング、DAT形式、subject.txt、bbs.cgi | バイナリレベルの正確性はサービス層テストでは検証不可 |
| 認証API | Cookie設定（Secure, HttpOnly, SameSite）、レスポンスヘッダー | HTTP属性はサービス層では検証不可 |
| エラーレスポンス | HTTPステータスコード、エラーボディ形式 | クライアントとの契約 |

### 9.3 対象外

- ビジネスロジックの検証（サービス層テストで十分カバーされている）
- 正常系のCRUD操作（APIルートは薄いアダプターであり、追加の検証価値が低い）

---

## 10. E2Eテスト方針

### 10.1 基本方針

Playwright + Supabase Localを使用したブラウザ自動テスト。**作成・維持コストが最も高いため、重要な一連のフローのみを対象とする。** 個別のビジネスルール検証には使用しない。

### 10.2 ツール

- **Playwright**（推奨）: クロスブラウザ対応、Next.jsとの親和性が高い

### 10.3 対象シナリオの選定基準

以下のいずれかに該当するフローのみE2Eテストを作成する:
- 認証 → 操作 → 結果確認の一連のフロー（複数APIをまたぐ）
- 複数ユーザー間の同時操作シナリオ
- 人間が手動確認の代替として指定したシナリオ

### 10.4 数量目標

数本〜10本程度。増やすよりも、各テストが重要なフローを確実にカバーすることを優先する。

## 11. 外部サービスのテスト環境対応

### 11.1 Cloudflare Turnstile

Turnstileはテスト自動化を妨げない設計とする。

| テストレベル | 対処方法 | 仕組み |
|---|---|---|
| BDD / 単体テスト | InMemoryモック | `features/support/in-memory/turnstile-client.ts` で常に成功を返す |
| 統合テスト / APIテスト | 環境変数未設定でバイパス | `turnstile-client.ts`: `TURNSTILE_SECRET_KEY` 未設定時は `return true` |
| E2E | Cloudflare公式テスト用サイトキー | `1x00000000000000000000AA`（常に成功）を使用 |

**注意:** テスト用の環境変数に本番の `TURNSTILE_SECRET_KEY` を設定してはならない。設定するとCloudflare APIへの実通信が発生し、テスト環境からのリクエストが検証失敗する。

## 12. テスト実行の役割分担

### 12.1 開発AI（bdd-coding）

- テストコードの作成（全レベル）
- InMemoryテストの実行（開発サイクル中の高速フィードバック）
- 単体テストの実行

### 12.2 ゲートAI（bdd-gate）

- **テストコードは書かない。実環境での実行と合否判定に特化する。**
- Supabase Local + Next.jsサーバーを起動
- 全テストスイートを実行（BDD + 統合 + API + E2E）
- 結果の判定・レポート作成（PASS/FAIL・差分分析）
- フェーズ5検証サイクルにおける最終品質ゲートとして機能する
