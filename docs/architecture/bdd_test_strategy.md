## 1. テストレベルの決定: サービス層テスト

**結論:** サービス層の公開関数を直接呼び出す。APIルート（Next.js Route Handler）は経由しない。

**理由:**
- APIルートは「リクエスト受付 → Service呼び出し → レスポンス整形」の薄いアダプターであり、ビジネスロジックを持たない
- HTTPサーバーの起動・リクエスト生成が不要になり、テスト実行速度とセットアップの単純さが向上する
- vitestの単体テストが同じサービス層を対象としており、テスト対象の粒度が一致する
- featureファイルのシナリオは全てビジネスロジックの振る舞いを記述しており、HTTPステータスコードやヘッダーの検証を含まない

**注意:** サービス間の内部呼び出し連鎖（例: PostService → IncentiveService）はモックせず、結合した状態でテストする。BDDシナリオが検証するのはユーザーから見た振る舞いの結果であり、サービス間の呼び出し構造ではない。

## 2. 外部依存のモック戦略

**方針:** リポジトリ層のモジュールをインメモリ実装に差し替える。実DB接続は行わない。

**理由:**
- Supabase（PostgreSQL）への実接続はCI環境での信頼性・速度に問題がある
- vitestでは `vi.mock()` を使っているが、Cucumber.jsの実行環境（ts-node/CommonJS）では利用不可
- 本プロジェクトではCucumber.jsをCommonJS（ts-node）で実行しており、モック機構もCJS前提で構築済みである。Cucumber.js v12.5.0以降はネイティブESMもサポートしているが、既存のモック基盤が安定動作しているため、現時点ではCJS方式を維持する

**インメモリ実装の設計方針:**
- 各リポジトリのエクスポート関数と同一シグネチャの関数を持つオブジェクトを用意する
- データは `Map<string, T>` や配列で保持し、Worldのライフサイクル（シナリオ単位）でリセットする
- DB固有動作（`ON CONFLICT DO NOTHING`、楽観的ロック等）は、インメモリ実装内で同等のロジックを再現する
- ID引数（UUID型カラムに対応するパラメータ）はUUID形式をバリデーションし、不正な形式は即座にエラーとする。PostgreSQLの `invalid input syntax for type uuid` を模倣する（See: `features/support/in-memory/assert-uuid.ts`、背景: `docs/architecture/lessons_learned.md` LL-001, LL-002）
- モック機構の具体的な実装方式（requireキャッシュ書き換え、tsconfig-paths誘導等）はコーダーが選定する。ただしプロジェクト内で1方式に統一すること

**ライフサイクル:**
- `BeforeAll`: モジュール差し替えをインストール
- `Before`（各シナリオ前）: インメモリデータをクリア
- `After`（各シナリオ後）: 必要に応じたクリーンアップ（時刻スタブの復元等）

## 3. Cucumber World 設計

**Worldが保持すべき状態カテゴリ:**

| カテゴリ | 説明 | 用途例 |
|---|---|---|
| 現在のユーザー | userId, edgeToken, ipHash, isPremium, username | 「ユーザーがログイン済みである」の状態管理 |
| 名前付きユーザーマップ | `Map<string, UserContext>` | 「ユーザー "UserA"」「ユーザー "UserB"」の複数ユーザー管理 |
| 現在のスレッド | threadId, threadTitle | 「スレッド "今日の雑談" を閲覧している」の状態管理 |
| 最後の操作結果 | lastResult, lastError | Then ステップでのアサーション対象 |
| 時刻制御 | currentTime (Date) | 日付変更・経過時間のシミュレーション |

**設計原則:**
- Worldのプロパティは型付きで定義する（anyを避ける）
- 各シナリオの独立性を保証するため、Beforeフックで全状態をリセットする
- テスト用エンティティの生成ヘルパーをWorldのメソッドとして提供する

## 4. ディレクトリ構成とファイル分割方針

```
features/
  support/
    world.ts              # World クラス定義
    hooks.ts              # BeforeAll / Before / After フック
    mock-installer.ts     # リポジトリモジュール差し替えの機構
    in-memory/            # インメモリリポジトリ実装群（1リポジトリ = 1ファイル）
  step_definitions/
    common.steps.ts       # 複数featureで共有されるステップ
    {feature}.steps.ts    # feature固有のステップ（1 feature = 1ファイル）
```

**分割基準:**
- 1 feature = 1 stepsファイルが原則
- 3つ以上のfeatureで使われるステップは common.steps.ts に集約する
- 2つのfeatureでのみ共有されるステップは、先に定義されたfeatureのstepsファイルに置く（重複定義エラーを避ける）

## 5. 時刻制御の方針

多くのシナリオが日付・時間に依存する（日次リセットID、ストリーク、低活性スレッド判定、時間内ボーナス等）。時刻依存テストはFlakyテスト（実行タイミングにより結果が変わる不安定テスト）の最大の原因であり、以下の原則を厳守する。

### 5.1 時計凍結の原則

**方針:** テスト中は実時間に一切依存せず、凍結した時計のみを使用する。

- `Date.now` をグローバルにスタブ化し、World経由で任意の時刻に設定・進行する
- Beforeフックでオリジナルの `Date.now` を保存し、Afterフックで復元する
- sinon の `useFakeTimers` やカスタム実装のどちらでもよいが、1つの方式に統一すること

### 5.2 相対時刻の禁止

テストデータの時刻を `Date.now() - offset` で算出してはならない。テスト実行中に実時間が経過するため、ステップ間でミリ秒単位の不整合が生じFlakyテストの原因になる。

```typescript
// NG: 実行タイミングでミリ秒がずれる
const pastTime = new Date(Date.now() - 25 * 60 * 60 * 1000)

// OK: 時計を固定時刻に設定してからデータを作る
world.setCurrentTime(new Date('2026-03-12T10:00:00+09:00'))
// この時点で Date.now() は固定値を返すため、サービス内の new Date(Date.now()) も同じ値になる
```

**時刻依存シナリオの標準パターン:**

1. 時計を基準時刻 T に凍結する
2. 時計を過去（T - N時間）に設定し、事前データを作成する
3. 時計を T に戻し、テスト対象の操作を実行する
4. 必要に応じて時計を T + M 分に進め、後続操作を実行する
5. 結果をアサートする

### 5.3 サービス層の時刻取得統一

サービス層で現在時刻を取得する際は `new Date(Date.now())` を使用し、`new Date()` を直接使用しない。`Date.now` のスタブがサービス内部の時刻生成にも反映されることを保証するためである。

### 5.4 テストデータ作成の優先順位

時刻依存の事前データ（過去のレス、スレッドの最終書き込み時刻等）を作成する際は、以下の優先順位に従う:

1. **サービス経由での作成**（推奨）: 時計を過去に設定し、通常のサービスAPI（PostService.createPost等）で作成する。サービスの実際の動作と一致するため最も信頼性が高い
2. **インメモリリポジトリの直接操作**（許容）: サービス経由では作成が困難な状態（途中状態、異常状態等）に限りリポジトリの `_insert` 等を使用する。この場合もcreatedAt等の時刻フィールドは凍結時計の値を使うこと

## 6. スコープ外シナリオの扱い方針

featureファイルは変更禁止（CLAUDE.md禁止事項）であるため、タグの追加によるスキップはできない。

**方針:**
- 対象外のfeatureファイルは cucumber.js の `paths` で除外する
- 対象featureファイル内のスコープ外シナリオは `name` オプション（正規表現 negative match）で除外する
- 具体的な除外対象はスプリント固有の実装ガイドで定義する

## 7. テストピラミッドと多層テスト戦略

> Phase 2以降に適用。Phase 1はサービス層テスト（§1）+ 単体テスト（Vitest）で完結していた。

### 7.1 全体像

```
    ┌──────────────────────┐
    │  スモーク（§11）      │  §10と同一テスト — 本番環境で実行
   ─┼──────────────────────┼─
   │      E2E（§10）        │  ナビゲーション(A) + フロー検証(B) — ローカル環境
  ──┼───────────────────────┼──
  │    API テスト           │  HTTPレベル検証 — 専ブラ互換・Cookie等（10〜20本）
 ───┼────────────────────────┼───
 │    統合テスト              │  Supabase Local実DB — 既存BDDシナリオを流用
────┼─────────────────────────┼────
│   BDD サービス層            │  InMemoryリポジトリ — 高速・開発中の主力（既存）
─────┼──────────────────────────┼─────
│    単体テスト (Vitest)       │  純粋関数・ドメインモデル（既存）
──────┴──────────────────────────┴──────
```

**原則:** 下層ほどテスト数を多く、上層ほど厳選する。各層の責務を明確に分離し、重複検証を避ける。

3つのツールチェーンが6つの検証層を構成する:

| ツール | プロファイル / プロジェクト | 検証層 | 実行コマンド |
|---|---|---|---|
| Vitest | — | 単体テスト | `npx vitest run` |
| Cucumber.js | default | BDDサービス層 | `npx cucumber-js` |
| Cucumber.js | integration | 統合テスト | `npx cucumber-js --profile integration` |
| Playwright | api | APIテスト | `npx playwright test --project=api` |
| Playwright | e2e | ナビゲーション(A) + フロー検証(B) — ローカル | `npx playwright test --project=e2e` |
| Playwright | prod | ナビゲーション(A) + フロー検証(B) — 本番 | `npx playwright test --config=playwright.prod.config.ts` |

### 7.2 各層の責務と検証対象

| 層 | 検証対象 | 検証しないこと |
|---|---|---|
| 単体テスト | 純粋関数の入出力、ドメインルール | 外部依存、統合動作 |
| BDDサービス層 | ビジネスロジックの振る舞い、サービス間連携 | SQL、HTTP、UI |
| 統合テスト | SQL・RLS・マイグレーション・リポジトリ層の実動作 | HTTP、UI |
| APIテスト | ルーティング、Cookie処理、Content-Type、エンコーディング | ビジネスロジック（サービス層で検証済み） |
| ナビゲーションテスト（Phase A） | 全ページの到達性、主要APIの応答正常性、リンク・ボタンの操作可能性 | ビジネスロジック、データの正確性 |
| フロー検証テスト（Phase B） | 認証→書き込み→コマンド→専ブラ反映の一連のフロー | 個別のビジネスルールの網羅的検証 |

### 7.3 BDDシナリオの検証層マッピング

BDDシナリオ（D-03）は振る舞いの正本であり、UIの振る舞いも含めて記述してよい。ただし全シナリオがCucumber（サービス層テスト）で検証できるわけではない。サービス層で検証不可能なシナリオは、テストピラミッドの適切な層で代替検証する。

#### 7.3.1 マッピングルール

シナリオの性質に応じて検証層を決定する（ステップ定義のコメント形式は§7.3.2で規定）。

| シナリオの性質 | 検証層 | Cucumberでの扱い |
|---|---|---|
| ビジネスロジック（状態遷移、計算、コマンド処理等） | BDDサービス層 | 通常実行 |
| DOM/CSS表示（文字色、表示切替、レイアウト等） | Vitestコンポーネントテスト | pending |
| ブラウザ固有動作（ナビゲーション、認証フロー等） | E2E | pending |
| インフラ制約（エンコーディング、HTTP属性等） | APIテスト | pending |

#### 7.3.2 Cucumber pending運用ルール

- ステップ定義は `return "pending"` で実装し、Cucumber実行時にスキップさせる
- ステップ定義のコメントに pending理由と代替テストのファイルパスを記載する
- 検証を先送りするステップは必ず `return "pending"` を使用する。`assert(true)` 等のスタブアサーションでPASSにすることを禁止する（See: LL-007）
- featureファイル自体は変更しない（CLAUDE.md禁止事項）

```typescript
// Web限定シナリオ — Cucumberサービス層では検証不可
// 代替検証: src/__tests__/app/(web)/thread/eliminated-bot-display.test.tsx
Then('撃破済みボットの過去のレスは目立たない文字色で表示される',
  async function () { return "pending"; }
);
```

#### 7.3.3 代替テスト側のトレーサビリティ規約

代替テストのファイル先頭に、対応するBDDシナリオへの参照を記載する。BDDシナリオの文言が変更された場合、代替テスト側も追従すること。

```typescript
/**
 * @feature bot_system.feature
 * @scenario 撃破済みボットのレスはWebブラウザで目立たない表示になる
 * @scenario 撃破済みボットのレス表示をトグルで切り替えられる
 */
describe('撃破済みボット表示', () => { ... });
```

#### 7.3.4 pendingシナリオの管理

- 代替テストが未作成のpendingシナリオは技術的負債として扱う
- UIコンポーネント実装時に、対応する代替テストの作成を必須とする
- ナビゲーションテスト（§10.2）の拡充により代替テストが包含される場合、Vitestコンポーネントテスト側は削除してよい（二重検証の回避）

## 8. 統合テスト方針

### 8.1 基本方針

既存BDDシナリオ（`.feature`）とステップ定義をそのまま流用し、リポジトリの注入先をInMemory実装からSupabase Local実装に切り替えて実行する。

**メリット:** 新規テストコードの作成がほぼ不要。既存BDDシナリオが自動的に統合テストになる。

### 8.2 切替の仕組み

- テストプロファイル or 環境変数（例: `TEST_DB=supabase`）でリポジトリ実装を分岐する
- `BeforeAll` フックで判定し、InMemory / Supabase のいずれかをインストールする
- ステップ定義側は変更不要（リポジトリのインターフェースが同一であるため）

### 8.3 前提条件

- Docker Desktop + Supabase Local が起動していること（`npx supabase start`）
- マイグレーションが適用済みであること（`npx supabase db push`）
- テスト用の `.env` に `TURNSTILE_SECRET_KEY` を設定しないこと（§12参照）

### 8.4 データライフサイクル

- `Before`（各シナリオ前）: テスト用テーブルのTRUNCATE
- シナリオ間の独立性はインメモリ版と同様に保証する

## 9. APIテスト方針

### 9.1 基本方針

Next.jsサーバー（`next build && next start` または `next dev`）を起動し、fetchでAPIエンドポイントを叩くHTTPレベルのテスト。**全APIに書くのではなく、サービス層テストでカバーできない検証が必要な箇所に限定する。**

### 9.2 対象領域

| 対象 | 検証内容 | 理由 |
|---|---|---|
| 専ブラ互換API | Shift_JISエンコーディング、DAT形式、subject.txt、bbs.cgi | バイナリレベルの正確性はサービス層テストでは検証不可 |
| 認証API | Cookie設定（Secure, HttpOnly, SameSite）、レスポンスヘッダー | HTTP属性はサービス層では検証不可 |
| エラーレスポンス | HTTPステータスコード、エラーボディ形式 | クライアントとの契約 |

### 9.3 対象外

- ビジネスロジックの検証（サービス層テストで十分カバーされている）
- 正常系のCRUD操作（APIルートは薄いアダプターであり、追加の検証価値が低い）

## 10. E2Eテスト方針

> **§10と§11は同一テストケースを環境を変えて2回実行する。** テスト設計は本セクション（§10）で定義し、§11は本番固有の差分のみを記述する。

E2Eテストはナビゲーションテストとフロー検証テストの2分類で構成する。

| 分類 | Phase | 性質 | 目的 | 配置先 |
|---|---|---|---|---|
| **ナビゲーションテスト** | A | GET のみ | 全ページの到達性、全リンク・ボタンの操作可能性 | `e2e/smoke/` |
| **フロー検証テスト** | B | POST + DELETE | 認証→書き込み→コマンド→確認の一連のフロー | `e2e/flows/` |

**BDDとの役割分担:** BDDサービス層テストは「振る舞いの正しさ」を検証する。ナビゲーションテスト（Phase A）は「画面が到達可能であり、UI要素が操作可能であること」を検証する。フロー検証テスト（Phase B）は「認証から書き込み・確認までの一連のフローが完結すること」を検証する。

### 10.1 基本方針

Playwrightを使用したブラウザ自動テスト。環境差分（baseURL、認証トークン取得、クリーンアップ方法）はPlaywright Custom Fixturesで吸収し、specファイル自体は環境に依存しない。

- **ローカル:** Supabase Local + Next.js dev で全テスト（A + B + ローカル限定）を実行
- **本番:** 同一テストを本番環境（Cloudflare Workers）で実行（本番固有事項は§11参照）。ローカル限定テスト（§10.3.1）は `test.skip` で自動スキップされる

#### 10.1.1 環境抽象化フィクスチャ

specファイルは `e2e/fixtures/` が提供するカスタムフィクスチャ経由で環境依存操作を行う。specファイル内で環境を判定する `if` 分岐を書くことを禁止する（ローカル限定テストの `test.skip(isProduction, ...)` を除く）。

| フィクスチャ | ローカル実装 | 本番実装 |
|---|---|---|
| `authenticate` | Supabase REST で user + edge_token 作成 → Cookie設定 | 事前シード済みedge-tokenをCookieに設定 |
| `adminSession` | Supabase Auth でテスト管理者作成 → `/api/admin/login` | `/api/admin/login`（email+password） |
| `seedThread` | Supabase REST INSERT | アプリAPI経由で作成 |
| `cleanup` | Supabase REST DELETE（全件） | 管理者API DELETE（テストデータのみ） |
| `isProduction` | `false` | `true` |

**`authenticate` ローカル実装の設計根拠:** `authenticate` は「認証済み状態を作る前提条件フィクスチャ」であり、認証UIを検証するテストではない（§10.3.1: 「認証UIの動作検証と書き込みフローの検証は別の関心事」）。AuthModal UI検証は `auth-flow.spec.ts`（ローカル限定テスト）が担う。DB直接シーディングにより高速・安定・副作用なしでテスト前提条件を作成できる。

フィクスチャの実装詳細はコーダーが決定する。上記のインターフェース（名前と役割）を維持すること。

### 10.2 ナビゲーションテスト（Phase A）

#### 10.2.1 目的と背景

過去に「ページは存在するがリンクが設置されておらず到達不可」という問題が発生した。BDDサービス層テストではUIの到達性を検証できないため、ナビゲーションテストで「全ページにアクセスでき、全ボタン・リンクがエラーなく動作する」ことを保証する。

#### 10.2.2 対象ページの特定方法

`src/app/` 配下の全 `page.tsx` が検証対象となる。新規ページ追加時に対応するテストも追加すること（増減基準は§10.2.5を参照）。

#### 10.2.3 各ページでの検証項目

各ページについて以下を検証する。ビジネスロジックの正しさやデータの正確性は検証しない。

| 検証項目 | 内容 |
|---|---|
| **アクセス可能性** | HTTPステータス200で応答する（404/500でない） |
| **JSエラーなし** | `page.on('pageerror')` でコンソールエラーが発生しない |
| **主要UI要素の存在** | ページ固有のランドマーク要素（ヘッダー、メインコンテンツ領域等）が表示される |
| **リンク・ボタンの操作可能性** | ページ内の主要リンクがクリック可能であり、遷移先が404/500でない |

#### 10.2.4 ファイル構成

```
e2e/
  smoke/
    navigation.spec.ts    # 全ページの到達性・ナビゲーション検証
```

ページ数が増加し1ファイルが肥大化した場合は、機能領域単位（例: `smoke/admin.spec.ts`, `smoke/thread.spec.ts`）に分割してよい。

#### 10.2.5 増減基準

- **追加:** `src/app/` に新しい `page.tsx` を追加した場合、対応するテストケースを必ず追加する
- **削除:** ページを削除した場合、対応するテストケースも削除する
- **変更:** ページのURL構造やランドマーク要素のIDが変更された場合、テストを追従させる
- **検証:** `npx tsx scripts/check-e2e-coverage.ts` でカバレッジを機械的に検証できる（終了コード 0=全カバー, 1=未カバーあり）

#### 10.2.6 認証が必要なページの扱い

認証が必要なページ（マイページ等）は、テスト内で事前に認証を完了させた状態で到達性を検証する。認証フロー自体の検証はフロー検証テスト（§10.3）の責務であり、ナビゲーションテストでは認証を前提条件として処理する。

#### 10.2.7 動的ルートの扱い

動的ルート（例: `/threads/[threadId]`）は、テスト前にDBにシードデータを投入し、実在するIDでアクセスする。存在しないIDに対する404応答の検証はスコープ外とする。

#### 10.2.8 pendingシナリオのUI到達性

BDDシナリオがpending（§7.3）である場合でも、そのシナリオの起点となるUI要素（ボタン、リンク等）が既にページに配置されているならば、ナビゲーションテストでリンク先の到達性（404/500でないこと）を検証する。pendingはビジネスロジックの検証を先送りするものであり、ページの存在確認を免除するものではない。

### 10.3 フロー検証テスト（Phase B）

スレッド作成→レス書き込み→コマンド実行→専ブラAPI反映確認→管理者によるテストデータ削除の一連のフローを直列実行する。コマンド追加時はシナリオ中間部にテストを追加する。

#### 10.3.1 テストケースの環境分類

フロー検証テストは「環境共通テスト」と「ローカル限定テスト」に分類する。

| 分類 | 実行環境 | 認証方式 | 目的 |
|---|---|---|---|
| **環境共通テスト** | ローカル + 本番 | フィクスチャ `authenticate` | 認証済み状態から書き込み→コマンド→確認のフローが完結すること |
| **ローカル限定テスト** | ローカルのみ | AuthModal UI直接操作 | 未認証→401→AuthModal→認証→リトライの連結フローが動作すること |

**設計根拠:** 認証UIの動作検証と書き込みフローの検証は別の関心事である。環境共通テストの `authenticate` フィクスチャはテストの前提条件を作る役割であり、認証UIを検証する役割ではない。認証UI連結は本番ではTurnstile制約により自動テスト不可能なため、ローカル限定テストとして独立させる。

**ローカル限定テストの実装規約:**
- `test.skip(isProduction, "本番では認証UIフローをテストしない")` で本番実行時にスキップする
- フィクスチャ `authenticate` を使用せず、AuthModal UI操作を直接記述する
- 対象: 「未認証で書き込み→AuthModal表示→認証コード入力→認証成功→操作リトライ」の連結フロー

#### 10.3.2 検証範囲

- 掲示板としての基本機能（書き込みパス、専ブラAPI整合、管理者操作　各１本）
- コマンドシステム（各コマンドごとに１本）
- 合計で10～20本程度を想定する（コマンド数に応じて増加）

| カテゴリ | 検証内容 |
|---|---|
| 書き込みパス | スレッド作成・レス投稿（Web API + 専ブラ bbs.cgi）が正常に完了すること |
| コマンドシステム | 各コマンドの実行結果がレスに反映されること（コマンド追加に応じて増加） |
| 専ブラAPI整合 | 書き込んだデータが subject.txt / DAT に反映されること（Shift_JIS） |
| 管理者操作 | テストスレッドの削除が正常に完了し、公開APIから消えること |

#### 10.3.3 ファイル構成

```
e2e/
  fixtures/              # 環境抽象化フィクスチャ（§10.1.1）
    index.ts             # 統合エクスポート（extend済みの test オブジェクト）
    auth.fixture.ts      # authenticate, adminSession
    data.fixture.ts      # seedThread, cleanup
  smoke/
    navigation.spec.ts   # Phase A（環境共通）
  flows/
    basic-flow.spec.ts   # Phase B 環境共通テスト
    auth-flow.spec.ts    # Phase B ローカル限定テスト（認証UI連結）
  api/                   # APIテスト（ローカル専用、§9）
  helpers/
    turnstile.ts         # Turnstileモック（ローカル専用）
```

#### 10.3.4 安全性制約

- テストデータは管理者削除で必ずクリーンアップする（テストシナリオの最後を管理者権限の削除で終わらせる。削除機能の検証と一石二鳥）。テスト失敗時も `afterAll` で試みる
- smoke専用ユーザーと通常ユーザーを区別可能にする（スレッドタイトルや書き込み内容にプレフィックスを付与する等）
- 並行実行しない（`workers: 1`）

## 11. スモークテスト（本番実行）

§10と同一のテストケースを本番環境（Cloudflare Workers）で実行し、本番固有の障害（ランタイム非互換、マイグレーション未適用、DI配線漏れ等）を自動検出する。テスト設計（分類・選定基準・検証範囲・安全性制約）は§10を参照。

**背景:** インシデント分析の結果、本番で発見されたバグの大半が書き込みパスに集中していた（`docs/operations/incidents/` 参照）。読み取り専用のナビゲーションテスト（Phase A）では原理的に検出不可能であり、フロー検証テスト（Phase B）を新設した。テスト失敗後のデバッグ・障害調査はテスト戦略のスコープ外であり、別エージェントの責務とする。

### 11.1 本番実行固有の方針

本番環境で実行する際の固有事項。ローカル実行には適用されない。

- **Turnstile回避:** 本番コードにバイパス経路は追加しない。edge-token は本番DBに事前シード（runbook参照）、admin_session は email+password 認証で取得（Turnstile不介在）
- **シークレット:** `.env.prod.smoke`（gitignore済み）に保管
- **実行タイミング:** Gitプッシュ後、自動的にCloudflareにデプロイされる（数分のラグあり）。CFデプロイ完了を確認してから実行
- **実行トリガー:** オーケストレーターが指示、または人間が手動実行
- **CI/CD自動化:** オーケストレーターの定常作業として自動化。GithubAction利用は将来検討（前提: health/version エンドポイント新設）

## 12. 外部サービスのテスト環境対応

### 12.1 Cloudflare Turnstile

Turnstileはテスト自動化を妨げない設計とする。

| テストレベル | 対処方法 | 仕組み |
|---|---|---|
| BDD / 単体テスト | InMemoryモック | `features/support/in-memory/turnstile-client.ts` で常に成功を返す |
| 統合テスト / APIテスト | 環境変数未設定でバイパス | `turnstile-client.ts`: `TURNSTILE_SECRET_KEY` 未設定時は `return true` |
| E2E | Cloudflare公式テスト用サイトキー | `1x00000000000000000000AA`（常に成功）を使用 |

**注意:** テスト用の環境変数に本番の `TURNSTILE_SECRET_KEY` を設定してはならない。設定するとCloudflare APIへの実通信が発生し、テスト環境からのリクエストが検証失敗する。

## 13. テスト実行の役割分担

### 13.1 開発AI（bdd-coding）

- テストコードの作成（全レベル）
- InMemoryテストの実行（開発サイクル中の高速フィードバック）
- 単体テストの実行

### 13.2 ゲートAI（bdd-gate）

- **テストコードは書かない。実環境での実行と合否判定に特化する。**
- Supabase Local + Next.jsサーバーを起動
- 全テストスイートを実行（BDD + 統合 + API + E2E）
- 結果の判定・レポート作成（PASS/FAIL・差分分析）
- フェーズ5検証サイクルにおける最終品質ゲートとして機能する

### 13.3 スモークAI（bdd-smoke）

- **テストコードは書かない。本番環境でのスモークテスト実行と結果報告に特化する。**
- CFデプロイ完了を確認後、Playwright本番スモークテストを実行（`playwright.prod.config.ts`）
- 結果を報告（PASS/FAIL・失敗テスト詳細）
- オーケストレーターのデプロイ後ステップとして機能する
- 障害調査は行わない（テスト失敗時はオーケストレーターが `auto-debugger` の起動を判断する）

## 14. CF Smokeテスト（技術検証）

`nodejs_compat` 経由のNode.js API（iconv-lite, crypto等）がCloudflare Workers Runtimeで動作するかの技術検証として実施済み。テストピラミッドの常設層ではなく、ランタイム互換性の初期確認として位置づける。テストコードは `e2e/cf-smoke/workers-compat.spec.ts` に残存する。

- **実行:** `npm run test:cf-smoke`（`wrangler dev` 事前起動が必要）
- **対象:** Shift_JIS生成、DATファイル、SSR基本動作等（7件）
- **追加検証の目安:** 新規Node.js固有APIを `src/lib/` 内で使い始める場合
