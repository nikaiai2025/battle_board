## 問題の内容

TASK-072 の作業中に、admin-service.ts の管理者削除フォールバックメッセージが
admin.feature の BDD 期待値と一致しないことが判明した。

### BDD 期待値（admin.feature L34-38）

```gherkin
And 独立したシステムレスが追加される:
  """
  ★システム
  🗑️ レス >>3 は管理者により削除されました
  """
```

### 現在のサービス実装（admin-service.ts L48）

```typescript
const ADMIN_DELETE_FALLBACK_MESSAGE = '管理者によりレスが削除されました'
```

### 差異

| 項目 | BDD 期待値 | 実装 |
|---|---|---|
| `🗑️ ` プレフィックス | あり | なし（コメント付き・なし共通） |
| `レス >>N は ` プレフィックス | あり | なし（コメントなしのみ） |
| メッセージ文言 | `管理者により削除されました` | `管理者によりレスが削除されました` |

### テスト結果

- 「管理者がコメント付きでレスを削除する」: PASS（ただし `🗑️ ` なしで許容している）
- 「管理者がコメントなしでレスを削除する」: FAIL

コメント付きシナリオが PASS しているのはステップ定義で `expectedBodyContent.includes(systemPost.body)` という逆包含チェックを行っているためで、BDD の本来の期待値（`🗑️ ` 付き body）は検証できていない。

## 選択肢と各選択肢の影響

### 選択肢 A: admin-service.ts を修正する（推奨）

admin-service.ts の以下を修正する:
1. `ADMIN_DELETE_FALLBACK_MESSAGE` を `'レス >>N は管理者により削除されました'` 形式に変更
2. `createPost` の body に `🗑️ ` プレフィックスを付与する

**影響**:
- locked_files 外のファイル変更（CLAUDE.md エスカレーション条件に該当）
- ユーザーから見たシステムレスのメッセージが変わる（CLAUDE.md エスカレーション条件に該当）
- BDD 正本に準拠した振る舞いになる
- フォールバックメッセージにレス番号を含めるため、deletePost の引数に postNumber が必要になる可能性がある（インターフェース変更）

具体的な変更案:
```typescript
// admin-service.ts
const systemMessageBody = comment
  ? `🗑️ ${comment}`
  : `🗑️ レス >>${post.postNumber} は管理者により削除されました`
```

これにより deletePost に postNumber の追加引数は不要（post オブジェクトから取得可能）。

### 選択肢 B: ステップ定義で緩い検証を行う（非推奨）

「独立したシステムレスが追加される:」ステップで body の完全一致ではなく
キーワードマッチングを行う（「管理者」「削除」などのキーワードが含まれるかを検証）。

**影響**:
- locked_files 外のファイルを変更しない
- BDD の期待値（DocString）が実質的に無視される
- サービス層の実装不足が隠蔽されたままになる
- テストとして意味が薄くなる

### 選択肢 C: コメント付きは PASS なので コメントなし シナリオのみ保留扱いにする

コメントなし削除シナリオの「独立したシステムレスが追加される:」ステップで
フォールバックメッセージのみ特別扱いする。

**影響**:
- 同上（サービス層の不足を隠蔽）

## 推奨する選択肢

選択肢 A（admin-service.ts の修正）を推奨する。

理由:
- BDD が正本であり、BDD 期待値に合わせてサービス実装を修正するのが正しい開発フロー
- admin-service.ts は locked_files にはないが、変更は1箇所（body 生成ロジック）のみで影響範囲が小さい
- postNumber は既に `post` オブジェクトから取得可能（インターフェース変更不要）
- TASK-069 の実装不足の修正であり、機能追加ではない
