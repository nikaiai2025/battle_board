# BOT cron設計 — 議論ログ

> 日付: 2026-03-18
> 参加者: 人間 + アーキテクトAI
> ステータス: **確定** → TDR-010 として D-07 に記録済み

## 背景

HUMAN-001（荒らし役BOT本番稼働）の仕様決定において、cron実行間隔の設計を議論。

## 確定事項

| # | 項目 | 決定 |
|---|---|---|
| ① | cron間隔 + 投稿タイミング制御方式 | 30分間隔（`0,30 * * * *`）+ DB予定時刻方式（`next_post_at`）。TDR-010 |
| ② | 認証方式 | Bearerトークン（BOT_API_KEY）※設定済み |
| ③ | 日次リセット時刻 | 15:00 UTC（= 00:00 JST） |

## 議論の経緯

### 1. cron間隔 vs BOT投稿間隔の違い

- **cron**: GitHub Actionsが定期的にAPIを叩く。「投稿する時間か？」と聞くだけ
- **BOT投稿間隔（60〜120分）**: BOT側ロジックが「前回投稿からN分経ったか」を判断する閾値
- cronは「目覚まし時計」、投稿間隔は「BOTの気分」

### 2. cron間隔の選択肢と影響

| cron間隔 | 実際の投稿パターン | ランダム性 |
|---|---|---|
| 1時間 | ほぼ2時間等間隔になる（閾値60-120分に対してcronが粗すぎる） | ほぼなし |
| 30分 | 1.5h or 2h の2パターン程度 | 低い |
| 15分 | 60〜120分の範囲内で15分刻み | 中程度 |
| 5分 | GitHub Actionsの実質下限 | 高い |
| 1分 | GitHub Actionsでは不可（5分に丸められる＋無料枠超過） | - |

**1時間間隔の問題**: BOT内部が90分後に書き込むタイプの場合、1回目(60分後)は空振り、2回目(120分後)で投稿。実質2時間おきの等間隔になり、60〜120分のランダム閾値が意味を成さない。

### 3. ランダム遅延（ジッター）の検討

15分刻みの投稿パターンを解消するため、ランダム遅延の導入を検討。

#### 案A: GitHub Actionsワークフロー内で `sleep $((RANDOM % 600))`

- 0〜10分のランダム待機後にAPI呼び出し
- **問題: sleep中もGitHub Actionsの実行時間に課金される**
- 月間無料枠（2,000分）を超過する可能性大

#### 案B: API側（BotService内）で `setTimeout` による遅延

```typescript
const jitterMs = Math.floor(Math.random() * 600_000);
await new Promise(resolve => setTimeout(resolve, jitterMs));
```

- **問題: Vercel/Cloudflareには実行時間の上限がある**
  - Vercel Hobby: 10秒
  - Cloudflare Workers Free: 30秒
- 最大10分のsleepはタイムアウトする。**この案は不可**

#### 案C: DB予定時刻方式

botsテーブルに `next_post_at` 列を追加。投稿完了時に「次回投稿予定時刻」を秒単位で計算してDBに保存。cronは現在時刻と予定時刻を比較するだけ。

```
投稿完了 → next_post_at = now() + random(60〜120分) + random(0〜秒) をDBに保存
cron起動 → now() > next_post_at なら投稿、そうでなければスキップ
```

- サーバー側でsleepしない
- 秒単位のランダム性が確保される
- 実装コストはDB列1本とチェックロジック追加

### 4. 根本的な制約

**どの方式でも、投稿はcron実行直後に発生する。** cronが15分間隔なら、外部から観測される投稿時刻は :00 :15 :30 :45 のいずれかになる。

- 案Cでも「どの15分枠で引っかかるか」がバラつくだけで、15分刻みの痕跡自体は残る
- 案Cの利点は投稿間隔のバリエーションが豊富になること（最初の方式より均一なパターンが出にくい）

### 5. 15分刻みの痕跡を完全に消すには

| 方法 | 実現性 |
|---|---|
| cron間隔を5分にする（GitHub Actions実質下限） | 可能。5分刻みの痕跡は残るが、人間の投稿パターンとの区別は困難になる |
| 常駐プロセス（別インフラ） | 秒単位の精度で投稿可能だが、インフラ追加が必要 |

## GitHub Actions cronの技術的制約

| 制約 | 内容 |
|---|---|
| 構文 | 5フィールド（分 時 日 月 曜日）。秒指定不可、小数点不可 |
| 最短間隔 | 実質5分（それより短くても5分に丸められる） |
| 実行精度 | 混雑時は数分〜十数分遅延する（保証なし） |
| 無料枠 | 月2,000分 |

## 最終決定（2026-03-18）

### 決定内容

- **cron間隔**: 30分（`0,30 * * * *`）
- **投稿タイミング制御**: DB予定時刻方式（`bots.next_post_at`）
- **グリッド痕跡の扱い**: ゲーム情報として受容（プレイヤーの推理・攪乱材料）

### 決定に至った判断

1. **GitHub Actions 無料枠（月2,000分）**: ジョブ1回 = 最低1分課金。15分間隔（2,880分/月）は超過、30分間隔（1,440分/月）は余裕あり
2. **真のランダム投稿は現行インフラでは不可能**: サーバーレスの実行時間制限（Vercel 10秒 / Cloudflare 30秒）により sleep 系は全滅。常駐プロセスはインフラ追加が必要
3. **:00, :30 のキリの良い時刻を採用**: 素数間隔等によるグリッド痕跡緩和を検討したが、cron `*/N` 構文は毎時固定分に展開されるため効果は限定的。痕跡をゲーム要素として積極的に受容する方向に転換。BOT投稿パターンの推理、人間がBOTを装う攪乱プレイなど、創発的プレイの材料になりうる
