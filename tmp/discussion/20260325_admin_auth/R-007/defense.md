# R-007 防御レポート

レビュアー: Blue Team
対象: ダッシュボード（統計/日次推移）実装

---

## ATK-007-1

**問題ID**: ATK-007-1
**判定**: ACCEPT（部分同意）

**根拠**:

指摘の核心である「`days=abc` で `RangeError` が発生し 500 が返る」は事実である。

`history/route.ts:49-51` で `Number.parseInt("abc", 10)` → `NaN` が生成され、`Math.min(NaN, 90)` も `NaN` のまま `getDashboardHistory({ days: NaN })` に渡る。`admin-service.ts:654` の計算 `today.getTime() - NaN * 24 * 60 * 60 * 1000` は `NaN` となり、`new Date(NaN).toISOString()` が `RangeError` をスローする。これは `try-catch` で 500 に変換されるため、**レスポンスは 500 になる**。

ただし重大度の評価には異論がある。

- 攻撃者は**管理者セッション**を保有している必要がある。認証後の行為であるため、通常の外部攻撃者は到達できない。
- 500 は `catch` ブロックで明示的に処理されており（`route.ts:59-65`）、サーバープロセスはクラッシュしない。エラーログが増加するという事象は DoS というよりログ汚染であり、重大度 CRITICAL は過大評価である。

ただし、**管理者 UI が誤った `days` を送信しうる状況**（URLの直接編集、バグ等）での誤動作は本番で発生しうる。`NaN` を `7` にフォールバックする防御処理が欠如しているのは実装の不備である。

`fromDate` に任意文字列を渡すと空配列が返るという指摘（DB側で黙ってフィルタ）は、**SQLインジェクションに至らない**こと、**500 ではなく空の正常レスポンスが返る**こと、**誤認させられる対象が管理者本人のみ**であることから、単独では CRITICAL に値しない。

**結論**: NaN フォールバック欠如は修正が必要な実装の不備であるため ACCEPT。ただし重大度は MEDIUM が妥当。

## ATK-007-2

**問題ID**: ATK-007-2
**判定**: REJECT

**根拠**:

`today` パラメータに任意の日付文字列を渡せるという指摘は技術的に正確だが、以下の理由でセキュリティ上の問題とは見なせない。

**利用者が管理者自身のみである設計上の前提**

当 API は `verifyAdminSession` による管理者認証を突破した者のみが利用できる（`route.ts:39-42`）。正当な管理者が `?today=2000-01-01` を意図的に送信してダッシュボードを操作する行為は、管理者権限の範囲内での「過去日付での確認」に相当する。これは不正アクセスではなく、設計上許容された操作である。

**BDD シナリオが要求する振る舞いとの整合性**

シナリオ「管理者がダッシュボードで統計情報を確認できる」は「本日の書き込み数が表示される」を期待しているが、これはデフォルト動作（`today` 省略時）の振る舞いを規定している。`today` パラメータは `admin-service.ts:616` のコメントに「省略時は現在日付」と明記されており、任意日付での集計取得は設計上の機能である。

**「改ざんの痕跡が残らない」という指摘について**

攻撃者が外部から管理者セッションを乗っ取り、ダッシュボード表示を改ざんしたとしても、`today` の値はブラウザの閲覧にのみ影響し、DB の値を変更しない。ダッシュボードの**表示値が誤る**だけであり、データ損失・データ改ざん・サービス停止のいずれにも該当しない。

**監査ログの欠如について**

監査ログ不在は `admin-service.ts:144-148` のコメント「将来の監査ログ用（現在は簡易ログのみ）」で既知事項として明示されており、現状の設計判断として記録されている。これは ATK-007-2 固有の問題ではなく全管理者操作共通の既知事項である。

**結論**: 管理者セッションを持つユーザーによる操作であり、データ損失・セキュリティ侵害・サービス停止のいずれも発生しない。REJECT。

## ATK-007-3

**問題ID**: ATK-007-3
**判定**: ACCEPT（部分同意）

**根拠**:

**BDD テストのアサーション強度に関する指摘**

`admin.steps.ts:1924-1999` の Then ステップが「型チェック + 非負」のみであることは事実であり、指摘は正確である。シナリオ「管理者がダッシュボードで統計情報を確認できる」に対応する Given（事前データ投入）が存在しないため、インメモリリポジトリが全件 0 を返す状態でテストが実行される。集計関数が常に 0 を返すバグが混入しても、このテストでは検出できない。

**ただし、単体テストによる補完を評価する**

`admin-dashboard.test.ts` では以下の具体的な値による検証が実施されている。

- `totalUsers` に `42` を注入し、`result.totalUsers` が `42` であることを `toBe` で検証（136-151行）
- `todayPosts` に `18`、`activeThreads` に `7`、`currencyInCirculation` に `3500` を注入し、各値を `toBe` で検証（同上）
- `today` オプションが `PostRepository.countByDate` に正確に渡されることを `toHaveBeenCalledWith` で検証（154-169行）

したがって、「集計関数が常に 0 を返すスタブを直接使っていてもテストは通る」という指摘は BDD テスト単体には当てはまるが、**単体テストと合わせた全体のテストカバレッジとしては集計ロジックのバグを検出できる**。

**日次推移の件数確認について**

`admin.steps.ts:2072-2090` が `length > 0` のみを確認し、投入した 7 件全件の返却を確認していないという指摘は正確である。`getDashboardHistory` の日付範囲計算（`admin-service.ts:649-656`）が誤って件数を削っても、BDD テストはパスする。この点は単体テスト（`admin-dashboard.test.ts:301-314`）でも `toHaveLength(7)` の確認はモックに対してのみ行われており、実際の日付範囲計算ロジック自体のバグは検出できない。

**結論**: BDD テストのアサーション強度不足は実装バグの見逃しリスクを生む。特に日次推移の件数検証（`length > 0` のみ）は改善が必要。ACCEPT。ただし、単体テストが数値の正確性を補完しているため、総合的なリスクは中程度（MEDIUM）である。
