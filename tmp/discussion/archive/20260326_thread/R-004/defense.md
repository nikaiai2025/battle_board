レビュアー: Blue Team
対象: @url_structure シナリオ群の実装

## ATK-004-1

**問題ID**: ATK-004-1
**判定**: ACCEPT（限定的）

**根拠**:

`findByThreadKey` が `board_id` でフィルタしないという事実は認める。ただし重大度の評価が異なる。

### 現時点の実態

`thread_key` は Unixタイムスタンプ（10桁）を文字列化した値であり、スレッド作成日時を起源とする。同一タイムスタンプで2件のスレッドが異なる板に同時生成される確率は現実的にほぼゼロであり、現行のDB制約（`thread_key` のユニーク制約の有無）に依存する。本番環境は現在 `livebot` 板1枚のみであり、**今すぐデータ整合性が崩れる状況にない**。

### 将来リスクの評価

複数板が追加された場合にはじめて顕在化する問題である。この点はRedチームの指摘通り。ただし現在のBDDシナリオ・要件定義（D-01）に「複数板」はMVPスコープ外として記載されており、機能追加のタイミングで同時に対処すべき設計課題である。

### テスト検出能力について

インメモリリポジトリ（`features/support/in-memory/thread-repository.ts:52-58`）が `boardId` フィルタを持たないため、仮に複数板テストデータをセットアップしても検出できない。この「テストのブラインドスポット」は問題だが、現行BDDシナリオ自体が複数板を前提としていないため、シナリオ追加なしにテストだけ修正しても受け入れ基準との整合が取れない。

### 判定根拠まとめ

- 本番で今すぐデータ損失・セキュリティ侵害が発生する条件（複数板の存在）がMVPスコープ外
- 将来的に複数板が追加される際に対処が必要な設計負債であることは認める
- 重大度はCRITICALではなく **MEDIUM**（複数板実装前に対処すべきタスク）が妥当

## ATK-004-2

**問題ID**: ATK-004-2
**判定**: ACCEPT

**根拠**:

### findById の is_deleted フィルタ欠落

`thread-repository.ts:76-90` の `findById` が `is_deleted` 条件を持たないことは事実であり、`softDelete` で論理削除されたスレッドでも `thread` は `null` にならない。`src/app/(web)/threads/[threadId]/page.tsx:41` の `notFound()` ガードを素通りし、削除済みスレッドの `boardId` + `threadKey` で新URLへリダイレクトされる。

### findByThreadKey の is_deleted フィルタ欠落

同様に `thread-repository.ts:97-114` の `findByThreadKey` も `is_deleted` 条件を持たない。`src/app/(web)/[boardId]/[threadKey]/[[...range]]/page.tsx:94` のフロー上、削除済みスレッドが `thread` として返るため `notFound()` を呼ばず、削除済みコンテンツが表示される。

### 現実的な再現条件

`softDelete` は管理者APIとして実装済みであり（`thread-repository.ts:285-294`）、管理者が実際にスレッドを削除してもURLを知っているユーザーがアクセスすると削除済みコンテンツが表示される。管理操作の効果が無効化されるため、これは**今すぐ発生しうる問題**である。

### 既存テストのカバレッジ

インメモリ `findById`（`features/support/in-memory/thread-repository.ts:43-46`）はストアに格納されたオブジェクトをそのまま返すため `is_deleted` フィルタを行わない。BDDテストはこの欠陥を検出できない。

### 対処方針

`findById` と `findByThreadKey` の両関数に `.eq("is_deleted", false)` フィルタを追加し、インメモリ実装も同様に修正する必要がある。なお `findAllForAdmin`（line 383）は管理者用として意図的に削除済みを含む全件取得であり、修正対象外。

## ATK-004-3

**問題ID**: ATK-004-3
**判定**: ACCEPT（限定的）

**根拠**:

### BDDテスト戦略書との整合

ステップ定義（`thread.steps.ts:1144-1153`）のコメントには以下が明記されている：

```
// / → /livebot/ のリダイレクトは Next.js redirect() で実装済み。
// サービス層テストでは livebot へのアクセスが有効であることを確認する。
```

BDDテスト戦略書 D-10 §7.3.1 では「ブラウザ固有動作（ナビゲーション、認証フロー等）」の検証層として **E2E** が指定されており、`redirect()` の動作検証はCucumberサービス層テストの責務外と明示されている。

### ただし実装の問題は認める

`redirectTarget: "/livebot/"` のハードコードは「意図的な設計（E2Eに委任）」と「テストが何も検証していない空振り」の境界が曖昧である。具体的には：

1. `redirectTarget` の値を検証する Then ステップが存在する（line 1188-1202）
2. しかしその値はステップ自身がセットしたハードコード文字列である
3. 結果として「`livebot` が板として実在すること」すら検証しておらず、`getThreadList(TEST_BOARD_ID)` の結果も Then ステップで参照されない

D-10 §7.3.2 の規定では、サービス層で検証不可能なシナリオは `return "pending"` とすべきとされている。現状は `pending` でもなく、かつ実質的な検証もしておらず、**LL-007（`assert(true)` 等のスタブアサーションでPASSにすることの禁止）に抵触する可能性がある**。

### 重大度の評価

`redirect()` の誤り（宛先変更）はE2Eナビゲーションテストで検出可能であり（D-10 §10.2.3）、`page.tsx` のリダイレクト実装自体は1行のシンプルなコードである。実害の発生経路は「`page.tsx` の変更を誰かがレビューなしで行う」ケースに限定される。重大度はHIGHではなく **LOW** が妥当だが、ステップ実装を D-10 §7.3.2 準拠の `return "pending"` に修正してテストの意図を明確にすべきである。
