# R-002 防御レポート

レビュアー: Blue Team
対象シナリオ: 本文が空の場合は書き込みが行われない

---

## ATK-002-1

**判定**: ACCEPT

**根拠**:

攻撃側の指摘は正確である。コードの流れを以下に示す。

1. Step 1 (`post-service.ts:357`) で `validatePostBody(input.body)` を呼ぶ。この時点の `input.body` は `"!iamsystem"` であり空ではないため、バリデーションを通過する。
2. Step 5.5 (`post-service.ts:499`) でコマンドが `isStealth && success` の場合に `resolvedBody = resolvedBody.replace(commandResult.rawCommand, "").trim()` が実行される。
3. `input.body === "!iamsystem"` のみの場合、コマンド除去後 `resolvedBody === ""` となる。
4. Step 9 (`post-service.ts:663`) で `body: resolvedBody` すなわち `""` をそのまま `PostRepository.create` に渡す。
5. DBスキーマ (`00001_create_tables.sql:71`) の `body TEXT NOT NULL` は空文字列を許容する（PostgreSQLにおいて `NOT NULL` は空文字列を拒否しない）。

コードコメント自体（`post-service.ts:488`）が「除去後の本文が空: 空文字列の書き込みとして投稿する」と明記しており、設計として意図的に許容している。しかし BDD シナリオ R-002 の振る舞い「本文が空の場合は書き込みが行われない」に反する。ユーザーが `!iamsystem` のみ入力してコマンドが成功した場合、空本文のレスがDBに保存されてしまう。

**影響評価**: ステルスコマンドを知っているユーザーが意図的に利用すると、空本文レスを量産できる。コンテンツの品質に直接影響し、現実的に再現可能な条件である。

## ATK-002-2

**判定**: ACCEPT（ただし範囲を限定）

**根拠**:

`createPost` が空本文を受け取ったとき `{ success: false, code: "EMPTY_BODY" }` を返すことを直接検証するテストは `post-service.test.ts` に存在しない。攻撃側の指摘はこの点において正しい。

ただし、以下の反論を付記する。

- `validatePostBody` 自体の空本文テストは `src/lib/domain/rules/__tests__/validation.test.ts` にて網羅されている（`"空文字列は無効"` 他、L153–L198）。
- BDDテスト (`posting.steps.ts:196–228`) が `body: ""` で `createPost` を呼び出すシナリオをカバーしており、`createPost` が `validatePostBody` を呼ぶ結合パスはBDDレベルで検証されている。
- したがって「バリデーション呼び出し自体が削除されても単体テストはグリーンのまま」という指摘は正しい。`createPost` の単体テストに空本文ケースが欠落しているため、サービス層のリグレッションを単体テストレベルで検出できない。

ATK-002-1 で指摘されたステルスコマンド後の空本文バグは、現在の単体テストでは絶対に検出できない構造になっている。この空白は実際のバグ（ATK-002-1）の見落としに直結しており、単体テストカバレッジの欠落として受け入れる。

## ATK-002-3

**判定**: REJECT

**根拠**:

指摘のコード動作（`targetThread?.isPinned` が `null` 返時に `undefined` となりガードをスルーする）は事実である。しかし以下の理由により、この指摘は R-002 シナリオのスコープ外かつ現実的な影響が限定的である。

**BDDスコープ外**: R-002 シナリオは「本文が空の場合は書き込みが行われない」のみを検証するものであり、「存在しないスレッドIDへの書き込み」は別シナリオ（スレッド存在チェック）の責務である。攻撃側自身もコメント内で「R-002 のシナリオとの関連は空本文 + 存在しないスレッドIDの組み合わせでは Step 1 のバリデーションエラーが正しく返る」と認めており、本指摘はシナリオ R-002 の核心的な振る舞いに対する問題ではない。

**エラー経路の保全**: 存在しないスレッドIDを指定した場合、最終的に `PostRepository.create` で `thread_id` の FK制約違反が発生し書き込みは失敗する。データ損失（空スレッドへの永続化）は起きない。500系エラーが返ることは UX 上の問題であり、`THREAD_NOT_FOUND` の明示的なエラーコードが返るべきという改善余地はあるが、セキュリティ侵害やデータ損失には至らない。

**本番到達条件が限定的**: 正規の API エンドポイント経由では `threadId` はスレッド一覧や閲覧から取得された有効な UUID が渡される。存在しない threadId を意図的に指定するには API を直接叩く必要があり、攻撃的な意図がない通常ユーザー操作では発生しない。

以上より、ATK-002-3 は R-002 シナリオ範囲外の改善提案として位置づけるべきであり、REJECT とする。
